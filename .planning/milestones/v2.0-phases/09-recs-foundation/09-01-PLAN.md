<objective>
Land the architectural seam for the v2.0 recommendations engine inside the player service: pluggable `SignalModule` interface, weighted-ensemble aggregator with per-pool min-max normalization, three persistence tables auto-migrated on startup, a thin GORM repository, and a precompute orchestrator stub. No concrete signals, no API endpoint, no frontend changes — silent infrastructure that Phases 10-13 bolt onto without touching `ensemble.go`, `normalize.go`, or any API handler.

Purpose: The architectural payoff of the weighted-ensemble pattern (spec §2.1) is that future signals plug in via a single registry entry. This phase pays the up-front cost so every later phase ships faster. The min-max normalizer (spec §2.3) is the architectural fix that makes signal weights coherent across raw scales; without it Phase 10's S3 ~[0,large_int] would silently dominate Phase 12's S5 ~[0,0.05] regardless of weight.

Output: Eleven files (eight new in `services/player/internal/service/recs/`, one new domain file, one new repo file, one modified main.go), three new Postgres tables, and four atomic test+impl+commit cycles (TDD discipline for every code-bearing component).
</objective>

From the binding design spec §6.1 (the contract for Task 2 — copy verbatim):
```go
type SignalModule interface {
    // Identifier for logging, debug page columns, weight registry.
    ID() string
    // Precompute step (runs in cron / on-write). May be a no-op for stateless signals.
    Precompute(ctx context.Context, userID string) error
    // Score a candidate pool against a user (request-time). Returns map of raw scores.
    // Signals that don't depend on user state (S3, S4) ignore userID.
    Score(ctx context.Context, userID string, candidates []AnimeID) (map[AnimeID]float64, error)
}
```

This phase implements the signature with the project's strong-typing conventions: `SignalID` (not raw `string`) for `ID()`, `UserID` and `AnimeID` aliases, and `RawScore` (not raw `float64`) for the `Score` map values. These are type-aliases (`type X = string`) not named types, so they remain assignable to `string`/`float64` and Phase-10+ signals do not need conversion noise.

From the binding design spec §2.3 (the formula for Task 3):
```
S_k(u, i) = (S_k_raw(u, i) − min_pool) / (max_pool − min_pool + ε)     # ε = 1e-9
```
Degenerate pool (max−min < ε) → all zeros, never NaN.
