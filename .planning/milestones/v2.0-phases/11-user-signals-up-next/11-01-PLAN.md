## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| logged-in client → /api/users/recs | Authenticated endpoint reachable with JWT; returns user-personalized payload pulled from per-user Redis cache key |
| 6h user cron → Postgres | In-process goroutine that iterates all users with `watch_history` rows and runs RunForUser for each |
| MarkEpisodeWatched → UserOrchestrator.TriggerForUser | Hot path on every watched-episode event spawns a goroutine that runs precompute under a Redis-debounce lock |
| handler → Redis (per-user key) | Read-through cache for 6h-TTL personalized payload per user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | I (Information disclosure) | GET /api/users/recs (logged-in) | mitigate | Per-user cache key is `recs:user:{user_id}:topN` — keyed on the JWT-derived user_id; user_id comes from `authz.ClaimsFromContext` (signed JWT) so it can't be forged. Anonymous request never sees logged-in payload because the claims branch only fires when claims are present and non-empty. Anime metadata in the response is the same public-elsewhere data Phase 10 returned. Filter ensures `hidden = true` and the user's `completed/dropped` statuses are excluded. |
| T-11-02 | S (Spoofing) | UserID extraction in handler | mitigate | UserID comes from `authz.ClaimsFromContext` which validates the JWT signature via `OptionalAuthMiddleware` (Phase 10 wiring at router.go:127-130). A request with a forged claim fails JWT verification and reaches the handler with `claims = nil`, falling through to the anonymous branch. No SQL injection risk: `claims.UserID` is bound via GORM's parameterized query (`Joins("LEFT JOIN anime_list al ON al.user_id = ?", userID)`). |
| T-11-03 | D (Denial of Service) | UserOrchestrator.TriggerForUser hot path | mitigate | Redis SETNX-based debounce caps re-runs at one per 5 minutes per user — even a malicious flood of MarkEpisodeWatched events for one user triggers at most one Precompute every 5 minutes. The trigger itself is `O(1)` (one Redis SETNX). The Precompute goroutine runs under `context.WithTimeout(5*time.Minute)` to bound runaway. |
| T-11-04 | D | 6h user cron | mitigate | RunOnce iterates `SELECT DISTINCT user_id FROM watch_history` (~10s of users at current scale; bounded by the user table). Per-user errors are logged but do NOT halt the tick. Boot tick on Start ensures cold-start signals appear within seconds of redeploy without waiting 6 hours. |
| T-11-05 | T (Tampering) | rec_user_signals JSONB writes | accept | Only the player service writes this table (no public API path). The orchestrator writes via `UpsertUserSignals` using GORM's parameterized clause.OnConflict — same SQL injection guards as Phase 9. JSONB content is server-generated; never echoes user input. |
| T-11-06 | E (Elevation of privilege) | personalized branch with claims | mitigate | Claims-based dispatch only — there is no impersonation path. A logged-in user can ONLY fetch their own recs (the cache key + handler logic both pin to `claims.UserID`). The handler does not accept `?user_id=` overrides. Admin force-recompute is deferred to Phase 14. |
| T-11-07 | I | recs:debounce:{user_id} key | accept | The key only stores the literal "1"; presence/absence is the only signal. No PII or semantically meaningful payload. Compromise of Redis already implies broader compromise. |
| T-11-08 | D | S1 Pearson computation per cron tick | mitigate | At current scale (~10 users, ~3500 anime), per-user S1 Precompute is bounded at `O(users² × shared_anime + candidates × neighbors) ≈ 100 × 50 + 3500 × 10 = ~40k tiny float ops`, sub-millisecond in Go. The full RunOnce tick is `O(users)` precomputes ≈ ~10 seconds at the upper bound. Far below the 6-hour cadence. If user count grows past ~100k, switch to a queue + worker pool (deferred to v3.0 per CONTEXT.md spec §13). |
