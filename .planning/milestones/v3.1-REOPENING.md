## Why was v3.1 reopened?

Three independent triggers landed within a 24-hour window:

### Trigger 1 — Regression of SCRAPER-HEAL-08 (2026-05-18)

v3.0 Phase 20 cutover (the planned deletion of HiAnime + Consumet dead code) **over-rotated**. The pre-existing untracked deletions on disk included not only `HiAnimePlayer.vue` and `ConsumetPlayer.vue` (intended targets), but also `EnglishPlayer.vue` (the v3.1 Phase 21 deliverable, SCRAPER-HEAL-08) and its e2e specs. These got swept into commit `fe3b487 docs(03): plan for torrent + job queue + metrics (raw-jp Phase 03)` whose message acknowledged the unintended sweep:

> "Also folds in pre-existing untracked deletions of legacy EN players (HiAnimePlayer.vue, ConsumetPlayer.vue, **EnglishPlayer.vue** + e2e specs + catalog hianime/consumet parsers + docker/megacloud-extractor) that the RAW JP provider (v0.1) replaces per the 'replace, don't preserve dead identity' decision."

The "replace" framing was correct for HiAnime + Consumet but wrong for EnglishPlayer — there was no replacement; the file just disappeared.

A second cleanup pass later that day (multiple commits between `9bb8a85` and `49b32f2`) finished the rip by:
- removing the EN tab markup from `frontend/web/src/views/Anime.vue`
- narrowing `VALID_LANGUAGES` to `'ru' | '18+' | 'raw'` (dropping `'en'`)
- narrowing `VALID_PROVIDERS` to `'kodik' | 'animelib' | 'hanime' | 'raw'` (dropping `'english'`)
- stripping the multi-source switcher i18n keys

Net effect: production has no user-facing path to the operational scraper backend. The backend works (gogoanime + animepahe respond, daily canary runs, alerts route to maintenance bot), but there is literally no tab to click. This regression is what Phase 24 (SCRAPER-HEAL-17..20) restores.

The cleanup itself was technically requested by the operator ("Fully delete legacy Consumet and HiAnime first, so it will not spoil the context yet again"). The over-rotation was the AI agent's error, not the operator's. Phase 24's CONTEXT.md D6 takes the position that restoring from git is the right repair — not rewriting from scratch — because the 1973-line snapshot at commit `8424e99` contains accumulated polish (sub/dub toggle, three-phase loader, multi-source dropdown ARIA, late-arriving preferredCombo fix) that's worth preserving.

## Why GSD methodology for this work

The operator instructed: "do it by gsd guidelines - rewrite the current v3.1 Scraper Self-Healing".

The standalone spec + plan written earlier 2026-05-19 (`17a7c86` + `893a4c6`) followed the superpowers brainstorming + writing-plans path. That produced clean artifacts but in a parallel docs tree (`docs/superpowers/`) disconnected from the GSD planning surface (`.planning/`). Reopening v3.1 brings the work into the same workstream that shipped Phases 21-23, with consistent file conventions (CONTEXT.md, PLAN.md per task, SUMMARY.md, VERIFICATION.md, REQUIREMENTS.md traceability) so:

- The operator can drive each phase with `/gsd-plan-phase --phase N` and the existing GSD command surface.
- The traceability table in REQUIREMENTS.md stays the single source of truth — every SCRAPER-HEAL-NN has exactly one phase home and a clear shipped/pending status.
- The final `/gsd-audit-milestone --milestone v3.1` re-run will produce a coherent audit document that supersedes the 2026-05-13 one without orphan findings.

The standalone spec + plan keep their copyright in `docs/superpowers/` as a historical record of the brainstorming path that led to this reopening, but the *authoritative* planning surface is `.planning/`.
