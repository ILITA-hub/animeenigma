# Roadmap: v3.1 Scraper Self-Healing

**Milestone:** v3.1
**Status:** Draft — ready for execution
**Defined:** 2026-05-13
**Requirements:** `.planning/milestones/v3.1-REQUIREMENTS.md`
**Spec:** `docs/plans/2026-05-13-scraper-self-healing-spec.md`
**Phase numbering:** continuous from v3.0 (Phases 15-20) — v3.1 starts at Phase 21.

## One-liner

After v3.0 built the in-house scraper, v3.1 makes it **self-healing**: streams are validated before users see "play", the orchestrator transparently routes around poisoned/broken servers, a daily canary surfaces upstream regressions within 24 h, and the existing maintenance bot is taught to dispatch the new failure modes against `.claude/maintenance-prompt.md` Patterns 6/7.

## Phases

### Phase 21 — Playability Foundation

**Goal:** A playback request that would have hit an ad-poisoned server instead silently rolls forward to the next server and plays real video. Latency cost is masked by a three-phase loader.

**Delivers:**
- `libs/streamprobe/` package (Probe + hardcoded ad-CDN blocklist + TODO-stub for Redis lift)
- gogoanime server-priority list (env-driven, default deprioritizes VibePlayer)
- gogoanime `GetStream` per-server fallback using the gate
- Winning-server cache per (anime, ep) for 5 min
- New metrics: `parser_unplayable_total`, `parser_ad_decoy_total`
- Scraper response: `meta.gated` field
- `EnglishPlayer.vue` three-phase loader (EN + RU)

**Requirements:** SCRAPER-HEAL-01 through SCRAPER-HEAL-08.

**Acceptance:** Production EnglishPlayer plays real video for at least one canary anime (Frieren) when VibePlayer is the first-source. Cold-path latency ≤ 8 s end-to-end; warm-path latency unchanged from current. No FE loader is ever visibly "stuck" — phase text updates within 1 s of each backend transition.

### Phase 22 — Provider Robustness

**Goal:** When a single CDN behind a server fails (signed-URL expired, 403, geo-block), the orchestrator transparently tries that server's secondary URL family before giving up on the server.

**Delivers:**
- streamhg / earnvids extract both `hls2` and `hls3` URLs from packed JS
- HLS proxy allowlist adds `managementadvisory.sbs` + `exoplanethunting.space`
- ISS-011 inline entry in `docs/issues/README.md` documenting the VibePlayer poisoning incident as the first one v3.1 catches automatically

**Requirements:** SCRAPER-HEAL-09 through SCRAPER-HEAL-11.

**Acceptance:** A synthetic test where `hls2` returns 403 still produces a playable stream via `hls3`. ISS-011 entry merged into README.

### Phase 23 — Self-Maintenance Loop

**Goal:** A regression at any upstream site (anitaku.to, vibeplayer.site, streamhg, earnvids, future) is detected within 24 h by a daily canary that exercises real production code paths, surfaces a labeled alert into the existing maintenance bot, and gets dispatched per Patterns 6/7 — without a human needing to notice.

**Delivers:**
- Scheduler canary cron job (daily 03:00, Frieren + One Piece + 3 dynamic from watch_history)
- `playability_canary_runs_total` metric with `anime_slot` label
- New Grafana dashboard `scraper-provider-health.json`
- Three alert rules → `services/maintenance` webhook
- End-to-end verification that maintenance bot routes the new alerts correctly (the prompt is already in place — `.claude/maintenance-prompt.md` was updated 2026-05-13 with Patterns 6/7 + Scraper Playability Regression section)

**Requirements:** SCRAPER-HEAL-12 through SCRAPER-HEAL-16.

**Acceptance:** Synthetic Pattern 6 alert injection into `/api/grafana-webhook` produces a maintenance bot response that correctly classifies tier (`button_fix` for code fixes, `escalate` for IP-block), names the right files, and stays within bot scope. Real canary fires at 03:00 the night after deploy and records pass/fail outcomes for all 5 anime × every server.

## Out of scope (deferred to v3.2 or later)

- **Phase 24 — VibePlayer Recovery via WARP egress** — separate spec; ships when there is appetite to revive VibePlayer as a working server.
- **Phase 25 — MinIO Hot Archival** — separate v3.2 spec; rip + serve popular titles from MinIO to decouple from upstream availability.

## Risks

| Risk | Mitigation |
|---|---|
| Playability gate latency (1-2 s per server) annoys users | FE loader phases (SCRAPER-HEAL-08); cached winning server skips gate on warm path |
| `hls3` CDNs (`managementadvisory.sbs`, `exoplanethunting.space`) also rotate, requiring further allowlist updates | Caught automatically by canary (Phase 23); maintenance bot Pattern 7 fix path covers allowlist additions as `button_fix` |
| watch_history-driven canary anime list goes empty on a quiet day | Fallback to `anime_list ORDER BY updated_at DESC` (SCRAPER-HEAL-12) |
| New Redis key for winning-server cache (SCRAPER-HEAL-05) collides with existing namespace | All scraper keys live under `scraper:*` prefix — verify no existing `scraper:winning_server:*` key. Add a one-line note to `libs/cache/keys.go` (or wherever scraper keys are listed) |
| Maintenance bot misroutes a new reason-enum value not present in prompt | Reason enum frozen in SCRAPER-HEAL-01 + reflected verbatim in `.claude/maintenance-prompt.md`. Adding a new reason requires updating both files. Phase 23 acceptance test exercises every enum value at least once. |
