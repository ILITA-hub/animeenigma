<objective>
Close the D-03 daily feedback loop that makes the plan-03 ReadGate dynamic: a daily job computes per-`(operation, table)` `db_read` P95 from the ClickHouse register and publishes a compact `read_thresholds` Redis hash; each GORM service runs a background `ThresholdRefresher` ticker that snapshots that hash and atomically feeds it into its in-memory `ReadGate` (plan 03). The hot-path Query callback consults only the in-memory snapshot (Pitfall 4 — never a synchronous Redis/HTTP lookup per SELECT). Cold-start (no day-2 baseline) falls back to the static default.

Purpose: AR-EFFECT-01 (the dynamic half of D-02/D-03). The P95 query lives in analytics (the only ClickHouse-connected service — A4); distribution is via a Redis hash (A3 — lowest friction, all services already Redis-connected); the daily trigger lives in scheduler (robfig/cron). Static cold-start default = 50ms / P95 (A1 — flagged; expose it as config).

Output: analytics P95 compute, scheduler cron entry, the shared `ThresholdRefresher` + its tests.

Effort: `UXΔ = +1 (Better)` (self-tuning read cost ledger; the sparse-but-meaningful read rows derive from here) · `CDI = 0.06 * 8` (one CH query + one cron entry + one refresher ticker across analytics/scheduler/libs; reuses the existing cron + CH-conn + Redis seams) · `MVQ = Griffin 84%/83%` (a clean ClickHouse→Redis→in-memory feedback loop; slop-resistant via the snapshot-not-sync-lookup discipline).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ClickHouse events → analytics P95 compute | Read-only over the existing analytics driver.Conn; no recording feedback (D-16) |
| read_thresholds Redis hash → service ReadGate | Internal Docker-network Redis; the daily job writes, services read on a ticker |
| analytics recompute endpoint (IF HTTP) | Must be /internal/* only, Docker-network, never gateway-proxied |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-14 | Denial of Service | a poisoned/oversized read_thresholds hash stalls/balloons the in-memory snapshot or the GORM read hook | mitigate | The refresher defensively bounds the accepted field count + skips malformed/oversized values; the gate consults an in-memory snapshot only (never a sync read in the callback) so a bad hash can never block a SELECT |
| T-03-15 | Elevation / Information Disclosure | daily-P95 recompute exposed via gateway | mitigate | Prefer Redis-direct publish (no route); IF an endpoint is used it is /internal/* only with a 256KB body cap (mirrors effects.go) |
| T-03-16 | Tampering | analytics' own ingestion DB writes recorded as effects via this path | mitigate | This plan only READS the events table from analytics; D-16 exclusion (no effect callback in analytics) is enforced in plan 06 boot wiring |
| T-03-17 | Denial of Service | missed daily run blanks the threshold table → read-row flood (no P95 → static default fires everywhere) | accept | Hash carries a 48h expiry so one missed run is tolerated; static default (50ms) bounds the flood; revisit cadence if it recurs |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages; clickhouse-go + robfig/cron + go-redis already present (RESEARCH Package Legitimacy Audit: zero new) |
