# Phase 1: Instrumentation Baseline - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the project's success metric — auto-pick override rate — observable in Grafana before any behavior changes ship. Specifically:

- Emit a `combo_override` event whenever a user (logged-in or anonymous) changes language, player, team, or episode within 30 seconds of player load — across all four players (Kodik, AnimeLib, HiAnime, Consumet).
- Stand up a Grafana tile showing override rate segmented by tier, language, anonymous-vs-auth, and player.
- Capture ≥ 24 hours of real-traffic baseline data and record the snapshot in PROJECT.md before Phase 6 starts.
- Deploy via `make redeploy-player` and verify live on production.

**Strictly out of scope for this phase:**
- Any behavior change to the resolver, the auto-pick logic, or the player UI itself
- Any change to `watch_progress.completed` semantics (Phase 3)
- Any analytics audit or schema gap-fill (Phases 2 / 5)

</domain>

<decisions>
## Implementation Decisions

### Detection Placement
- **D-01:** A single Vue composable — `frontend/web/src/composables/useOverrideTracker.ts` — is imported by all four player components (`KodikPlayer.vue`, `AnimeLibPlayer.vue`, `HiAnimePlayer.vue`, `ConsumetPlayer.vue`). The composable watches the props/refs that drive the player (language, watch_type, team/translation_title, current episode) and detects user-initiated changes. No per-player divergence — the composable is the single definition of "what counts as an override."
- **D-02:** Detection happens at the Vue prop / reactive-state level, not inside the iframe. This means **Kodik IS covered** — combo changes flow through the Vue parent (the user picks a translation in the picker UI, which swaps the iframe `src`), so the composable observes them even though the iframe itself emits no events.
- **D-03:** On detection, the composable POSTs to a new endpoint on the player service: `POST /api/users/preferences/override`. The endpoint is unauthenticated-friendly — accepts both JWT-authenticated requests (user_id from claims) and anonymous requests (X-Anon-ID header from localStorage). One handler, one source of truth.

### Storage Shape
- **D-04:** Two write paths, both reusing existing infrastructure — no new GORM table, no migration:
  1. **Prometheus counter** — extend `libs/metrics/watch.go` with a new `CounterVec`:
     ```go
     ComboOverrideTotal = promauto.NewCounterVec(
         prometheus.CounterOpts{Name: "combo_override_total", ...},
         []string{"tier", "dimension", "language", "anon", "player"},
     )
     ```
     This drives the live Grafana tile via PromQL.
  2. **Structured Loki log line** — `log.Infow("combo_override", ...)` with the same labels plus per-event detail (anon_id or user_id, anime_id, load_session_id, original_combo, new_combo, ms_since_load). Loki + Promtail are already in the stack; LogQL queries via Grafana Explore give Phase 6 retroactive per-event analysis without us designing for it now.
- **D-05:** A matching `combo_resolve_total` counter is also emitted on every successful resolve (the denominator for override rate). Add to `libs/metrics/watch.go`. The resolve handler in player service emits this; the composable does NOT (the composable only emits overrides, the backend already knows about resolves).
- **D-06:** Loki retention is ~31 days (per `docker/loki/loki-config.yml`). This is sufficient for the 24h baseline (success criterion 3) and the Phase 7 before/after comparison. If Phase 6 needs > 30 days of per-event detail for time-decay validation, that becomes a Phase 5 schema-add — explicitly out of scope here.

### What Counts as an Override
- **D-07:** "First user-initiated change per `(load_session_id, dimension)` within 30s of player mount."
  - `load_session_id` is a UUIDv4 generated by the composable on mount.
  - "Dimension" is one of `{language, player, team, episode}` — emitted as a label so we know WHICH choice the auto-pick got wrong, not just that it did.
  - Only the FIRST change to each dimension per session counts; subsequent changes are ignored (one event per dimension per session, max).
- **D-08:** Excludes auto-advance (player-initiated episode change at end-of-episode), scrubbing, pause/resume, quality switches — none of these are part of the auto-picked combo. The composable distinguishes user-initiated changes from auto-advance by gating on a Vue event from the user-facing picker UI (NOT on raw prop changes, which can fire from auto-advance).
- **D-09:** Re-visiting the same anime mints a new `load_session_id` → fresh 30s window. Each visit gets graded once. This is correct: "did the auto-pick land THIS time."
- **D-10:** The 30s window starts when the resolved combo is applied to the player props, not when the page navigation begins. Avoids false positives during slow first-load.

### Anonymous User Identity
- **D-11:** Generate a UUIDv4 once on first visit and store it as `anon_id` in localStorage (key: `aenig_anon_id`). The composable adds `X-Anon-ID: <uuid>` to both the resolve call and the override emit for any user without a JWT.
- **D-12:** This unlocks a real per-anon-user override rate (denominator = unique anon_ids that resolved, numerator = unique anon_ids that overrode), not just a ratio of anon overrides over anon resolves.
- **D-13:** Pulled forward from Phase 7 — D-01 (anonymous localStorage preferences) needs exactly this key. By creating it in Phase 1, Phase 7 inherits the infrastructure and there is no second invention.
- **D-14:** No PII. UUIDv4 only. Cleared with cookies/localStorage. Not joined to any auth identity unless the user later signs in (and even then we don't claim that join in Phase 1).

### Grafana Tile
- **D-15:** Add to the existing `docker/grafana/dashboards/preference-resolution.json` dashboard (already lives in the same domain) rather than a new dashboard. New panel: "Auto-Pick Override Rate (combo_override / combo_resolve)" with 5 segmentations: tier, language, anon-vs-auth, player, dimension.
- **D-16:** Tile refreshes within 1 minute (success criterion 2). Use Grafana's standard auto-refresh; PromQL `rate(combo_override_total[5m]) / rate(combo_resolve_total[5m])` over the chosen segmentation.
- **D-17:** Provisioning: dashboard JSON is checked in to `docker/grafana/dashboards/` and auto-loaded by the Grafana provisioning config (no manual import). Verifying live = open `https://admin.animeenigma.ru/grafana` after `make redeploy-player` and confirm the panel renders with non-zero data once a few overrides land.

### Claude's Discretion
- Exact Vue composable API shape (return values, lifecycle hooks) — pick the idiom most consistent with existing composables under `frontend/web/src/composables/`.
- The denominator emit (combo_resolve) — whether to also emit on the resolve cache-hit path or only on cold resolve. Default: emit on every resolve outcome (cache or live), because tile rate should reflect "every player load," not "every uncached load."
- Naming of the new endpoint — `POST /api/users/preferences/override` is the working name; planner may rename if there's a closer existing convention in player handlers.
- Whether to gate the override POST behind a debounce (e.g. coalesce two changes within 200ms into one event) — recommendation: yes, debounce 250ms, ignore if dimension already emitted in this session.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`libs/metrics/watch.go`** — extend with `ComboOverrideTotal` and `ComboResolveTotal` `CounterVec`s; same `promauto.NewCounterVec` pattern as `TranslationSelectionsTotal` already in this file. Zero new dependencies.
- **`libs/logger`** — structured zap logger already used everywhere; `log.Infow("combo_override", k1, v1, ...)` lands in Loki via promtail with no extra wiring.
- **Player service handler pattern** — `services/player/internal/handler/*.go` shows the chi-route + handler-method shape. New `OverrideHandler` follows the same template.
- **Anon localStorage pattern** — none yet (this phase introduces it). Becomes the foundation for Phase 7's D-01.

### Established Patterns
- **Prometheus metric registration** — `promauto.NewCounterVec` at package init; no manual `MustRegister`.
- **Grafana dashboards as code** — JSON in `docker/grafana/dashboards/` auto-loaded by provisioning; do not edit live dashboards in the UI.
- **Frontend composables** — co-located in `frontend/web/src/composables/`, named `use<Thing>.ts`, return reactive refs + functions.
- **Loki structured logging** — every service uses `libs/logger` zap with structured fields; promtail already ships container stdout to Loki.

### Integration Points
- **Player service router** — `services/player/internal/transport/router.go` registers all routes; new `POST /api/users/preferences/override` joins the existing `/api/users/preferences/...` group (no gateway routing change needed — gateway already proxies `/api/users/*` → `player:8083`).
- **Resolve endpoint** — currently emits `PreferenceResolutionTotal` per outcome; we extend it to also emit `ComboResolveTotal` with the same labels we'll use for `ComboOverrideTotal` (so PromQL `rate(override) / rate(resolve)` lines up by label).
- **All four players** — each currently mounts the picker + the playback surface; the composable hooks into both: it reads the resolved combo (the picker's selected values) and watches for user-initiated change events.

</code_context>

<specifics>
## Specific Ideas

- The user emphasized that Phase 1 must be measurable BEFORE behavior changes ship — the override rate is "a single observable behavior that measures the project's actual goal" (PROJECT.md Key Decisions). This biases every choice in this phase toward "trustworthy first, fancy second."
- The denominator (`combo_resolve_total`) was added on Claude's initiative because override-count alone is not a rate. Without it, Phase 7's success criterion 5 ("measurable drop versus the Phase 1 baseline") cannot be evaluated.
- The Grafana tile lives on the existing `preference-resolution.json` dashboard rather than a new file because the user pre-built that dashboard for the same domain — extending it preserves the operator's mental model.

</specifics>

<deferred>
## Deferred Ideas

- **Per-event DB table** — considered for richer retroactive analysis. Deferred to Phase 5 (Analytics Gap-Fill). If Phase 6 ends up needing > 30d retention or per-event SQL, the audit in Phase 2 will surface it as a candidate.
- **Override reason capture** — "did the user override because the language was wrong, the team was wrong, or just clicked the wrong thing?" The dimension label answers WHICH but not WHY. A free-text "why did you change?" prompt was discussed elsewhere as out of scope; revisit if Phase 7's Advanced Settings panel surfaces a natural place for it.
- **A/B segmentation of the resolver** — "ship Tier 2 rewrite to 50% of users and compare override rates" was implicit in the success metric framing. Out of scope for Phase 1; revisit at the start of Phase 6 if it becomes load-bearing.
- **Cross-device join of anon → auth** — when an anon user signs up, we technically know "this anon_id became this user_id" and could join their pre-signup override rate to their post-signup data. Privacy-sensitive; defer.

</deferred>
