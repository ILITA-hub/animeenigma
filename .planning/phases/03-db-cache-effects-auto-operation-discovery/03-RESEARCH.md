## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** DB writes are ALWAYS fact-rowed. INSERT/UPDATE/DELETE → one `db_write` effect row carrying `table`, `op`, and `rows` (rows-affected, from GORM `RowsAffected`).
- **D-02:** DB reads are gated by a dynamic per-query P95 threshold. A SELECT is fact-rowed (`db_read`) only when its execution exceeds that specific query's own daily-recomputed P95 for its `(operation, table)` key. A static fallback threshold applies at cold-start. The long tail of trivial reads is NEVER fact-rowed.
- **D-03:** The dynamic threshold requires a daily ClickHouse → threshold-table feedback loop. Planner decides where the daily job runs (likely `services/scheduler`), how the threshold table is distributed, the static cold-start defaults, and the percentile (P95 default; P90 acceptable).
- **D-04 (rationale, locked):** "Most popular operation" is answered for free by span-metrics (D-13), NOT by logging all reads. DB fact rows are the deliberately-sparse cost ledger. Logging all reads was explicitly rejected.
- **D-05:** Cache effects are AGGREGATED per key-class, not one row per op. In-process counters keyed by `(key_class, result[hit|miss], operation)`; a flush goroutine emits summed `cache` rows (`requests = N`) on a ~10s interval (tunable). Mirrors the Phase-2 HLS reaper.
- **D-06:** Aggregated cache rows carry no per-`trace_id` (they span many traces). They still carry `operation` (D-09 caveat) + `key_class` + result.
- **D-07:** key-class = prefix + sub-namespace. ~20-30 stable classes derived from `libs/cache/ttl.go` key builders with variable IDs stripped (e.g. `anime:list`, `anime:similar`, `anime:detail`, `user:profile`, `search`, `progress`, `video:manifest`, `extid`, …).
- **D-08:** UNIFIED stack-frame attribution for ALL effects (egress + db + cache). Nearest `*/internal/service/*` frame from `runtime.Callers` is the PRIMARY `operation`. This RETROFITS Phase-2 egress (currently coarse baggage label as primary) to stack-frame-primary.
- **D-09:** Fallback chain, never empty: nearest service frame → baggage endpoint label (`tracing.ReadBaggage`) → origin name (`goroutine(name)` / `scheduled_job(name)`).
- **D-10 (aggregate caveat):** Timer-flushed aggregates have no live stack at flush time — they MUST capture `operation` at record-time, or fall back to baggage. HLS reaper retrofit captures the session's operation at session-start. For cache, planner's discretion whether aggregated cache uses fine stack-frame op (walk per op) or coarser baggage op (cheaper); coarser-for-cache is acceptable.
- **D-11 (perf, locked):** The stack-walk runs ONLY on the async recording path, never the request hot loop. Capture PCs synchronously (cheap), resolve frame→function names on the async side where feasible.
- **D-12:** Enable Tempo's metrics_generator (span-metrics + service-graphs processors) with `remote_write` to Prometheus — config only, no per-span code.
- **D-13 (rationale):** This is the free total-coverage popularity/RED layer complementing the sparse cost ledger.
- **D-14 (cross-phase hand-off):** The span-metrics generator lives in Tempo, which Phase 6 retires. At consolidation the per-operation RED metrics must be regenerated by the OTel Collector's spanmetrics connector. Enable in Tempo now; carry the hand-off into Phase 6.
- **D-15:** All 7 data services with `gormtrace.InstrumentGORM` get DB effect recording (catalog, player, themes, auth, notifications, scheduler, library); cache-using services get cache effects.
- **D-16 (correctness, locked):** EXCLUDE `services/analytics` from effect recording (it is the EffectStore sink — self-referential).
- **D-17:** Redis-only edge services (gateway / rooms / watch-together — no GORM) get cache effects only if the signal is useful (e.g. gateway rate-limit cache). watch-together uses Redis as a datastore, not a cache. Planner decides per-service.

### Claude's Discretion
Flush interval for cache aggregation (~10s default), exact key-class taxonomy, static cold-start read-threshold defaults + percentile (P95 default), where the daily-P95 job runs and how the threshold table is distributed, db_read vs db_write detection in the GORM callback set, sync-PC-capture vs async-resolve split for the stack-walk, whether aggregated cache uses fine vs coarse `operation` (D-10), and the per-service Redis-only decisions (D-17).

- Pyroscope continuous profiling → v2 (AR-V2-02).
- AggregatingMergeTree / SummingMergeTree pre-aggregated rollups → v2 (AR-V2-01); Phase 5 builds dashboards on per-effect fidelity first.

## Summary

This is a **brownfield extension**, not a greenfield build. Phase 1 (ClickHouse wide-event store) and Phase 2 (async/batched/drop-on-full `Producer` + `EffectSink` + W3C-baggage attribution + the HLS aggregation reaper) already shipped and are live-verified. **Roughly 70% of the plumbing this phase needs already exists.** The analytics ingestion side is fully ready: `domain.Event`, the ClickHouse `events` table, and the `/internal/effects` handler already carry `effect_kind`, `target_kind`, `target`, and the `row_count` measure. The GAP is on the producer side (`libs/tracing`): the `Effect` struct and `wireProducerEffect` lack a rows field and an `anime_id` field, and `EffectKind`/`Target` are currently hard-coded to the egress shape.

The four net-new pieces are: **(1)** a GORM callback set in `libs/tracing/gormtrace` that emits `db_write` (always) / `db_read` (P95-gated) effects via the existing global sink; **(2)** an in-process cache aggregator in `libs/cache` modeled byte-for-byte on the existing `HLSSessions` reaper; **(3)** a `runtime.Callers` operation resolver in `libs/tracing` that becomes the primary attribution for ALL effects (retrofitting egress); and **(4)** a `metrics_generator` config block in `infra/tempo/tempo.yaml` plus the daily ClickHouse→threshold-table feedback loop for read-gating (D-03).

**Critical scope note:** `libs/cache` and `libs/tracing` are **already in `go.work` and already imported** by every affected service. This phase adds **NO new `libs/` module** — it only EXTENDS existing ones. The 13-Dockerfile `COPY libs/{name}/go.mod` rule from MEMORY.md does **NOT** apply here. `[VERIFIED: go.work + service go.mod inspection]`

**Primary recommendation:** Extend `Effect`/`wireProducerEffect` with `Rows int` + `AnimeID string` first (small, unblocks everything). Build the `runtime.Callers` resolver as a standalone testable helper in `libs/tracing`. Add the GORM callback hook in `gormtrace`. Clone `HLSSessions` into a `libs/cache` cache aggregator. Add the Tempo `metrics_generator` block. For D-03, have **analytics** (the only ClickHouse-connected service) compute the per-`(operation, target)` P95 daily and expose it via a new internal read-only endpoint or write it to Redis; the 7 services poll/cache it and consult it in the GORM `Query` after-callback.

### Pattern 1: GORM after-callback for DB effects (D-01)
**What:** Register `After` callbacks on Create/Update/Delete (writes) and Query (reads) that read `db.Statement.Table`, the operation kind, and `db.Statement.RowsAffected`, then emit an `Effect` to the global sink.
**When to use:** Once per `*gorm.DB`, right where `InstrumentGORM` already runs (after `database.New()`, before serving).
**Example:**
```go
// Source: gorm.io/docs/write_plugins.html + gorm.io/docs/hooks.html [CITED]
// Register alongside the existing otelgorm plugin in gormtrace.
func RegisterEffectCallbacks(db *gorm.DB, sink tracing.EffectSink, gate ReadGate) error {
    cb := db.Callback()
    // WRITES: always fact-rowed (D-01)
    _ = cb.Create().After("gorm:create").Register("register:create", effectHook(sink, "insert", gate, false))
    _ = cb.Update().After("gorm:update").Register("register:update", effectHook(sink, "update", gate, false))
    _ = cb.Delete().After("gorm:delete").Register("register:delete", effectHook(sink, "delete", gate, false))
    // READS: gated by per-(operation,table) P95 (D-02)
    _ = cb.Query().After("gorm:query").Register("register:query", effectHook(sink, "select", gate, true))
    return nil
}

func effectHook(sink tracing.EffectSink, op string, gate ReadGate, isRead bool) func(*gorm.DB) {
    return func(db *gorm.DB) {
        table := db.Statement.Table
        rows := db.Statement.RowsAffected            // writes: rows changed; reads: rows scanned
        durMS := /* measure: stash start in Before-callback ctx, diff here */ 0
        // operation captured cheaply NOW; func-name resolved on the async sink side (D-11)
        op := tracing.CaptureOperationPCs(db.Statement.Context)
        if isRead && !gate.ShouldRecord(op, table, durMS) {
            return // trivial read — never fact-rowed (D-02/D-04)
        }
        sink.Record(tracing.Effect{
            EffectKind: map[bool]string{true: "db_read", false: "db_write"}[isRead],
            Operation:  op, Target: table, TargetKind: "table",
            Rows: int(rows), DurationMS: durMS, Requests: 1,
            UserID: tracing.UserIDFromContext(db.Statement.Context),
        })
    }
}
```
**GOTCHA (verified):** A query operation *inside* an after-Update callback resets `RowsAffected` to 0. The effect hook must NOT issue any DB query — read `RowsAffected` directly and emit to the in-process sink only. `[CITED: github.com/go-gorm/gorm/issues/7044]`

### Pattern 2: `runtime.Callers` operation resolver (D-08/D-09/D-11)
**What:** Capture program counters cheaply and synchronously at record-time; resolve PC→function name on the async side; pick the nearest frame whose function path contains `/internal/service/`.
**When to use:** For every effect kind (egress retrofit + db + non-aggregated cache).
**Example:**
```go
// Source: pkg.go.dev/runtime#Callers + #CallersFrames [CITED]
// SYNC (cheap): capture up to N PCs, skipping this helper + the hook frame.
func CaptureOperationPCs(ctx context.Context) Operation {
    var pcs [32]uintptr
    n := runtime.Callers(3, pcs[:]) // skip Callers, this fn, immediate caller (the hook)
    return Operation{pcs: pcs[:n], ctx: ctx}
}

// ASYNC (on the Producer side, where CPU is off the hot path — D-11):
func (o Operation) Resolve() string {
    if len(o.pcs) > 0 {
        frames := runtime.CallersFrames(o.pcs)
        for {
            f, more := frames.Next()
            if i := strings.Index(f.Function, "/internal/service/"); i >= 0 {
                return normalizeServiceFrame(f.Function) // → "catalog.UpdateAnimeInfo"
            }
            if !more { break }
        }
    }
    if _, op := ReadBaggage(o.ctx); op != "" { return op } // fallback (D-09)
    return originName(o.ctx)                                // final fallback (D-09)
}
```
**Cost (verified):** `runtime.Callers` is a cheap stack copy (a few hundred ns for ~10-30 frames); the expensive part is `CallersFrames` symbol resolution, which is why D-11 mandates capturing PCs sync and resolving on the async path. `[CITED: pkg.go.dev/runtime]`
**Frame normalization:** GORM-internal frames (`gorm.io/...`) and the project's repo layer (`/internal/repo/`) are NOT service frames — keep walking until `/internal/service/` is found, then strip the module prefix to yield `catalog.UpdateAnimeInfo` / `spotlight.AnimeOfDayResolver.Resolve`. The 7 services all follow the `services/{name}/internal/service/...` layout per CLAUDE.md, so the `/internal/service/` substring is a robust, uniform anchor. `[VERIFIED: CLAUDE.md Go service layout + repo inspection]`

### Pattern 3: Cache aggregator = clone of `HLSSessions` (D-05/D-10)
**What:** An in-process struct holding `map[counterKey]*tally` guarded by a mutex, with a ~10s reaper goroutine that emits summed `cache` effect rows and a `Stop()` that flushes-all on shutdown. The existing `services/streaming/internal/service/hls_sessions.go` is the exact, production-proven template.
**When to use:** Instantiate once per cache-using service; hook `RedisCache.Get`/`Set` to call `Observe`.
**Key reuse:** `HLSSessions` already demonstrates: mutex-guards-map-only (never across IO), record-time attribution capture (`Mint`), bounded map with oldest-eviction (`evictIfFullLocked`), deterministic-clock testing (`now func()`), and a non-blocking sink contract. Copy its shape; swap `sessKey`→`(key_class, result, operation)` and `sessionTally`→`{requests uint32}`.
**D-10 recommendation:** Use the **coarser baggage operation** for aggregated cache rows, NOT a per-op stack-walk. Rationale: keying the counter by a fine stack-frame op would require a `runtime.Callers` walk on EVERY `Get`/`Set` (the hot loop) to compute the map key — directly violating D-11. Baggage op is already in ctx (free to read) and D-06 already accepts that cache rows are not the headline causation signal. Cost tradeoff: fine-op cache would fragment ~25 key-classes × N operations into far more rows for marginal pivot value. **Coarse-for-cache is the correct call.**

### Anti-Patterns to Avoid
- **Stack-walking on the request hot path.** Capturing the *function name* (not just PCs) synchronously on every DB/cache op would add symbol-resolution cost to every request (D-11 violation). Capture PCs sync, resolve async.
- **Querying the DB inside a GORM after-callback.** Resets `RowsAffected` to 0 (issue #7044) and risks recursion (the query itself fires the Query callback). The effect hook must be pure in-memory.
- **Adding new scraper CDNs to the HLS proxy allowlist.** Unrelated to this phase but a standing CLAUDE.md rule — irrelevant here since no proxy changes.
- **Recording analytics' own DB writes.** D-16. The effect hook must be wired into the 7 gormtrace services ONLY, never analytics.
- **One cache row per operation.** D-05 — always aggregate by key-class.
- **Blocking the request on the threshold lookup.** The D-03 threshold table must be a cached in-memory snapshot refreshed periodically, never a synchronous Redis/HTTP call inside the Query callback.

## Common Pitfalls

### Pitfall 1: `RowsAffected` zeroed by a query inside the callback
**What goes wrong:** Issuing any DB read inside the after-Update/Create callback silently resets `db.Statement.RowsAffected` to 0, so every `db_write` row records `rows=0`.
**Why it happens:** GORM reuses the statement's affected-rows tracking across nested operations (issue #7044).
**How to avoid:** The effect hook reads `RowsAffected` and emits to the in-process sink only — zero DB access inside the callback.
**Warning signs:** All `db_write` rows show `rows=0` in ClickHouse.

### Pitfall 2: Stack-walk symbol resolution on the hot path
**What goes wrong:** `runtime.CallersFrames(...).Next()` (name resolution) is materially more expensive than `runtime.Callers` (PC capture). Doing it inline per request adds measurable latency at scale (D-11 risk).
**How to avoid:** Capture `[]uintptr` synchronously in the hook; carry it on the `Effect`/`Operation`; resolve names on the Producer goroutine.
**Warning signs:** P99 latency creep on DB-heavy endpoints after rollout; CPU profile dominated by `runtime.(*Frames).Next`.

### Pitfall 3: Cache aggregator lock held across Redis IO
**What goes wrong:** Holding the aggregator mutex across the actual Redis Get/Set serializes all cache traffic.
**Why it happens:** Naively wrapping the whole cache call in the lock.
**How to avoid:** Mirror `HLSSessions.Observe` — measure the result OUTSIDE the lock, then take the lock only for the map increment (a few ops). `[VERIFIED: hls_sessions.go T-02-LOCK contract]`
**Warning signs:** Redis-bound endpoints slow down under concurrency; mutex contention in profiles.

### Pitfall 4: D-03 threshold lookup made synchronous
**What goes wrong:** Reading the threshold table from Redis/HTTP inside the Query callback adds a network round-trip to every SELECT.
**How to avoid:** Each service keeps an in-memory snapshot of the threshold map, refreshed by a background ticker (e.g. every 5-15 min). The callback consults the snapshot (a map read).
**Warning signs:** Every SELECT shows an extra Redis GET in traces.

### Pitfall 5: ClickHouse measure column is `row_count`, not `rows`
**What goes wrong:** Naming the CH column `rows` breaks the clickhouse-go/v2 native batch binder (it treats `rows` inconsistently).
**Why it happens:** `rows` is reserved-ish in the native protocol.
**How to avoid:** The CH column is already `row_count` (DEFAULT 0). On the Go domain/wire side use field name `Rows`/`RowCount` but JSON tag `row_count`. `[VERIFIED: clickhouse_schema.go:21 + handler/effects.go]`
**Warning signs:** Batch insert errors mentioning column binding.

### Pitfall 6: Tempo metrics_generator silently no-ops without per-tenant enable
**What goes wrong:** Adding the `metrics_generator` block but forgetting the `overrides.defaults.metrics_generator.processors` (or the global `metrics_generator_processors`) list leaves the generator disabled — no metrics appear in Prometheus.
**How to avoid:** Set BOTH the `metrics_generator:` config (ring/storage/remote_write) AND the per-tenant/processor enable list. Single-tenant Tempo uses the `overrides` defaults block.
**Warning signs:** `traces_spanmetrics_*` / `traces_service_graph_*` metrics absent in Prometheus despite traces flowing.

### Daily P95 query (D-03) — runs where the ClickHouse conn lives (analytics)
```sql
-- Source: ClickHouse quantile docs [CITED: clickhouse.com/docs/sql-reference/aggregate-functions/reference/quantile]
SELECT operation, target,
       quantile(0.95)(duration_ms) AS p95_ms
FROM events
WHERE effect_kind = 'db_read'
  AND timestamp >= now() - INTERVAL 1 DAY
GROUP BY operation, target
HAVING count() >= 20          -- ignore tiny-sample keys; they use the static default
```
