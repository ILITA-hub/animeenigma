<objective>
Build Logic B (TRIG-02): when an active JP-audio-combo user starts watching episode N of a `watching`
anime, fire a `next_ep` demand for episode N+1 to the library so the autocache Planner (Plan 09-02)
pre-downloads it ahead of the watcher. The fire point is `player UpdateProgress` (the first progress
heartbeat for N — maximum lead time, per spec §4 / RESEARCH A2 / Open-Q1), NOT MarkEpisodeWatched (too
late). The producer is a verbatim clone of the proven `RecsHintProducer` fire-and-forget pattern:
buffered channel + single worker, 3s timeout, drop-on-full, nil-safe, Start/Stop lifecycle — so a slow
or down library can never block or fail the heartbeat (RESEARCH Pitfall 5: catalog/player WriteTimeout).

JP-audio determination (CONTEXT decision 1 / RESEARCH A1): combo `Player ∈ {ae, raw}` OR `Language == "ja"`.
The gating data (anime `status=watching`, `shikimori_id`, `episodes_aired`) lives in the shared
`animeenigma` DB that player already reads — a single repo lookup keyed by (user_id, anime_id) gives the
watching-status + mal_id + aired-count needed to decide and target the demand.

Purpose: pull the next episode into the pool ahead of an active JP watcher.
Output: `DemandProducer` + the UpdateProgress fire point + a (user,anime,episode) first-heartbeat guard + config/DI.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-08 | Spoofing | client-supplied combo forcing a demand | mitigate | The demand only fires when the anime is `status=watching` for the authenticated user (server-side anime_list lookup) AND N+1<=episodes_aired — a spoofed combo on a non-watched/over-aired episode fires nothing; the library PK-dedup + Planner backoff bound the blast radius |
| T-09-09 | Denial of Service | per-heartbeat demand flood | mitigate | Fire-and-forget drop-on-full (channel cap 256) + autocache_demand composite-PK dedup collapse repeated N+1 wants to one row; optional in-memory once-per-episode guard |
| T-09-10 | Denial of Service | slow/down library blocking the heartbeat | mitigate | Async producer (3s client timeout, returns immediately); the heartbeat never waits on the POST (RESEARCH Pitfall 5) |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs (stdlib net/http + existing logger); RESEARCH audit confirms zero new deps |
</threat_model>
