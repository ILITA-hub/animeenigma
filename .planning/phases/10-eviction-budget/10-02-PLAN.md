## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Evictor → MinIO + Postgres | a sweep/pre-admit race could double-spend freed headroom or delete a row whose objects survived |
| master switch → eviction | a disabled config must halt eviction (POOL-05), not just downloads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04 | Tampering | EnsureRoom vs Sweep concurrency | mitigate | one `e.mu` held across the entire read→evict cycle in BOTH paths (factored reclaim loop) so freed headroom is never double-counted (CONTEXT line 99) |
| T-10-05 | Tampering | evictOne ordering | mitigate | DeletePrefix before DeleteByID; object-delete failure skips the row (continue) — tolerate orphaned objects, never an orphaned serving pointer (CONTEXT lines 56-57) |
| T-10-06 | Elevation of Privilege | master switch bypass | mitigate | runOnce re-reads `cfg.Enabled` live each tick and no-ops the sweep when off (asserted by the disabled-no-op test), matching the Planner's T-09-06 guard |
| T-10-07 | Denial of Service | Fresh row deleted under pressure | mitigate | ListStaleEvictionCandidates (Plan 01) returns ONLY Stale; Classify protects Fresh; a freshly-downloaded autocache row is Fresh ≥auto_fresh_download_days(10d) so it cannot be evicted immediately |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | no package installs — stdlib `sync`/`time` + existing prometheus only |
</threat_model>
