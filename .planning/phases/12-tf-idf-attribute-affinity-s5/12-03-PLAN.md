Per-attribute weights (Decision §A2, sum = 1.00):
- tags 0.30
- studios 0.25 (was 0.20 + producers 0.05; collapsed because Shikimori does not separate them)
- genres 0.15
- demographic 0.10 (sourced from animes.rating per Decision §A3)
- source 0.10 (sourced from animes.material_source)
- type 0.10 (sourced from animes.kind)

Final ensemble weight in handler (spec §13): S5 contributes 0.20 of the final ranking score. Sum of all 5 signals: 0.30 + 0.20 + 0.20 + 0.10 + 0.20 = 1.00.

TF-IDF formula (spec §3.1 — binding):
```
unit(u, anime) = max(duration_watched / 60, 1) minutes  if player ∈ {hianime, consumet, animelib}
              = 1 (integer, episode count)               if player = 'kodik'   ← Kodik fallback

tf(u, attr) = Σ unit(u, anime) for anime in u.history if attr ∈ anime.attrs
              ─────────────────────────────────────────────────────────────
                                 total_units(u)

idf(attr)   = log( total_users / (1 + users_with_any_history_in[attr]) )

affinity(u, attr) = tf(u, attr) × idf(attr)

S5_raw(u, candidate) =
    0.30 × Σ affinity(u, t) for t in candidate.tags
  + 0.25 × Σ affinity(u, s) for s in candidate.studios
  + 0.15 × Σ affinity(u, g) for g in candidate.genres
  + 0.10 × affinity(u, candidate.rating)            ← demographic
  + 0.10 × affinity(u, candidate.material_source)   ← source
  + 0.10 × affinity(u, candidate.kind)              ← type
```

JSONB storage shape (s5_affinity column):
- Keys: "{dim}:{attr_id}" — e.g., "tag:slice_of_life", "studio:1", "genre:42", "rating:pg_13", "source:manga", "kind:tv".
- Values: float64 (the affinity score).
- Cold-start: an empty {} object — same shape as Phase 11's s1_vector cold-start.

Cold-start contract (spec §3.3, Phase 11 precedent):
- User with zero watch_history: S5.Score returns an empty map (not nil, not NaN). Ensemble normalizer treats absent entries as zero — same contract as S1/S2.
- User with sparse history: S5 returns whatever it can compute; missing dimensions for a candidate contribute zero on those dimensions.

IDF caching:
- IDF is global (population-scope, not per-user). Compute once per cron tick across all attributes.
- **Decision (Claude's discretion per CONTEXT.md):** cache IDF in an in-memory sync.Map on the S5Attribute struct, recomputed at the start of each Precompute call. At ~3500 anime × 6 dimensions × ~tens of users, the IDF computation runs in tens of milliseconds — no need for Redis. The cache is implicitly per-process; on multi-replica deploys each replica computes independently and converges to the same values (idempotent).
- Alternative considered: Redis hash with 6h TTL, shared across replicas. Adds Redis dependency to S5 with no measurable performance benefit at current scale; rejected for v1.

JSONB key format (Claude's discretion per CONTEXT.md):
- **Decision: `{dim}:{attr_id}` colon-separated.** Reasoning: matches the spec §4.1 example shape, consistent with Redis key conventions used elsewhere in the codebase. Concrete examples: "tag:slice_of_life", "studio:1", "genre:42", "rating:pg_13", "source:manga", "kind:tv".

Persistence strategy:
- S5.Precompute writes the FULL affinity vector at request time. The Score path reads the full vector once, then computes per-candidate raw scores by traversing the candidate's attributes (tags, studios, genres, rating, source, kind) and summing the per-attribute affinity from the vector.
- Per-candidate complexity: O(C × A_avg) where C is candidate count (~3500) and A_avg is average attributes per anime (~30 tags + ~3 studios + ~3 genres + 3 single-value = ~40). Total: ~140k map lookups per request. Sub-millisecond in Go.

Ensemble integration boundary:
- This plan modifies handler/recs.go to add S5 to the WeightedSignal slice. The ensemble.go and normalize.go files are NOT touched (verified by grep diff). Phase 9 success criterion #4: "Adding a second test signal does not require diff in ensemble.go, normalize.go, or any API handler beyond a one-line registry entry."

Open verification items (decided in plan-phase):
1. Where IDF is computed: inside S5.Precompute, at the start of every per-user precompute. Why not in a shared Orchestrator-level pre-step? Because the Orchestrator interface (signal.go) is intentionally simple — adding "compute IDF first" hooks would require interface changes. At ~10 users running on the 6h cron, recomputing IDF 10 times per cycle wastes ~100ms total per cycle. Acceptable.
2. SQL strategy: one batched query per dimension. For "studios" the query is roughly:
   ```sql
   SELECT s.id AS attr_id,
          SUM(CASE WHEN wh.player = 'kodik' THEN 1 ELSE GREATEST(wh.duration_watched/60, 1) END) AS user_units
   FROM watch_history wh
   JOIN anime_studios as_ ON as_.anime_id = wh.anime_id
   JOIN studios s ON s.id = as_.studio_id
   WHERE wh.user_id = ?
   GROUP BY s.id
   ```
   Repeated for each m2m dimension (genres, tags). For single-value dimensions (rating, material_source, kind), the join collapses:
   ```sql
   SELECT a.kind AS attr_id,
          SUM(CASE WHEN wh.player = 'kodik' THEN 1 ELSE GREATEST(wh.duration_watched/60, 1) END) AS user_units
   FROM watch_history wh
   JOIN animes a ON a.id = wh.anime_id
   WHERE wh.user_id = ? AND a.kind != ''
   GROUP BY a.kind
   ```
3. IDF user-counting: for each dimension, count distinct users who have ANY watch_history row touching that attribute. SQL example for studios:
   ```sql
   SELECT s.id AS attr_id, COUNT(DISTINCT wh.user_id) AS user_count
   FROM watch_history wh
   JOIN anime_studios as_ ON as_.anime_id = wh.anime_id
   JOIN studios s ON s.id = as_.studio_id
   GROUP BY s.id
   ```
   Total user count: SELECT COUNT(DISTINCT user_id) FROM watch_history.
4. Score path persistence: rec_user_signals.s5_affinity is the single source of truth. The Score function does NOT recompute affinity; it parses the JSONB once, builds a flat map keyed by `{dim}:{attr_id}`, then per-candidate looks up the candidate's attributes and sums.

## Trust Boundaries

| watch_history -> S5 SQL aggregate | User-controlled rows; existing trust posture (every other recs signal already reads the same table). The new aggregation logic does not introduce a privilege escalation path. |
| rec_user_signals.s5_affinity JSONB writes | Server-side only; written by the cron + debounced trigger. No user-supplied content; no XSS / injection vector since the keys are derived from server-controlled attribute IDs. |
