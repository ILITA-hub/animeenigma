## Deviations from Plan

### 1. [Rule 1 - Bug Adjacent / Architectural Reconciliation] Handler accepts `mal_id` query param, not `providerID`; added `FindID` to orchestrator

**Found during:** Task 2 implementation — reading the catalog scraper Client wire format.

**Issue:** The plan's Task 2 text says

> Add a query-param parser: providerID = r.URL.Query().Get("providerID") [...] (Note: the catalog already resolves UUID → MAL ID and passes MAL ID as providerID, per services/catalog/internal/service/scraper.go.)

But the actual catalog wire format (`services/catalog/internal/parser/scraper/client.go`) sends `mal_id=<int>`, not `providerID=...`. Worse, the value passed is a MAL ID — not an AnimePahe session ID — and AnimePahe's `ListEpisodes` expects an AnimePahe session ID (passed to `/api?m=release&id=<session>`). A direct pass-through would call `/api?m=release&id=<malid>` and return ErrNotFound.

**Fix:** Two correlated changes:

1. **Handler reads `mal_id` query param** (matches catalog wire format), then builds `AnimeRef{ShikimoriID: malID}` (project memory: Shikimori IDs == MAL IDs) and resolves to a provider-internal ID via the orchestrator.
2. **Added `FindID(ctx, ref, prefer)` to the orchestrator** — wraps the standard `runFailover` loop around each provider's `FindID` method. This keeps provider-specific resolution OUT of the handler and reuses the failover/error-classification machinery for free.

The handler now does `resolveProviderID(malID) → ListEpisodes(providerID)`, two orchestrator hops per request. malsync's 24h positive cache absorbs the FindID cost after the first call.

**Files modified:** `services/scraper/internal/handler/scraper.go`, `services/scraper/internal/service/orchestrator.go`.

**Commit:** `f45f5fd` (Task 2 GREEN).

**Rule applied:** Rule 1 (the literal plan instruction would have produced wrong behavior — FindID is required by the architecture even though the plan text glossed over it).

### 2. [Rule 3 - Blocking] Live smoke test deferred to post-merge deploy

**Found during:** Task 3 execution.

**Issue:** The plan's Task 3 step 3 prescribes `make redeploy-scraper` + a curl roundtrip against a real UUID inside the worktree. As a sequential executor agent in a worktree branch, redeploying the live `animeenigma-scraper` container from this branch (a) requires merging this work first (the deploy operator owns that), and (b) competes with whatever the orchestrator is running on the host docker daemon. Per the plan executor's `parallel_execution` block, runtime `make redeploy-*` is the parent orchestrator's responsibility post-merge.

**Fix:** Documented the exact smoke-test recipe in this SUMMARY (above) so the deploy operator can run it after the merge. Substituted the live test with the docker-compose `config --quiet` validation, which catches structural errors in the env-var block / depends_on additions.

**Files modified:** None. Trade-off recorded here.

**Rule applied:** Rule 3 (blocking — the live deploy step is structurally out of scope for a sequential worktree executor).

### 3. [Rule 2 - Critical] Added `TestScraperHandler_GetEpisodes_MissingMalID` to lock the 400 input-validation contract

**Found during:** Task 2 RED test design.

**Issue:** The plan's eight enumerated test cases include `TestScraperHandler_GetEpisodes_NoProviders` (503 when zero providers registered) but did NOT explicitly call for a test of the "providers registered, but no mal_id query param" path. The handler returns 400 INVALID_INPUT for that case — without a test, a future regression to a silent "return zero episodes" or a 500 would slip through.

**Fix:** Added `TestScraperHandler_GetEpisodes_MissingMalID` as a 9th test case in the same file. Verifies status==400 AND `meta.tried` is still surfaced (per SCRAPER-NF-05 even on a pure input-validation error).

**Files modified:** `services/scraper/internal/handler/scraper_test.go`.

**Commit:** `49c1e62` (Task 2 RED).

**Rule applied:** Rule 2 (critical functionality — input validation + meta.tree-completeness on every error path is a contract, not a feature).
