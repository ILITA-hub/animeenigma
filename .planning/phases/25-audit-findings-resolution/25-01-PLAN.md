<objective>
Rewrite `TestGetStreamWithGate_AdDecoy_Skipped` in `services/scraper/internal/providers/gogoanime/client_gated_test.go` so the parallel-probe race no longer drops the `parser_ad_decoy_total` counter increment before the assertion runs. The audit observed 5/5 reproducible failures on the current shape: the parCancel fires on the first successful probe (earnvids/position-1) BEFORE the AdDecoy probe (streamhg/position-0) has finished its counter Inc, so the assertion that `parser_ad_decoy_total{server=streamhg} == 1` flakes between 0 and 1.

Purpose: Per SCRAPER-HEAL-22 acceptance, the test contract that locks "ad_decoy probes increment both counters even when a faster sibling wins" must be deterministically green. The production code at `client.go:881-887` IS correct — it Incs both counters synchronously when the probe returns ReasonAdDecoy. The race lives entirely in the test's wall-clock-vs-cancellation timing. Fix the test, do NOT add `runtime.Gosched()` / `time.Sleep(50ms)` hacks to production code (per CONTEXT.md D2).

Output: A rewritten test that either (a) swaps server priority order so the AdDecoy server occupies the sequential-remainder branch where parCancel-vs-Inc cannot race, OR (b) adds an explicit synchronization barrier (waitgroup, channel, or test-controllable Incr-completion signal in the FakeProbe) so the test reads the counter only after the Inc has landed. Approach (a) is simpler and preferred — see Task 1 design notes.
</objective>
