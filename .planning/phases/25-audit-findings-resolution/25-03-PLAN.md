<objective>
Fix the silent-200 bug in `services/streaming/internal/handler/stream.go::HLSProxy`. Currently, when `videoProxy.ProxyWithReferer(...)` returns the "domain not allowed for HLS proxy" error (a plain `fmt.Errorf`, NOT an `UpstreamError`), the handler falls through to the generic `log.Errorw(...)` branch which does NOT write an HTTP response. Because no `w.WriteHeader` was called yet, Go's `net/http` defaults to 200 OK with Content-Length:0 — the audit's W-INT-03 finding.

Per CONTEXT.md D3, the fix returns **HTTP 502** specifically (NOT 503, NOT 4xx). Rationale captured in D3:
- 502 ("Bad Gateway") accurately describes the failure ("upstream gave us a URL on a domain we can't proxy").
- 503 implies temporary unavailability (wrong — the domain is permanently outside our allowlist).
- 4xx implies caller error (wrong — the caller passed a legitimate request; the upstream gave us a bad URL).
- Matches the existing UpstreamError path which also returns 502 — keeps observable failure modes consistent.

Purpose: Per SCRAPER-HEAL-24 acceptance, an allowlist gap must surface as a real error in (a) user reports (the FE error boundary in restored EnglishPlayer treats 5xx as a failure to display), (b) Prometheus metrics (the streaming service emits `proxy_upstream_errors` on 502), and (c) the BLK-INT-01 self-heal loop (the canary detects 502 as a failure and the maintenance bot's Pattern 7 dispatcher kicks in to propose an allowlist update). The silent 200 currently breaks all three signal channels.

Output: ~5-10 line patch to `HLSProxy` adding a typed-error check for the domain-not-allowed string OR a sentinel error from `libs/videoutils/proxy.go`. A new unit test asserting 502 for non-allowlisted URLs. `make redeploy-streaming` + curl smoke verifying the response code/body.

Implementation decision: prefer adding a typed sentinel error in `libs/videoutils/proxy.go` (`ErrDomainNotAllowed` or `DomainNotAllowedError` type) over substring-matching the error message. Substring matching is fragile — a future refactor of the proxy library could change the wording and silently break the handler again. The handler should use `errors.Is` / `errors.As`.
</objective>

```go
if err := h.videoProxy.ProxyWithReferer(r.Context(), sourceURL, referer, cw, r); err != nil {
    var upstreamErr *videoutils.UpstreamError
    if errors.As(err, &upstreamErr) {
        metrics.ProxyUpstreamErrors.WithLabelValues(
            strconv.Itoa(upstreamErr.StatusCode),
            upstreamErr.Domain,
        ).Inc()
        h.log.Warnw("upstream CDN error",
            "status", upstreamErr.StatusCode,
            "domain", upstreamErr.Domain,
            "html_response", upstreamErr.HTML,
            "url", sourceURL,
        )
        http.Error(w, "upstream stream unavailable", http.StatusBadGateway)
        return
    }

    var domainErr *videoutils.DomainNotAllowedError
    if errors.As(err, &domainErr) {
        // Allowlist gap — emit observable 502 (per Phase 25 D3 / SCRAPER-HEAL-24).
        // Was previously a silent 200 / Content-Length:0 because this branch
        // fell through to the generic log path below.
        metrics.ProxyUpstreamErrors.WithLabelValues("403", domainErr.Domain).Inc()
        h.log.Warnw("HLS proxy rejected non-allowlisted domain",
            "domain", domainErr.Domain,
            "url", sourceURL,
        )
        http.Error(w, "domain not allowed for HLS proxy", http.StatusBadGateway)
        return
    }

    h.log.Errorw("failed to proxy HLS stream", "error", err, "url", sourceURL, "referer", referer)
    // Belt-and-braces: if we reach here, headers may or may not have been
    // written. Try a final 502 — http.Error is a no-op when headers are
    // already committed, so this is safe in both states.
    http.Error(w, "internal proxy error", http.StatusBadGateway)
}
```

Three substantive changes:
1. NEW: typed-check for `*videoutils.DomainNotAllowedError` → 502 with descriptive body. Emits a `proxy_upstream_errors` metric with status="403" (the upstream domain itself isn't 403'ing — the streaming handler is treating an allowlist-rejected domain as an upstream-403 equivalent for canary signal parity). This guarantees the BLK-INT-01 canary path emits the right reason label downstream.
2. CHANGED: the generic fallback now ALSO emits 502 ("internal proxy error") with `http.Error`. The previous "Don't send error response if headers already sent" comment is misleading — at this code point in `ProxyWithReferer`, error returns mostly happen before any writes (`ProxyWithReferer` returns errors from URL parse, allowlist check, and upstream-request build BEFORE touching `w`). Writing a 502 after `w` has been committed is a no-op in Go's http package (it logs a "superfluous WriteHeader" message but doesn't break the response), so this is safe in both pre-write and post-write states.
3. PRESERVED: `metrics.ProxyUpstreamErrors` increments for both error paths so Grafana dashboards register the failure.

Verify the import block still contains `errors` and `videoutils`. No new imports required (errors.As already used; videoutils already imported for UpstreamError).

Trade-off: the metric label `"403"` for the domain-not-allowed branch is a polite lie — it's the streaming handler's choice to surface allowlist gaps as "looks like a 403 from upstream's POV" so the Pattern 7 dispatcher routes it identically to a real 403_upstream. The alternative is a dedicated `proxy_domain_blocked_total` metric but that requires a new metric registration and Grafana panel — out of scope for Phase 25 (CONTEXT.md "out of scope" excludes dashboards).
