## Threat Model

This plan owns the sidecar threat surface. ASVS Level 1; block-on=high.

| Threat ID | STRIDE | Component | Disposition | Mitigation |
|-----------|--------|-----------|-------------|------------|
| T-27-01-01 | Tampering / Info disclosure | `upstream.js` `fetchWithRetry` | mitigate (HIGH) | Hardcode the upstream base URL to `https://animepahe.pw` in `server.js`; build the URL server-side from validated query params (`q`, `session`, `animeSession`, `episodeSession`). Add a host-allowlist guard in `upstream.js` that rejects any URL whose hostname is not exactly `animepahe.pw` BEFORE calling `page.evaluate(fetch(url))`. Tested by `server.test.js -- host-allowlist` (VALIDATION 27-01-02). |
| T-27-01-02 | Elevation of privilege | Chromium sandbox under `--no-sandbox` | accept (MEDIUM) | Documented in `server.js` header comment + STEALTH-PINS.md: "This sidecar is HARDCODED to `animepahe.pw`. Adding a second upstream requires either sandbox re-enablement with a Docker `cap_add: SYS_ADMIN` grant or explicit security review." Plan-checker should block any PR that adds a second `page.goto` host. |
| T-27-01-03 | DoS | Sidecar OOM (Chromium leak → exceeds 500 MB cap → kills scraper `depends_on` chain) | mitigate (HIGH) | Pattern 3 page-recycle at N=100 fetches (configurable via `PAGE_RECYCLE_AT` env); `--disable-dev-shm-usage` + `--single-process` + `--js-flags=--max-old-space-size=384` Chromium flags; close-first recycle order per Pitfall 4 if 100-request soak peaks > 450 MB RSS. Compose enforces `mem_limit: 500m` (owned by Plan 27-03). |
| T-27-01-04 | Info disclosure | `/metrics` and Pino logs emit DDoS-Guard cookie values | mitigate (MEDIUM) | `metrics.js` exposes only counter shapes (no cookie values as label values); Pino logger is configured with `redact: ['req.headers.cookie', 'res.headers["set-cookie"]']`. `page.cookies()` is NEVER logged or surfaced via HTTP. |
| T-27-01-05 | Repudiation / DoS | Stealth plugin silently defeated by new DDoS-Guard challenge | mitigate (MEDIUM) | `stealth_challenge_failures_total` counter increments on second-403; Pattern 7 in `.claude/maintenance-prompt.md` gains an animepahe-resolver branch (this plan) referencing `STEALTH-PINS.md` refresh procedure; `SCRAPER_DEGRADED_PROVIDERS` env-override escape hatch is preserved (owned by Plan 27-05 default flip; operator can re-add `animepahe` on outage). |

### Task 2: Upstream fetch wrapper + 403 retry + page recycle + metrics

<action>
Create `services/animepahe-resolver/upstream.js` implementing Pattern 2 + Pattern 3 from RESEARCH. Export `fetchWithRetry(url)` which: (1) host-allowlist-guards `url` — parse it, reject with a thrown `ResolverError(400, 'host_not_allowed')` if `hostname !== 'animepahe.pw'` (T-27-01-01, V4 ASVS); (2) calls `page.evaluate(async (u) => { const r = await fetch(u, { credentials: 'include' }); return { status: r.status, body: await r.text() }; }, url)`; (3) on `status === 403` re-navigates the warm page to `https://animepahe.pw/` with `waitUntil: 'networkidle2'`, increments `stealth_challenge_solves_total`, retries the evaluate ONCE; (4) on second `403` increments `stealth_challenge_failures_total` and throws `ResolverError(502, 'stealth_challenge_failed')`; (5) increments `upstream_403_total{stage:'first'|'second'}` on each 403. Implement `maybeRecycle()` per Pattern 3: a module-scoped `requestCount` increments after every `fetchWithRetry`; when `requestCount % PAGE_RECYCLE_AT === 0` (default 100, env-overridable via `PAGE_RECYCLE_AT`), call `recyclePage()` from `browser.js` which closes the old page AFTER the new page completes its `page.goto('https://animepahe.pw/')` warmup (Pitfall 4 — overlap strategy; if 27-01 soak peaks > 450 MB switch to close-first strategy AND lower default to N=50 in `STEALTH-PINS.md`). Increment `page_recycle_total` on each recycle. Export `lastChallengeSolveAt` getter (a module-scoped timestamp updated on each successful `page.goto` after a 403). Also export `getMetricsRegister()` re-export from `metrics.js`. Create `services/animepahe-resolver/metrics.js` with prom-client: register `stealth_challenge_failures_total` (Counter), `stealth_challenge_solves_total` (Counter), `page_recycle_total` (Counter), `upstream_403_total{stage}` (Counter with one label `stage`), and the default `prom-client` process metrics. Wire `server.js` `/healthz` to call `getPage()` + the `page.evaluate(() => 1)` probe per Pattern 4 — return `{ browser: 'up', lastChallengeSolveAt, pageCount: requestCount }` on success, `{ browser: 'down', reason }` with 503 on probe timeout / browser-down. Wire `server.js` `/search`, `/release`, `/play` handlers to construct the hardcoded-base URL and call `fetchWithRetry`; `/search` and `/release` `JSON.parse(body)` and reply JSON; `/play` replies `text/html` with the body verbatim. On `ResolverError(502)`, reply 502; on `ResolverError(400, 'host_not_allowed')`, reply 400 (this path should be unreachable in production since handlers build the URL — it is defense-in-depth per T-27-01-01).
</action>
