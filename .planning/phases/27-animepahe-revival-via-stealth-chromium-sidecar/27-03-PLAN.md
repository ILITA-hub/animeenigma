### Task 1: Add the `animepahe-resolver` service block to `docker/docker-compose.yml`

<action>
In `/data/animeenigma/docker/docker-compose.yml` insert a new service block immediately AFTER the `megacloud-extractor:` block (i.e. after line ~107 — the existing block's `start_period: 30s` line) and BEFORE the `scraper:` block (current line 109). Use this exact shape (matching RESEARCH "Code Examples — Compose service block"):

```yaml
  animepahe-resolver:
    build:
      context: ..
      dockerfile: services/animepahe-resolver/Dockerfile
    container_name: animeenigma-animepahe-resolver
    restart: unless-stopped
    init: true                  # SIGTERM cleanup for Chromium subprocesses
    shm_size: 256m              # paired with --disable-dev-shm-usage; Chrome still wants some shm
    security_opt:
      - seccomp:unconfined      # documented exception — Chrome's internal sandbox needs userns
    mem_limit: 500m             # HARD ship gate per CONTEXT.md D5 + T-27-03-01
    cpus: "0.5"
    environment:
      NODE_ENV: production
      LOG_LEVEL: info
      UPSTREAM_BASE_URL: https://animepahe.pw   # hardcoded in code per T-27-01-01; this env is informational/auditable
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:3000/healthz"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 20s
```

DO NOT declare `ports:` (the sidecar is reachable only inside the docker-compose network — per CLAUDE.md "Service Ports" the sidecar is internal-only). DO NOT declare `cap_add:` of any kind (T-27-03-02). DO NOT add `cap_add: SYS_ADMIN`. The `build.context: ..` is relative to the `docker/` directory (matches the existing `services/scraper` build pattern at line 110–112).
</action>

### Task 3: Cold-compose smoke — verify `depends_on: service_healthy` actually blocks scraper boot

<action>
This is the manual smoke gate from VALIDATION row 27-03-02. Run from `/data/animeenigma`:
1. `docker compose -f docker/docker-compose.yml down animepahe-resolver scraper 2>/dev/null` — ensure both containers are stopped.
2. `docker compose -f docker/docker-compose.yml build animepahe-resolver scraper` — rebuild both images so the new compose block + new scraper config land.
3. `docker compose -f docker/docker-compose.yml up -d animepahe-resolver scraper`
4. IMMEDIATELY (within the first 5–10 seconds) run `docker compose -f docker/docker-compose.yml ps animepahe-resolver scraper` and capture the STATUS column. EXPECTED: `animepahe-resolver` is `health: starting` and `scraper` is `Created` (NOT `Up`). The `depends_on: service_healthy` should hold scraper at `Created` until the resolver passes its healthcheck.
5. Watch the resolver come up: `until docker compose -f docker/docker-compose.yml ps animepahe-resolver | grep -q '(healthy)'; do sleep 5; docker compose -f docker/docker-compose.yml ps animepahe-resolver; done` — should flip green within ~60 seconds (Chromium warmup + first DDoS-Guard solve).
6. Re-check `docker compose -f docker/docker-compose.yml ps scraper` — STATUS should NOW be `Up` (or `health: starting` if its own healthcheck has not yet ticked).
7. Confirm the resolver container has the 500 MB cap actually enforced at runtime: `docker inspect animeenigma-animepahe-resolver --format '{{.HostConfig.Memory}}'` MUST return `524288000` (500 MiB in bytes). If the value is `0` or different, the compose `mem_limit` directive did not apply — STOP and surface the bug.
8. Confirm the scraper container sees the new env: `docker exec animeenigma-scraper sh -c 'env | grep SCRAPER_ANIMEPAHE_RESOLVER_URL'` MUST return `SCRAPER_ANIMEPAHE_RESOLVER_URL=http://animepahe-resolver:3000`.
9. Tail scraper logs for 10s: `docker compose -f docker/docker-compose.yml logs --tail=30 scraper` should show the boot log line `animepahe_resolver_url=http://animepahe-resolver:3000` (added by Plan 27-02 to main.go). It should NOT show `animepahe_base_url=` (the old log key — removed in 27-02).

If any of steps 4 / 7 / 8 / 9 fails, STOP and surface the failure — do NOT proceed to Plan 27-04. The failure modes per CONTEXT.md "Failure-mode catalog": (a) scraper booted before resolver healthy → `depends_on` shape wrong (revisit Task 2); (b) mem cap not enforced → compose `mem_limit` syntax wrong for current Docker version (check `docker compose version` and the documented `mem_limit` vs `deploy.resources.limits.memory` ambiguity — this plan uses the legacy `mem_limit` syntax to match the broader compose file's style, but if it doesn't apply at runtime, fall back to `deploy.resources.limits.memory: 500M` under a `deploy:` block); (c) env not passed → typo in env key name.
</action>
