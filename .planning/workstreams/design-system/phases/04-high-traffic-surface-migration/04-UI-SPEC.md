# Phase 4 вЂ” UI Design Contract

> Visual and interaction contract for the High-Traffic Surface Migration phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Contract Premise вЂ” READ FIRST

This is a **value-preserving migration phase**. The milestone premise is **"zero rendered change."** There is **no new visual design to invent.** Every migrated surface (Home, Browse, Watch/player Г—5, nav/layout, anime detail + children) MUST render **pixel-identically to its current state**, using the already-canonical token system and `@/components/ui` primitives delivered in Phases 1вЂ“3.

The design system is **already locked and canonical** вЂ” this contract codifies it, it does not redefine it:
- `frontend/web/src/styles/DESIGN-SYSTEM.md` вЂ” canonical token reference (read by executor)
- `frontend/web/src/styles/main.css` вЂ” single layered token source of truth (Tier-1/2/3 tokens + `@theme inline` + value-preserving aliases)
- `frontend/web/src/components/ui/` + `index.ts` barrel вЂ” shadcn-vue (Reka UI) primitives, all token-driven

**The acceptance bar is the contract:** (1) the token/primitive system the migrated surfaces MUST bind to, (2) the "zero rendered change" bar, (3) the standing 5-surface in-browser visual-regression smoke set at desktop + mobile widths (DS-NF-06 вЂ” jsdom cannot catch cascade/render regressions).

Color в†’ token mapping is **pre-validated and locked** by DS-MIGRATE-02. Do NOT treat any of it as an open design question.

## Token Binding Contract (the migration target)

Migrated surfaces bind to canonical tokens via Tailwind utilities generated from `@theme inline`. The locked off-palette в†’ semantic-token mapping (DS-MIGRATE-02), applied **per-occurrence with a semantic judgment**:

| Off-palette source | Canonical token | Utility examples |
|--------------------|-----------------|------------------|
| `red-*` | `--destructive` (+ `-soft`/`-foreground`) | `bg-destructive`, `text-destructive`, `bg-destructive-soft` |
| `amber-*` / `yellow-*` | `--warning` (+ `-soft`/`-foreground`) | `bg-warning-soft`, `text-warning` |
| `emerald-*` / `green-*` | `--success` (+ `-soft`/`-foreground`) | `bg-success-soft`, `text-success` |
| `blue-*` / `sky-*` | `--info` (+ `-soft`/`-foreground`) | `bg-info-soft`, `text-info` |
| `purple-*` / `violet-*` | `--brand-violet` | `bg-brand-violet`, `text-brand-violet` |
| `gray-*` / `slate-*` / `zinc-*` | `--muted` / `--card` / `--border` (judge per role) | `bg-muted`, `bg-card`, `border-border`, `text-muted-foreground` |

**Hardcoded hex (DS-MIGRATE-03):** Replace `#hex` in `.vue` with the nearest canonical token. If a value is **legitimately novel** (no token within tolerance вЂ” e.g. `BrandMark.vue` brand glow, `SubtitleOverlay.vue` render values), add a new token to `main.css` rather than inlining. Per-case judgment, documented in the commit.

**Deprecated aliases (DS-MIGRATE-04, partial this phase):** Repoint `var(--ink)` / `var(--accent)` / `var(--pink)` brand usages found in the high-traffic files to canonical token names. **The `--accent` semantic flip itself stays in Phase 5 вЂ” do NOT flip `--accent` here.** Phase 4 only repoints usages.

**Primitive adoption (DS-MIGRATE-06, partial this phase):** Replace hand-rolled buttons / cards / badges on these surfaces with the `@/components/ui` primitive where an equivalent exists. Leave bespoke structures with no primitive analog (player chrome, spotlight frame, etc.) structurally intact вЂ” they still get token-only colors.

**Button variant map (locked, Phase 2):** `primary в†’ default` (cyan), `secondary в†’ brand` (pink, NOT shadcn secondary), `ghost в†’ ghost`, `outline в†’ outline`, plus `destructive`. Sizes `sm / md / lg / icon`.
