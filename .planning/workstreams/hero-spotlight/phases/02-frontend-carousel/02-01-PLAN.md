1. Declares three refs: `const cards = ref<SpotlightCard[]>([])`, `const loading = ref(true)`, `const error = ref<Error | null>(null)`.
2. Defines `async function refresh()` that:
   - Sets `loading.value = true`, `error.value = null`
   - Calls `apiClient.get<SpotlightResponse | { data: SpotlightResponse }>('/home/spotlight')`
   - Defensively unwraps: `const body = (res.data as { data?: SpotlightResponse })?.data ?? (res.data as SpotlightResponse)`
   - Assigns `cards.value = Array.isArray(body?.cards) ? body.cards : []`
   - On catch: `console.warn('[spotlight] fetch failed', e); error.value = e instanceof Error ? e : new Error('spotlight fetch failed'); cards.value = []`
   - `finally { loading.value = false }`
3. Calls `onMounted(refresh)` (single fetch on mount; no auth-watcher in Phase 2 per CONTEXT.md "Phase 3 will add it together with the 30s interval").
4. Returns `{ cards, loading, error, refresh }` — names MUST match exactly so Plan 03 can destructure them.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser ↔ gateway | All `/api/home/spotlight` requests; existing apiClient interceptor handles JWT/X-Anon-ID — no new boundary in this plan |
| build-time env ↔ runtime | `VITE_HERO_SPOTLIGHT_ENABLED` baked at build time by Vite; the runtime never sees raw env strings — Vite replaces `import.meta.env.VITE_HERO_SPOTLIGHT_ENABLED` with the literal string at compile time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-01 | Tampering (flag bypass) | `VITE_HERO_SPOTLIGHT_ENABLED` | accept | The flag is a UI kill switch, not a security boundary. Backend `SPOTLIGHT_ENABLED=false` (Phase 1) returns 404 — that is the real off switch. Frontend flag is operational convenience. Documented as such in .env.example comment. |
| T-2-02 | Denial of Service (5xx) | useSpotlight error path | mitigate | `cards.value = []` on catch silently hides the block; one `console.warn` for observability; no toast/banner that could be exploited. Plan 06 e2e asserts the silent self-hide. |
| T-2-03 | Information disclosure | apiClient interceptor stack | mitigate | Composable goes through `apiClient` (research anti-pattern #1) — JWT refresh, X-Anon-ID, prefs cache all apply. A bypass via raw `fetch()` would skip those; grep-gated in acceptance criteria. |
</threat_model>
