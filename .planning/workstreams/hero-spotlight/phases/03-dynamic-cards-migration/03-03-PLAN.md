<objective>
Build the 5 new spotlight resolvers — `personal_pick`, `telegram_news`, `now_watching`, `not_time_yet`, `continue_watching_new` — each implementing the existing `spotlight.Resolver` interface (`Type()` + `Resolve(ctx, userID *string) (*Card, error)`). Each mirrors Phase 1's discipline:
- Manual `cache.Get` + `cache.Set` using `errors.Is(err, cache.ErrNotFound)` (DELIBERATE DIVERGENCE 1 stays in force)
- `(nil, nil)` for ineligible (empty data) — no log emitted
- `(nil, err)` for failures — aggregator emits `spotlight.card_failed{type, error}`
- `(*Card, nil)` for success

Three resolvers (`personal_pick`, `telegram_news`, `now_watching`) handle BOTH anon and login users. Two (`not_time_yet`, `continue_watching_new`) are login-only and return `(nil, nil)` when userID is nil.

Adaptive 1-2-3 rule (`spotlight.AdaptiveSlice` from Plan 02) is applied at the resolver layer for the 4 multi-item resolvers.

Purpose: These 5 resolvers are the actual product value of Phase 3. Plan 04 wires them into the aggregator + retrofits `latest_news` to use AdaptiveSlice. Plan 05 builds the matching frontend cards.

Output:
- 5 new resolver files + 5 test files under `services/catalog/internal/service/spotlight/cards/`
- 5 new Data structs in `services/catalog/internal/service/spotlight/types.go` extending the Card union
</objective>

SQL for now_watching (verbatim from REQUIREMENTS.md HSB-BE-22):
```sql
SELECT DISTINCT ON (wp.user_id)
       u.username, u.public_id, a.id, a.name, a.name_ru, a.poster_url,
       wp.episode_number,
       to_char(wp.updated_at, 'YYYY-MM-DD"T"HH24:MI:SS"Z"') AS updated_at
FROM watch_progress wp
JOIN users u  ON u.id = wp.user_id
JOIN animes a ON a.id = wp.anime_id
WHERE wp.updated_at > NOW() - INTERVAL '5 minutes'
ORDER BY wp.user_id, wp.updated_at DESC
LIMIT 5;
```
