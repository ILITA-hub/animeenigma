## Definition of Done — re-stated from PROJECT.md

> A user with 3-8 ongoing shows in their `anime_list.status = 'watching'` sees a bell badge + a toast (if foregrounded) within ~1 hour of a new episode landing on the same player/translation they were already watching with. Click → straight to the right episode. Engine generic enough that adding `new_comment` or `system_announcement` in v1.1 needs only a payload schema + a frontend renderer component.

All three structural elements of the DoD are in place:

- ✅ **Sensing layer** — hourly `NewEpisodeDetectorJob` cron with ±5min jitter (Phase 2). Catalog `/internal/anime/{id}/episodes` with 5min Redis cache, kodik + animelib parsers wired (Phase 2).
- ✅ **Engine layer** — generic `user_notifications` table with `type` + `dedupe_key` + `payload jsonb` columns; two partial indexes (`uk_user_dedupe`, `idx_user_unread`) on `WHERE dismissed_at IS NULL` so UPSERT semantics work cleanly (Phase 1). `services/notifications/` on port 8090, 6 public REST routes behind JWT, 1 internal producer endpoint behind gateway-non-routing (Phase 1).
- ✅ **UX layer** — Pinia store with 60s polling (paused on `document.hidden`), NotificationBell + Dropdown + Toast + NewEpisodeCard + type-pluggable renderer registry + i18n in 3 locales, mounted in Navbar + App.vue (Phase 3).
- ✅ **Type-pluggability** — `frontend/web/src/lib/notification-renderers.ts` is the SOLE registry; adding `new_comment` in v1.1 needs ONLY a new payload contract + a new renderer component. Backend already accepts arbitrary `type` strings; the unique partial-index is keyed on `(user_id, dedupe_key)` not `type`.

## Tech Debt — 2 items (non-blocking)

These are flagged for v1.0.x / v1.1 follow-up. Not blockers for shipping v1.0.

### TD-NOTIF-01 — `notifications_parser_failures_total` only emits after first `.Inc()`

**Where:** `services/notifications/internal/metrics/...` (Phase 2)
**What:** Prometheus `CounterVec` doesn't expose label series until first observation. Until a parser fails for a given `{player}` label, the metric line is absent from `/metrics`. All other 5 metrics emit eagerly via pre-registration.
**Impact:** Grafana panel for "parser failures by player" will show "No data" instead of "0" until a real failure occurs. Cosmetic, not functional.
**Mitigation:** v1.1 — call `.WithLabelValues("kodik").Add(0)` and `.WithLabelValues("animelib").Add(0)` at boot in `metrics/init.go` to force-register the label series. 2-line change.

### TD-NOTIF-02 — AnimeLib `LatestEpisodeForTeam` is N+1 HTTP calls

**Where:** `services/catalog/internal/parser/animelib/client.go` (Phase 2)
**What:** AnimeLib's API doesn't expose a single "latest episode for team X" endpoint. The adapter walks the episode list calling `GetEpisodeStreams` per episode to find the highest one matching the team_id. Documented as accepted in 02-PLAN.md risk R-02-06.
**Impact:** Per-combo detector call for AnimeLib costs O(episodes) upstream HTTP requests. At 5-worker concurrency × ~12 episodes per ongoing show × 6 active AnimeLib combos = ~360 upstream requests per hour. Acceptable for current load but visible in catalog logs.
**Mitigation path:** v1.0.x — investigate AnimeLib's filtered episode endpoint (`?team_id=X` if it exists), or cache the `team_id → max_episode` map per anime in Redis with a 15min TTL. Either is a localized improvement; the EpisodeChecker interface keeps the change scoped to the adapter.
