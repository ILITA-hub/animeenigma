# Phase 01 (workstream `social`) — UI Design Contract

> Visual and interaction contract for the Reviews + Comments tabbed UI on `frontend/web/src/views/Anime.vue`. Generated by `gsd-ui-researcher`. The scope is **local** to one section (lines 590–715) — no new pages, no new routes, no new top-level components. Reuses existing `components/ui/Tabs.vue` and the project's "Neon Tokyo" design tokens defined in `frontend/web/src/styles/main.css`.

---

## Scope (this phase only)

This phase adds a single visual surface change inside the existing Anime detail page:

- The existing **Reviews section** (`Anime.vue` L590–715) is wrapped in a `<Tabs>` strip with two tabs: `Reviews ({count})` and `Comments ({count})`.
- Reviews tab content is **unchanged** — exact same write-form, star rating, login prompt, and list rendering already in place.
- Comments tab content is **new** — auth-gated textarea, Post button, paginated newest-first list (50/page), inline edit, soft-delete with `window.confirm`.
- Tab state is URL-persisted via `?ugc=reviews|comments` (default: `reviews`).

Out of scope for this UI contract: anything outside the Reviews section of Anime.vue. No other view, route, modal, or layout changes.

## Interaction Contract

### Tab strip

- Default tab: **Reviews**. First paint with no `?ugc=` query param renders Reviews tab.
- `?ugc=comments` on first paint → Comments tab active. `?ugc=reviews` or any other value → Reviews tab active.
- Tab click → `router.replace({ query: { ...route.query, ugc: <next> }})` (no history entry — Back button skips through tabs and lands on the previous page).
- Other query params on the route (e.g. `?ep=3`) are preserved across tab switches.
- Keyboard: `Tab` lands on the active tab button, `←` / `→` arrow keys move between tabs (inherit from `Tabs.vue` `role="tablist"` semantics — note: existing `Tabs.vue` does **not** ship arrow-key handling; this phase does **not** add it either to keep scope minimal; defer to a separate a11y pass).
- Active tab visual: 2px cyan underline + cyan label color (inherited from `Tabs.vue:87`).
- Focus ring: 2px cyan, 4px base offset — inherited globally from `:focus-visible` at `main.css:91-94`.

### Comments write form

- **Logged-in user:** glass-card with a single `<textarea rows="3">` (smaller than review textarea's `rows="4"` because comments are conversational), placeholder = `anime.ugc.commentPlaceholder`, char counter `{n}/2000` below-right of the textarea in `text-white/40 text-sm`, turns `text-pink-400` when `n > 2000`. Post button is disabled when the trimmed body is empty or exceeds 2000 chars.
- **Logged-out user:** glass-card centered with copy `anime.ugc.loginToComment` and a `<router-link to="/auth">` button styled identically to the existing review login button (L671–676), label = `nav.login`.
- Post button label: `anime.ugc.postComment` while idle, `anime.ugc.posting` while submitting (disabled + 50% opacity).
- On success: textarea clears, list refetches its first page (the new comment appears at the top because the list is newest-first), no toast (matches the current review-submit UX which also has no toast).
- On 429 (rate-limit): show `text-pink-400 text-sm mt-2` line below the button with `anime.ugc.rateLimitError`. Do not clear the textarea (user may want to retry later).
- On 400 (validation): inline `text-pink-400 text-sm mt-2` with `anime.ugc.bodyTooLong` or `anime.ugc.bodyEmpty` depending on which guard fired.

### Comments list

- Sort: newest first. Date format: relative via existing `formatDate()` helper.
- One card per comment, `glass-card p-4 space-y-3` stack. Inside: header row (avatar + author link + timestamp on the left, action icons on the right if it's the user's own comment), body paragraph below.
- Action icons (own-comment only, all three states — visible when row is hovered on desktop, always visible on touch / mobile):
  - **Edit** (pencil SVG): `text-white/40 hover:text-cyan-400 p-2 rounded-lg` — clicking swaps the body `<p>` for an inline `<textarea>` + Save (cyan-500 bg, same as Post) + Cancel (pink-500/20 wash, same as the existing delete-review button).
  - **Delete** (trash SVG): `text-white/40 hover:text-pink-400 p-2 rounded-lg` — clicking triggers `window.confirm(t('anime.ugc.deleteCommentConfirm'))`; on confirm, call DELETE endpoint and optimistically remove from the local list. On error, restore the card and show `text-pink-400 text-sm` error toast under the list header.
- On edit save → optimistic update of the card body, textarea collapses back to `<p>`. On error → keep the textarea open and show inline error.
- Admin user (`authStore.isAdmin`) sees the trash icon on every comment — but **not** the pencil icon (admins do not edit other users' comments, only soft-delete). The trash icon position and style is identical to the user's own delete icon.

### Pagination

- 50 comments per page, cursor-based (opaque cursor string from API).
- Initial load: `GET /api/anime/:id/comments?limit=50` (no cursor).
- "Load more" button at the bottom of the list **only when** `hasMore === true` (signaled by API returning a `next_cursor` field).
- Button label: `anime.ugc.loadMore` while idle, `anime.ugc.loading` while fetching, disabled + 50% opacity during fetch.
- New page is appended to the existing array — do not replace. Scroll position preserved.
- After posting a new comment: the list refetches **only the first page**; existing loaded pages stay (acceptable race: a freshly-posted comment may also appear via the refetch, but server is the source of truth so dedup by `id`).

### Empty states

- No comments yet: `glass-card p-8 text-center` with `text-white/50` text → `anime.ugc.emptyComments` ("No comments yet. Start the conversation."). Identical visual treatment to the existing `noReviews` empty state at L712.
- Anonymous + no comments: still show the empty state below the login prompt. Logged-in + no comments: show the empty state below the write form.

### Loading states

- First paint of Comments tab: render the write form (or login prompt) immediately; show a single skeleton card (3 grey-pulse glass-cards using `Skeleton.vue`) below it while the initial GET resolves. Matches the existing skeleton pattern used in `Skeleton.vue` elsewhere in the view.
- "Load more" fetching: button shows `anime.ugc.loading`. No additional skeletons.
- Post submitting: button shows `anime.ugc.posting`, button disabled. Textarea stays interactive (so user can copy text out if needed) but blurred-disabled visually via `opacity-50 pointer-events-none`.

### Error states

| Source | Visual | Copy key |
|--------|--------|----------|
| Initial GET fails | Glass-card with red-tint border + retry button | `anime.ugc.loadFailed` + `common.retry` |
| POST 429 | Inline `text-pink-400 text-sm mt-2` below Post button | `anime.ugc.rateLimitError` |
| POST 400 (empty) | Inline `text-pink-400 text-sm mt-2` below Post button | `anime.ugc.bodyEmpty` |
| POST 400 (too long) | Inline `text-pink-400 text-sm mt-2` below Post button | `anime.ugc.bodyTooLong` |
| PATCH fails | Inline error under the still-open edit textarea | `anime.ugc.editFailed` |
| DELETE fails | Restore the card + `text-pink-400 text-sm` error toast under list header (auto-dismiss 5s) | `anime.ugc.deleteFailed` |
| Load-more fails | Replace button label with `text-pink-400 text-sm` retry hint | `anime.ugc.loadMoreFailed` |

**Tone rules (applies to JA / RU translations):**
- Use the same register as existing `anime.*` strings — e.g. `anime.noReviews` ("No reviews yet. Be the first!") sets the tone. Match its energy in `anime.ugc.emptyComments`.
- Russian: use formal `Вы` form, matching the existing locale (e.g. `anime.loginToReview` = "Войдите, чтобы оставить отзыв"). Avoid colloquialisms.
- Japanese: polite (です/ます) form. Match existing pattern in `ja.json`.
- Do not over-localize: leave the char counter as `{count}/2000` literally — numbers don't need translation.

**Destructive copy contract:**
- The only destructive action in this phase is **delete a comment**. Confirmation runs via the native `window.confirm()` dialog (per CONTEXT.md decision; mirrors the existing review-delete UX which uses the same pattern at L660). Copy must include both "Delete this comment?" and "This cannot be undone." in a single sentence — the native dialog has no body/title split.

## URL & State Contract

| Concern | Behavior |
|---------|----------|
| First paint, no `?ugc=` | Reviews tab active. URL **not** rewritten — Reviews is the implicit default. |
| First paint, `?ugc=comments` | Comments tab active. URL preserved as-is. |
| First paint, `?ugc=reviews` | Reviews tab active. URL preserved as-is (explicit default is allowed). |
| First paint, `?ugc=garbage` | Reviews tab active (fallback for any unknown value). URL **not** rewritten — leave the bad value alone so the user / external link doesn't get clobbered on first paint, but switching tabs **does** normalize it. |
| Click Reviews tab when already on Reviews | No-op — no URL push. |
| Click Comments tab from Reviews | `router.replace({ query: { ...route.query, ugc: 'comments' }})`. |
| Click Reviews tab from Comments | `router.replace({ query: { ...route.query, ugc: 'reviews' }})` (sets explicit value, so the URL is reversible/shareable). |
| Browser back button | Goes to the previous **page** (e.g. Home), not the previous tab — because we use `router.replace`, not `router.push`. Per CONTEXT.md. |
| Page reload | `route.query.ugc` is the source of truth; tab restores to whatever was last active. |
| Other query params (e.g. `?ep=5`) | Preserved on every tab switch via `...route.query` spread. |
| Deep link from external site | `/anime/<id>?ugc=comments` works first-paint, no flicker (initial tab state is set synchronously in the component setup, before the first render). |

**Implementation hint (not a contract requirement, but the cleanest path):** keep a local `ref('reviews' | 'comments')` initialized from `route.query.ugc`, watch the ref to call `router.replace`, and watch `route.query.ugc` to update the ref. Two-way binding via `v-model` on `<Tabs>`.

## Activity Feed (out of scope for this UI contract)

`ActivityFeed.vue` will receive new `type='comment'` events automatically from the existing feed endpoint. Per CONTEXT.md, the visual treatment of comment events in the activity feed is **deferred** — for v0.1, comment events appear via the existing generic event renderer (whatever fallback `ActivityFeed.vue` has for unknown types). A dedicated icon + label for comment events is a follow-up phase. This UI contract does NOT specify the activity-event UI; the implementer should add no special-casing unless `ActivityFeed.vue` lacks a fallback path, in which case mirror the existing `review` event treatment with the comment icon (chat-bubble SVG) and a label like "commented on {anime}".
