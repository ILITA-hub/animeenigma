# Phase 01 Plan 01.2: Redis Repo Summary

**One-liner:** Redis persistence layer with centralized `wt:` key builders, atomic sliding-TTL refresh via TxPipelined, capped-list chat (LPUSH + LTRIM 0 99), JSON-encoded members HASH, and PubSub primitives for forward-compat multi-instance fanout — all covered by 16 miniredis tests passing under `-race`.

## What Was Built

### `internal/repo/keys.go`

4 exported key builders + 1 const, the ONLY place that emits a `wt:`-prefixed
string in the watch-together service. Downstream packages (hub, handlers,
service layer) MUST route through these helpers — enforced as a grep gate.

```go

```

## Test Approach: miniredis (not testcontainers)

**Choice:** `github.com/alicebob/miniredis/v2` — in-process pure-Go Redis fake.

**Rationale:**

| Criterion | miniredis | testcontainers/redis |
|---|---|---|
| Startup time | ~1ms | ~500ms-2s (Docker pull/start) |
| Docker dep | No | Yes |
| TTL fast-forward | `mr.FastForward(t)` (native) | Real-time wait (~unusable) |
| PubSub support | Yes (v2.30+) | Yes |
| Already in workspace | Yes (`services/gateway`) | Yes (`services/notifications` — Postgres only) |
| CI parallel-friendly | Yes (`miniredis.RunT(t)` auto-cleans) | Container-port collisions |

`mr.FastForward(t)` is the killer feature — `TestUpdateRoomState_PartialUpdateRefreshesTTL`
proves the sliding-TTL guarantee in <1ms by advancing the miniredis clock 100s
then re-checking the TTL. The equivalent testcontainers test would either need
to wait 100s of real time or skip the TTL-drift assertion entirely.

`services/notifications` uses testcontainers but only because it needs real
Postgres (GORM + datatypes.JSON columns). For pure-Redis testing miniredis is
strictly better.

## Deviations from Plan

### Auto-fixed (Rule 2 — added critical functionality not in plan)

**1. [Rule 2 — Defensive] Refactored 3 `pipe.Expire` calls into `expireAll` helper.**
- **Found during:** Task 01.2.2 implementation
- **Issue:** The plan's literal grep gate (`grep -c "pipe\\.Expire" ... >= 5`) would have been satisfied by inlining 3 `pipe.Expire` calls into each of 5+ methods (15-18 raw lines). That's WET code and a future-bug hazard: a new state-mutating method added in 01.5 could easily forget one of the 3 EXPIREs.
- **Fix:** Extracted `expireAll(pipe, ctx, roomID)` private helper that always pipelines all 3 EXPIREs. Every state-mutating method calls `r.expireAll(pipe, ctx, roomID)` inside its TxPipelined block.
- **Files modified:** `services/watch-together/internal/repo/redis_repo.go` (lines 105-113)
- **Semantic equivalence:** The behavioral contract ("every state-mutating method refreshes TTL on all 3 keys atomically") is identical — and now physically un-skippable.
- **Commit:** `0a4a391`

**2. [Rule 2 — Defensive] `UpdateRoomState` field-name allowlist check.**
- **Found during:** Task 01.2.2
- **Issue:** Plan said "Unknown field → return errors.New("invalid field name")". I treated this as a security/correctness requirement and made it pre-flight (validate BEFORE TxPipelined), so a typo'd field name returns an error without any Redis side-effects. Test `TestUpdateRoomState_RejectsUnknownField` enforces this.
- **Files modified:** `services/watch-together/internal/repo/redis_repo.go` (`allowedRoomFields` map + check in `UpdateRoomState`)

**3. [Rule 2 — Defensive] Tolerant per-row JSON decode in `ListMembers` + `GetMessages`.**
- **Found during:** Task 01.2.2 implementation
- **Issue:** If a single corrupt JSON entry exists in `wt:room:{id}:members` or `wt:room:{id}:messages`, the strict approach would fail the whole call.
- **Fix:** Skip-and-log the corrupt row (warn-level) and return the rest. AddMember/AppendMessage are the only writers so this should never fire in practice — it's purely defensive against future schema drift.

### Plan check note (semantically equivalent, textually different)

The plan check `grep -c "pipe\\.Expire" services/watch-together/internal/repo/redis_repo.go >= 5` literally counts to 3 (because of the helper). The behavior the check was intended to verify ("TTL refresh appears in CreateRoom + UpdateRoomState + AddMember + RemoveMember + AppendMessage") is satisfied by `expireAll` being called from 6 sites (`grep -c "r.expireAll" → 6`). Documented here for the next reviewer.

## How 01.3 / 01.4 / 01.5 Will Consume This

- **01.3 (hub):** `RoomRepo.Subscribe(ctx, roomID)` → `*redis.PubSub`. Hub owns the lifecycle (subscribe on first member-in-room, close on last-out). Publish path: `repo.Publish(ctx, roomID, eventJSON)` — payload is the WS envelope marshalled.
- **01.4 (REST handlers):** `POST /rooms` → `CreateRoom`. `GET /rooms/{id}` → `GetRoom` + `ListMembers` + `GetMessages(50)`. `DELETE /rooms/{id}` → host check via `GetRoom.HostUserID` then `DeleteRoom`. 404 on `errors.Is(err, ErrNotFound)`.
- **01.5 (WS handler):** Upgrade path → `Exists` + `CountMembers` (capacity gate at 10) + `AddMember` (with JoinedAt). Inbound `playback:*` → `UpdateRoomState({"playback_state": ..., "playback_time": ..., "playback_time_updated_at": ...})`. Inbound `chat:message` → `AppendMessage`. Inbound `presence:heartbeat` → `RefreshTTL` + `AddMember` (with bumped LastSeenAt — same user_id, overwrites). Disconnect → `RemoveMember`.
- **01.6 (drift detection):** Pure function consumes `Room.PlaybackTime` + `Room.PlaybackTimeUpdatedAtMs`; emits corrections via the hub. Updates on echo via `UpdateRoomState`.
