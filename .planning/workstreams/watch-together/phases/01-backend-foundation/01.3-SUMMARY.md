# Phase 01 Plan 01.3: WebSocket Hub Summary

One-liner: per-room WebSocket connection registry with broadcast / per-recipient / pubsub fanout semantics, race-detector clean, fully unit-testable without spinning real websockets.

## Hub Public Interface (locked — downstream contracts in 01.4–01.6 build against this)

```go
// Constructor + lifecycle
func NewHub(repo *repo.RoomRepo, log *logger.Logger, instanceID string) *Hub
func (h *Hub) Close()

// Connection registry
func (h *Hub) Register(roomID, userID, username string, conn *websocket.Conn) (*Connection, error)
func (h *Hub) Unregister(c *Connection)
func (h *Hub) MemberCount(roomID string) int
func (h *Hub) MemberUserIDs(roomID string) []string

// Fanout
func (h *Hub) Broadcast(ctx context.Context, roomID string, env domain.Envelope, excludeUserID string) (int, error)
func (h *Hub) SendTo(ctx context.Context, roomID, userID string, env domain.Envelope) (int, error)

// Telemetry / introspection
func (h *Hub) InstanceID() string

// Connection (exported fields are immutable after Register)
type Connection struct {
    UserID, Username, RoomID string
    OnMessage func(*Connection, domain.Envelope) // set by 01.5/01.6 wiring
    // unexported: conn (wsConn), sendCh, hub, log, closed, closeOnce
}
func (c *Connection) Send(payload []byte) bool
func (c *Connection) Close()
```

## The fakeConn Test Pattern

Real WebSocket upgrades are messy in unit tests (httptest server, real TCP, race-prone teardown). Instead `connection.go` types the transport as a `wsConn` interface — the narrow subset of `*gorilla/websocket.Conn` the hub actually uses — and the production constructor accepts any value satisfying it.

`hub_test.go` defines a `fakeConn` that:

- Records every `WriteMessage(payload)` to a mutex-guarded slice
- Parks `ReadMessage()` on `<-closed` so the readPump doesn't tear down the connection during a test
- Counts `WriteControl` calls (ping/close frames)
- Has an idempotent `Close()` that releases ReadMessage

A `registerFake(t, hub, room, userID, username)` helper attaches a fakeConn through the package-private `Hub.registerInternal` seam (which takes `wsConn`, whereas the exported `Hub.Register` is locked to `*websocket.Conn` per the 01.3 contract). The public API surface stays clean; tests reach behind only via package-private helpers.

The pubsub round-trip test (Behavior 10) uses a real `redis.Subscribe` on a parallel client pointed at the same miniredis instance, so the production code path (`repo.Publish` → Redis SUBSCRIBE) is exercised end-to-end without the hub's own subscriber loop confusing the test.

The own-echo dedup test (Behaviors 11 + 12) bypasses Redis entirely and synthesizes pubsubFrame payloads directly into the exported `Hub.handlePubsubMessage` — testing only the dedup logic in isolation, since Behavior 10 already covers the Redis path.

## Race-Detector Pass Count

`go test ./internal/hub/... -count=2 -race` — clean. All 15 cases pass on both runs. The `TestHub_ConcurrentSafety` case explicitly stresses concurrent Register/Unregister/Broadcast across 16 goroutines × 50 operations per goroutine; the race detector flagged exactly one issue during development (close-of-sendCh racing with concurrent Send), which was fixed by adopting the "signal-channel-only" termination pattern (close `closed`, never close `sendCh`).

## Connection Interface Amendments

Two amendments to the contract sketched in the plan's `<hub_contract>` block were needed during implementation:

1. **`wsConn` includes `SetReadLimit(int64)`.** The plan's interface didn't include it, but the readPump uses `c.conn.SetReadLimit(maxMessageSize)` to cap inbound frames at 64 KiB (gorilla canonical pattern). Adding it kept `*gorilla/websocket.Conn` satisfying the interface without changes.

2. **`Connection.Close` does NOT close `sendCh`.** The plan implied a typical "close the channel to signal writePump exit" approach, but that races with `Hub.Broadcast → Connection.Send → sendCh <- payload` under concurrent Unregister. The race detector caught this immediately. The fix: rely solely on the `closed` signal channel — Send selects on `<-c.closed` and returns false; writePump selects on `<-c.closed` and exits. Buffered messages already queued in sendCh are simply abandoned when both pumps exit, which is the desired behavior anyway (a closed connection can't deliver pending writes).

Both amendments are local to the hub package and don't affect any downstream plan's public-API expectations.

**1. [Rule 3 — Blocking issue] Pinned `google.golang.org/genproto` post-split version**

- **Found during:** Initial `go build` after adding `gorilla/websocket`
- **Issue:** `go.work` workspace mode pulls a pre-split monolithic `google.golang.org/genproto v0.0.0-20190425155659-357c62f0e4bb` onto the import graph via `services/library`'s transitive deps. That module exposes `googleapis/rpc/status` at the same path as the modern split `google.golang.org/genproto/googleapis/rpc` (which `libs/errors` uses), and the Go loader emits `ambiguous import: found package google.golang.org/genproto/googleapis/rpc/status in multiple modules` at `grpc@v1.77.0/status/status.go:35`. The error blocks every `go build` / `go test` invocation against the watch-together module.
- **Fix:** Added `require google.golang.org/genproto v0.0.0-20240528184218-531527333157` to `services/watch-together/go.mod`. The May-2024 release is the first one published after the repo split that omits the moved-out subpackage entirely, so it cannot collide with the split module at the same import path.
- **Scope:** Only `services/watch-together/go.mod` and `go.sum` are touched. No other service's go.mod / go.sum / go.work.sum was modified (verified via `git checkout` of all transitively-touched files before commit).
- **Commit:** Folded into the main feat commit (rationale documented in the commit body).

**2. [Rule 1 — Bug] sendCh close-during-send race**

- **Found during:** `TestHub_ConcurrentSafety` failed first run under `-race` with "Write at … by goroutine N: runtime.closechan() / Previous read … by goroutine M: runtime.chansend()" — the writer (`Connection.Send` in a Broadcast goroutine) and the closer (`Connection.Close` from Unregister) raced on the chan header struct.
- **Fix:** Removed the `close(c.sendCh)` call from `Connection.Close`. The `closed` signal channel is now the sole termination signal: `Send` selects on `<-c.closed` to return false; `writePump` selects on `<-c.closed` to exit. Buffered messages already queued in sendCh are abandoned (acceptable — a closing connection can't deliver pending writes anyway).
- **Commit:** Folded into the main feat commit.
