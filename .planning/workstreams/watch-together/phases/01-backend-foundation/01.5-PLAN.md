<lifecycle_contract>
1. Client opens `GET /api/watch-together/ws?token=<jwt>&room=<roomId>` (HTTP 1.1 with Upgrade headers).
2. Handler reads `token` query param. If empty OR ValidateAccessToken fails → `httputil.Unauthorized(w)` (HTTP 401). DO NOT upgrade. (WT-NF-01)
3. Handler reads `room` query param. If empty → HTTP 400 (no upgrade). (Bad request — client error before we even open a socket.)
4. Handler calls `repo.Exists(ctx, roomID)`. If false → HTTP 404 (no upgrade). DESIGN NOTE: CONTEXT.md mentions "validates room exists in Redis; rejects with `error: {code: 'ROOM_NOT_FOUND'}` close frame if not". We diverge from CONTEXT.md here: rejecting BEFORE upgrade is cleaner and more debuggable than rejecting after. Document this in 01-05-SUMMARY.md. Frontend handles both shapes (HTTP 404 OR close-frame ROOM_NOT_FOUND) the same way; the frontend in Phase 2 will see HTTP 404 in the browser network panel.
5. Handler calls `hub.MemberCount(roomID)`. If >= cfg.MaxMembers → upgrade THEN send `error: {code: CAPACITY_FULL}` envelope + close frame.
6. Upgrade succeeds. Construct MemberMeta with username/avatar pulled from JWT claims (avatar may be empty for now), joined_at=now, last_seen_at=now. Call `repo.AddMember(ctx, roomID, userID, meta)`.
7. Call `hub.Register(roomID, userID, username, conn)` → returns *Connection.
8. Compute fresh RoomSnapshot via `roomService.Get(ctx, roomID)`. Marshal to Envelope{Type: MsgRoomSnapshot, Data: snapshot}. Call `conn.Send(payloadBytes)`. If Send returns false → log error, Unregister, RemoveMember, close conn.
9. Construct member:joined envelope `{type: "member:joined", data: {user_id, member}}`. Call `hub.Broadcast(ctx, roomID, env, userID)` to send to OTHER members (excludeUserID = self).
10. Set `conn.OnMessage = nil` for now — 01.6 will set it via a setter or via re-wrapping the registration. Leave a TODO comment: `// TODO(01.6): wire conn.OnMessage = inboundRouter.Dispatch`.
11. The hub's readPump runs in the background; when readPump terminates (peer close, ping timeout, error), hub.Unregister fires automatically. We also need a deferred cleanup: when Unregister is called, the handler-level defer calls `repo.RemoveMember(ctx, roomID, userID)` and `hub.Broadcast(ctx, roomID, MsgMemberLeft{user_id})`. NB: This cleanup must run even on abnormal close — wire it as a Connection-close-callback OR via a goroutine that watches the connection.

Practical wiring for step 11: extend `hub.Connection` with an optional `OnClose func(*Connection)` callback. Set it in the WS handler before returning. The hub's existing Unregister fires this callback right before final cleanup. (This is an interface extension — coordinate with 01.3's SUMMARY: if 01.3 didn't ship OnClose, this plan amends it.)
</lifecycle_contract>
