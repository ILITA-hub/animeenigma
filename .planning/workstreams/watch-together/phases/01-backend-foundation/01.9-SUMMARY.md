## Auto-fixed Issues (Rules 1 + 3)

### **1. [Rule 1 — Bug] libs/metrics responseWriter blocks WebSocket upgrades**

- **Found during:** Task 01.9.1 — running the smoke script for the first time. The WS connect through the gateway returned `Expected 101 status code`; gateway logs showed `status: 500`; watch-together logs showed `WARN: watch_together ws upgrade failed err: "websocket: response does not implement http.Hijacker"`.
- **Root cause:** `libs/metrics/metrics.go`'s `responseWriter` wraps the inbound `http.ResponseWriter` to capture status code + size, but DOESN'T re-expose `http.Hijacker` (or `http.Flusher`). The metrics middleware is mounted globally on **every** Go service via `libs/metrics.Collector.Middleware`, so the upstream `httputil.NewSingleHostReverseProxy` ServeHTTP path can never see Hijacker — and Hijacker is mandatory for WS upgrades (the proxy hijacks the TCP socket after the 101 handshake to ferry bytes bidirectionally).
- **Fix:** Add `Hijack() (net.Conn, *bufio.ReadWriter, error)` and `Flush()` methods that delegate to the inner writer if it supports the interface, returning `http.ErrNotSupported` otherwise.
- **Impact:** Backwards-compatible — no caller previously type-asserted `Hijack` against this writer (it never worked, so no code depended on the no-op behavior). All existing services pick up the fix transparently on next rebuild. Verified by `make redeploy-watch-together && make redeploy-gateway`, both healthy.
- **Files modified:** `libs/metrics/metrics.go`, `libs/metrics/go.sum`
- **Commit:** `132c16f`

### **2. [Rule 3 — Blocking] Gateway Dockerfile missing watch-together go.mod COPY**

- **Found during:** Task 01.9.1 — first `make redeploy-gateway` failed at `go mod download` step: `cannot load module ../watch-together listed in go.work file: open ../watch-together/go.mod: no such file or directory`.
- **Root cause:** Plan 01.1 added `./services/watch-together` to `go.work` but plan 01.7 didn't update `services/gateway/Dockerfile` to COPY the new module's `go.mod`. The builder stage needs `go.mod` from every workspace member for `go mod download` to succeed under `go.work` mode.
- **Fix:** Add `COPY services/watch-together/go.mod services/watch-together/go.sum* ./services/watch-together/` next to the existing 13 service COPY lines.
- **Impact:** Without this, the gateway can't be rebuilt — which means the WS proxy (added in 01.7) can never reach production. This is the hidden cost of cross-service dependencies in go.work; 01.7's plan reasonably assumed someone would have caught the Dockerfile gap, but 01.8 (infra) focused on docker-compose + Makefile and didn't touch any service's Dockerfile. Documented in the PHASE-SUMMARY's deviations.
- **Files modified:** `services/gateway/Dockerfile`
- **Commit:** `132c16f`

## Known limitations / future Phase work

- **Criterion 8 TTL expiry is opt-in.** The default smoke skips the 7-second TTL-wait step because it requires restarting the watch-together container with `WATCH_TOGETHER_ROOM_TTL=5s`. The TTL path is unit-tested via `miniredis.FastForward` in `internal/repo/redis_repo_test.go`; the live deep-check is a Phase 5 production-readiness gate.
- **CI integration deferred to Phase 5.** The smoke is a developer-feedback tool, not a CI gate. Phase 5 will add a CI-friendly variant (probably a Go program rather than a bash script + bun, for portability).
- **No `member:joined` test.** The smoke uses one user (`ui_audit_bot`) connecting twice (same JWT), which is multi-tab semantics — by design, the second connection does NOT trigger `member:joined`. Adding a second test user would require a second seeded account; deferred until Phase 2 (which needs a multi-user fixture anyway for `RoomSidebar` testing).
