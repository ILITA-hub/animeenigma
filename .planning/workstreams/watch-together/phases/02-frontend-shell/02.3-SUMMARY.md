# Phase 02 Plan 02.3: useWatchTogetherRoom composable Summary

Authored the single composable that owns the watch-together WebSocket lifecycle and exposes the reactive surface every Wave 3+ component (RoomSidebar, ChatPanel, ReactionPalette, ReactionBurstOverlay) consumes. Replays `room:snapshot` on every (re)connect, auto-reconnects with exponential backoff, applies an echo guard on playback/state events, and provides typed emit + subscribe methods that lock the Phase 3 player-sync integration contract.

## Final public API

```typescript
export type ConnectionStatus =
  | 'idle' | 'connecting' | 'open' | 'reconnecting' | 'closed' | 'failed'

export interface ReactionEvent {
  id: number          // monotonic counter for Vue v-for :key
  emoji: string
  user_id: string
  ts: number          // wall-clock ms (drives 5s prune)
}

export interface UseWatchTogetherRoomReturn {
  // Reactive state
  room: Ref<Room | null>
  members: Ref<Member[]>
  messages: Ref<ChatMessage[]>
  reactions: Ref<ReactionEvent[]>
  connectionStatus: Ref<ConnectionStatus>
  lastError: Ref<{ code: ErrorCode | string; message?: string; hint?: string } | null>

  // Emit (client ŌåÆ server) ŌĆö with built-in validation
  emitPlay(time: number): void
  emitPause(time: number): void
  emitSeek(time: number): void
  emitTimeTick(time: number): void
  emitChangeEpisode(episode_id: string): void
  emitChangePlayer(player: PlayerKind): void
  emitChangeTranslation(translation_id: string): void
  sendChat(body: string): void        // drops empty/whitespace/>500-char silently
  sendReaction(emoji: string): void   // drops non-whitelist silently
  heartbeat(): void

  // Subscribe (server ŌåÆ client) ŌĆö each returns its own unsubscribe closure
  onPlaybackEvent(handler: (e: PlaybackEventData) => void): () => void
  onStateChanged(handler: (e: RoomStateChangedData) => void): () => void
  onChatMessage(handler: (m: ChatMessage) => void): () => void
  onReaction(handler: (r: ChatReactionOutData) => void): () => void
  onMemberJoined(handler: (m: MemberJoinedData) => void): () => void
  onMemberLeft(handler: (m: MemberLeftData) => void): () => void
  onCorrection(handler: (c: PlaybackCorrectionData) => void): () => void
  onError(handler: (e: ErrorData) => void): () => void
  onRoomClosed(handler: (r: RoomClosedData) => void): () => void
```

## Reconnect backoff schedule

```typescript
const BACKOFF_MS = [1000, 2000, 4000, 8000, 16000, 30000] as const
// Index advances after scheduling; clamps at length-1 = 30s.
// Index resets to 0 on every successful room:snapshot receipt.
```

| Close # | Delay before reconnect |
|---------|------------------------|
| 1       | 1000 ms                |
| 2       | 2000 ms                |
| 3       | 4000 ms                |
| 4       | 8000 ms                |
| 5       | 16000 ms               |
| 6+      | 30000 ms (cap)         |

After a successful `room:snapshot` the index resets, so a stable connection followed by a single drop reconnects at 1s again (not 30s).

## Echo guard rules

The re-emission guard suppresses **local handler fire** when an inbound event's `by_user_id` matches `useAuthStore().user?.id`. Reactive state is still updated (so the room.playback_state field stays canonical for late readers).

| Channel | Echo-guarded? | Rationale |
|---|---|---|
| `playback:event` (onPlaybackEvent) | **Yes** | Prevents the "I clicked play and my own player jumped" feedback loop |
| `room:state_changed` (onStateChanged) | **Yes** | Same ŌĆö clicking the player switcher shouldn't trigger a player remount in your own view |
| `playback:correction` (onCorrection) | No (N/A) | Per-recipient message; by construction the recipient is the drift source |
| `chat:message` (onChatMessage) | No | Server is the persistence anchor; sender sees their own message via the echo broadcast (Phase 1 chat-echo rule) |
| `chat:reaction` (onReaction) | No | Same as chat ŌĆö burst overlay must fire for the sender too so they see their own emoji ascend |
| `member:joined` / `member:left` | No (N/A) | These ARE inherently about other users (you don't join your own room twice) |
| `error` (onError) | No | Errors are sender-only to begin with |
| `room:closed` (onRoomClosed) | No | Terminal broadcast |

## Deviations from CONTEXT.md

**None.** The composable's public surface, reconnect schedule, echo-guard rules, and WS URL construction all match the locked decisions in `02-CONTEXT.md ┬¦decisions`. Six minor scope clarifications surfaced and were resolved within the plan's existing latitude:

1. The `connect()` function now mirrors the REST snapshot into the refs BEFORE the WS opens. This is purely additive (the WS-delivered snapshot still overwrites it). Rationale: gives `WatchTogetherView` something to render during the 100-500ms connecting window.
2. The `connect()` REST failure path emits a synthetic `ErrorData{code:'REST_FAILED'}` through `onError` subscribers. `ErrorData.code` is typed `ErrorCode | string` in `@/types/watch-together` for exactly this kind of synthetic emission.
3. `member:joined` dispatch is idempotent ŌĆö the same user_id appearing twice (backend race on reconnect) won't duplicate the roster entry.
4. `room:closed` and `error` (with terminal codes) both set `stopped=true` to permanently disable reconnection, matching Phase 1 close-frame semantics.
5. Subscribe factory pattern: a single `subscribe<T>(set)` higher-order function generates all 9 `on*()` methods, ensuring identical unsubscribe semantics across channels.
6. The `onerror` WS handler is intentionally a no-op ŌĆö `onclose` is the unified reconnect trigger; recording in both would double-schedule.
