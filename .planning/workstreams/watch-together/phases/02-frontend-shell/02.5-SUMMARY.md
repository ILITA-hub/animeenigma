# Phase 02 Plan 02.5: ReactionPalette.vue + ReactionBurstOverlay.vue Summary

24-emoji clickable palette plus pointer-events-none CSS-keyframes burst overlay; consumes `useWatchTogetherRoom().sendReaction` + `.reactions` from 02.3 with no new dependencies.

## What Was Built

Two co-located Vue SFCs and their Vitest specs:

1. **`ReactionPalette.vue`** — `<section>` with `font-semibold` heading + 6-col (mobile) / 8-col (md+) grid of 24 emoji `<button>`s. Each button is `aria-label`-tagged with its own emoji so screen readers speak the symbol rather than the raw codepoint name. Click → 200ms-throttled `props.sendReaction(emoji)`.

2. **`ReactionBurstOverlay.vue`** — `<div class="pointer-events-none absolute inset-0 overflow-hidden">` containing one `<span>` per `ReactionEvent`. Each span gets an inline `left:NN%` coordinate (memoized per `reaction.id` in a module-scoped Map) and a `.burst` class wired to a `@keyframes burst-rise` block in a `<style scoped>` section. Animation: 3s ease-out, opacity 1 → 0, translateY(0 → -200px).

## Key Decisions

### 1. Palette order = REACTION_WHITELIST order (no override)

The palette renders `v-for="emoji in REACTION_WHITELIST"`. Hardcoding a separate array would create a synchronization point that nobody updates when emojis get added in Phase 5+. Consuming the const directly means: edit `@/types/watch-together`, palette + server filter stay in lockstep.

### 2. Throttle ms = 200, anchored on `Date.now()`

Why 200ms specifically: empirically, the fastest double-tap on a touch device is ~50ms; legit "I really want a second reaction right now" cadence is ~400ms+. 200ms sits comfortably in the gap. It's strictly a UX-feedback throttle — the server already rate-limits at 5/sec, but a silent server drop on the 2nd click within ~50ms looks like the button broke, so we surface the cooldown locally.

`Date.now()` over `performance.now()` because `vi.useFakeTimers()` fake-times `Date` by default but not `performance.now` — using `Date` lets the throttle test stay deterministic without extra timer setup.

### 3. Burst keyframe shape: 3s, rise 200px, fade to 0

The composable prunes reactions from `reactions.value` at the 5s mark. The 3s animation finishes (opacity 0) well before pruning, so there's never a visual pop when the array entry actually disappears. Shape:

```css
.burst {
  bottom: 10%;
  animation: burst-rise 3s ease-out forwards;
}
@keyframes burst-rise {
  from { opacity: 1; transform: translateY(0); }
  to   { opacity: 0; transform: translateY(-200px); }
}
```

`forwards` keeps the end-state (opacity 0) applied after the animation completes — without it the emoji would snap back to opacity 1 before being pruned, causing a flicker.

### 4. Per-id `leftFor()` memoization, module-scoped Map

Naïve approach: re-roll `Math.random()` inside the template on every render. Result: every parent re-render shuffles all the floating emoji horizontally, which looks like a teleport bug. Fix: a `Map<number, number>` keyed by `reaction.id` caches the first roll; subsequent calls return the cached value.

The Map lives at module scope (not inside `setup()`) on purpose: if the overlay is unmounted and remounted (e.g. user toggles fullscreen), reactions that haven't aged out yet still get their original horizontal slot rather than re-shuffling.

### 5. `pointer-events: none` is non-negotiable

The overlay is a child of the player wrapper. If it caught pointer events, every click in the player area (play/pause, scrubber drag, fullscreen button) would hit the invisible overlay first. Spec test 5 locks this — any future refactor that drops the utility will fail CI.

## Wire-Up Path (for Phase 2 downstream plans)

The two components are pure renderers; the consumer (RoomSidebar in 02.6, OurEnglishPlayer in 02.7) wires:

```vue
<template>
  <ReactionPalette :send-reaction="sendReaction" />

  <div class="relative">
    <OurEnglishPlayer ... />
    <ReactionBurstOverlay :reactions="reactions" />
  </div>
</template>

<script setup>
const { sendReaction, reactions } = useWatchTogetherRoom(roomId)
</script>
```

The composable already owns the 5s prune and reconnect-replay semantics — these components are purely visual.
