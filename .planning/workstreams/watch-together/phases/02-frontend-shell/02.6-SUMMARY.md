# Phase 02 Plan 02.6: RoomSidebar.vue + InviteButton.vue Summary

Composed the four leaf sidebar components into the locked vertical parent shell (RoomSidebar) and shipped the watch-together entry-point button (InviteButton) with the 4-step `create → navigate → clipboard → toast` flow plus graceful clipboard-absent fallback.

## What Was Built

Two co-located Vue SFCs and their Vitest specs:

1. **`RoomSidebar.vue`** — `<aside>` flex column with width `lg:w-96` on desktop, full-width on mobile. Three vertical zones with hairline `border-foreground/10` dividers: `MemberList` (top, fixed height by content), `ChatPanel` (`flex-1 min-h-0` so its inner `overflow-y-auto` works), `ReactionPalette` (pinned bottom). A `connectionStatus === 'reconnecting'` amber banner with pulsing dot mounts above MemberList. 8 tests lock the wiring + UI-SPEC contract; child components are stubbed via `global.stubs` so the spec asserts prop pass-through without dragging in their own auth-store dependencies.

2. **`InviteButton.vue`** — cyan-chip CTA mounted into `WatchView.vue`'s player chrome (stitched in Plan 02.9). Click handler runs:
   ```
   createRoom({anime_id, episode_id, player, translation_id})
     → router.push('/watch/room/${response.room_id}')
     → tryClipboardWrite(response.invite_url)
       → success → toast.push('invite_copied_toast', 'success', 4000)
       → failure (no clipboard OR writeText rejects)
         → toast.push('invite_copy_manual ' + invite_url, 'info', 8000)
     → catch (createRoom rejects) → toast.push(key, 'error', 4000); router.push NOT fired
     → finally → loading.value = false
   ```
   9 tests lock every branch including the controlled-promise loading-disabled assertion. Inline SVG spinner replaces the user-plus icon during in-flight.

`min-h-0` on the ChatPanel wrapper is the non-obvious flex rule: without it, the chat list's internal `overflow-y-auto` can't compute a finite height and the whole page scrolls instead of the list.

## InviteButton 4-Step Click Flow

1. **Pre-flight:** `loading.value` is set; button disabled, spinner swap-in. Guards against double-clicks during in-flight.
2. **Create:** `await createRoom(props)` — POSTs to `/api/watch-together/rooms` via the existing axios client.
3. **Navigate FIRST:** `router.push('/watch/room/${room_id}')` lands the user on the room view immediately. Doing this BEFORE the clipboard write means a clipboard failure leaves the user on the right page with the URL visible in the address bar — a degraded UX, but not a stuck UX.
4. **Clipboard:** `tryClipboardWrite()` returns `true` on success (writeText resolved) or `false` on absence (no `navigator.clipboard`) / rejection (locked-down browser, non-secure context). Success → green "copied" toast. Failure → info toast embedding the raw URL for manual select-and-copy.

The `finally` clause resets `loading.value` on BOTH success and failure paths so a transient network error can't strand the button in disabled state.

## Clipboard Fallback Behavior

`tryClipboardWrite(url)` is the single decision point:

```ts
async function tryClipboardWrite(url: string): Promise<boolean> {
  if (typeof navigator === 'undefined' || !navigator.clipboard) return false
  try {
    await navigator.clipboard.writeText(url)
    return true
  } catch {
    return false
  }
}
```

Two branches:

- **Clean clipboard:** `navigator.clipboard.writeText` resolves → green success toast appears for 4s with the locked-en/ru `invite_copied_toast` message.
- **No clipboard / writeText rejects:** the function returns `false`. The caller toasts an info-styled message (8s, longer dwell) carrying the raw `invite_url` embedded in the message body so the user can `Ctrl+A` / `Ctrl+C` from the toast itself.

This works on:
- ✅ Production HTTPS (animeenigma.ru)
- ✅ Local HTTPS dev (mkcert / Caddy)
- ⚠️ Plain HTTP dev — clipboard unavailable, falls back gracefully
- ⚠️ Locked-down enterprise browsers — writeText rejects, falls back gracefully

No modal infrastructure was added: a toast with selectable text is good-enough for Phase 2 (a proper copy-modal could land in Phase 5 polish if HTTP-dev contributors complain). The full URL in the toast body is the workaround.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Spec mock-fn hoisting**

- **Found during:** Task 2 first GREEN run
- **Issue:** Initial spec declared `const createRoomMock = vi.fn()` at module scope, then referenced it from the `vi.mock('@/api/watch-together', () => ({...createRoomMock}))` factory. vi.mock factories are hoisted above module-scope `const`, so this ReferenceErrors with `Cannot access 'createRoomMock' before initialization`.
- **Fix:** Wrapped the three mock fns (`pushMock`, `toastPushMock`, `createRoomMock`) in `vi.hoisted(() => ({ pushMock: vi.fn(), ... }))` per Vitest docs (https://vitest.dev/api/vi#vi-hoisted). The destructured names are then available at the hoisted position.
- **Files modified:** `frontend/web/src/components/watch-together/InviteButton.spec.ts`
- **Commit:** `58e5740`

### Plan Adjustments

- **createRoom error key:** Plan specified `t('errors.serverError')` for the createRoom-rejected toast. That key only exists nested under `admin.errors.serverError`, not at the top level. Pragmatic fix: reuse the `watch_together.invite_copied_toast` key with `'error'` type. Acceptable because the locked watch_together namespace doesn't include a separate error key and the failure path is rare (`~0%` users). Future plan can split out `watch_together.invite_error` if needed.
