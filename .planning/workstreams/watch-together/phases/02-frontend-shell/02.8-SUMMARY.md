## Route entry (literal block)

The new entry in `frontend/web/src/router/index.ts` (placed directly
after the `/game/:roomId` route for thematic grouping with other
room-style routes):

```typescript
{
  // Workstream watch-together / Phase 2 (WT-SHELL-01) — synchronized
  // watch-with-friends room. The view at /views/WatchTogetherView.vue
  // fetches the room snapshot via REST, opens a WS through the
  // useWatchTogetherRoom composable, and dispatches to one of the 5
  // existing <*Player> components based on room.player.
  //
  // requiresAuth via meta is honored by the global beforeEach guard
  // below: unauthenticated users are written into
  // sessionStorage.returnUrl and redirected to /auth, which restores
  // them to the room URL after successful login. This satisfies
  // CONTEXT.md §"Routing"'s "redirect to /login?next=…" requirement
  // using the project's existing convention (sessionStorage.returnUrl,
  // NOT a ?next= query param) — same as every other requiresAuth route.
  //
  // Lazy-loaded so the WatchTogether dependency graph (composable +
  // sidebar + 5 player components via defineAsyncComponent) is
  // chunk-isolated and only paid for on this route — WT-NF-04 budget.
  path: '/watch/room/:roomId',
  name: 'watch-together-room',
  component: () => import('@/views/WatchTogetherView.vue'),
  meta: { titleKey: 'watch_together.title', requiresAuth: true }
},
```

## 5-way player switch (v-if/v-else-if chain)

The view's template chain — strict `v-if/v-else-if` without any
non-conditional elements between branches (MEMORY rule):

```vue
<KodikPlayer
  v-if="livePlayer === 'kodik'"
  :anime-id="animeId"
  :initial-episode="initialEpisode"
  :room="roomHandle"
/>
<AnimeLibPlayer
  v-else-if="livePlayer === 'animelib'"
  :anime-id="animeId"
  :initial-episode="initialEpisode"
  :room="roomHandle"
/>
<OurEnglishPlayer
  v-else-if="livePlayer === 'ourenglish'"
  :anime-id="animeId"
  :initial-episode="initialEpisode"
  :room="roomHandle"
/>
<HanimePlayer
  v-else-if="livePlayer === 'hanime'"
  :anime-id="animeId"
  :initial-episode="initialEpisode"
  :room="roomHandle"
/>
<RawPlayer
  v-else-if="livePlayer === 'raw'"
  :anime-id="animeId"
  :room="roomHandle"
/>
<!-- Forward-compat: unknown player kind → empty state, NOT a crash. -->
<div
  v-else
  class="flex items-center justify-center w-full h-full text-foreground/60 font-medium"
>
  {{ t('watch_together.loading') }}
</div>

<ReactionBurstOverlay :reactions="liveReactions" />
```

`<ReactionBurstOverlay>` is a sibling of the chain (sits inside the
same relative-positioned player wrapper) — the v-else fallback prevents
it from being mounted into the chain's middle (which would break the
chain).

## State branches

The view has 4 mutually-exclusive top-level states (rendered in this
priority order via `v-if/v-else-if`):

| State                          | Trigger                                              | Render                                               |
| ------------------------------ | ---------------------------------------------------- | ---------------------------------------------------- |
| Loading                        | `bootstrap()` in flight                              | `watch_together.loading` text                        |
| Room ended (`errorState='gone'`) | REST 410 / RoomGoneError / composable.onRoomClosed   | room_ended_title + back-to-anime button              |
| Capacity full (`errorState='capacity'`) | composable.onError({code:'CAPACITY_FULL'})    | capacity_full_title + back-to-anime button           |
| Live room                      | (default — snapshot loaded, composable connected)    | Player chain + ReactionBurstOverlay + RoomSidebar    |

AUTH_EXPIRED is special — it doesn't render a view state; it
immediately `router.push`'s to `/auth?next=…` (and writes
sessionStorage.returnUrl as belt + suspenders).

## Deviations from CONTEXT.md

### Deviation 1: `sessionStorage.returnUrl` instead of `?next=` query param

**CONTEXT.md §"Routing"** specifies: "redirects unauthenticated to
`/login?next=/watch/room/:roomId`."

**Actual implementation:** Uses the project's existing
`sessionStorage.returnUrl` convention (router/index.ts:196) for the
pre-mount guard, AND additionally pushes `?next=…` for mid-session
AUTH_EXPIRED — belt + suspenders.

**Why:** The pre-mount auth guard (router.beforeEach) already exists
and uses `sessionStorage.setItem('returnUrl', to.fullPath)` for EVERY
requiresAuth route in the project. Introducing a `?next=` query
parameter ONLY for /watch/room would be inconsistent. The view itself
adds `?next=` on mid-session AUTH_EXPIRED so the resume target is
URL-encoded for users who bookmark or share the auth URL (e.g. invite
URL → click → "you need to log in first" → share that URL → friend
clicks → friend lands on /auth?next=/watch/room/abc).

### Deviation 2: `loading` state instead of `reconnect_failed_title`

**Plan §<behavior>** mentions a `<div v-else-if="connectionStatus ===
'failed' && errorState !== 'auth_expired'">` reconnect-failed branch
with a `<button @click="reload">` Refresh action.

**Actual implementation:** This branch was omitted from the view. The
composable's `onError` already redirects on the three terminal codes
(CAPACITY_FULL → capacity state; AUTH_EXPIRED → router.push;
ROOM_NOT_FOUND → would surface via REST 410 first); REST-level failure
during bootstrap surfaces as the "room ended" state.

**Why:** Phase 5 will polish the reconnect/capacity/auth-expired full
UX (per CONTEXT.md §deferred). For Phase 2 the simplified four-branch
view covers the two-browser smoke scenarios; a separate "Unable to
reconnect" branch with a Refresh button is a Phase 5 polish item. Plan
02.10 will document this as a follow-up if the smoke test exposes any
UX gap.
