## Visibility-gate rationale

The compound `v-if="authStore.isAuthenticated && playerActivated && anime"` enforces three preconditions:

1. **`authStore.isAuthenticated`** — non-logged-in users can't host rooms (API returns 401 anyway). Hiding the button avoids dead clicks and confusing toast errors. The Anime.vue route is publicly readable, so `authStore.isAuthenticated` IS load-bearing here — we can't rely on a route-level `requiresAuth` flag.
2. **`playerActivated`** — Anime.vue defaults to a click-to-load placeholder before the user activates the player. Until activation, `videoProvider` may still be in flux (provider switching is allowed pre-activation via the language tabs). Waiting for activation gives us a stable `videoProvider` to forward into the room.
3. **`anime`** — `useAnime()` loads async; `anime.value` is `null` until the catalog request resolves. The `anime.id` template binding would crash without this guard.

## Phase 4 deferral note

Two prop values are placeholders in Phase 2 and will be wired through in Phase 4 (State Switching):

| Prop | Phase 2 value | Phase 4 plan |
|------|---------------|--------------|
| `episode-id` | `String(resumeStartEpisode ?? 1)` — parent-owned hint, may be stale if the user navigates episodes inside the active player | Plumb the active episode via a player-emitted `currentEpisode` event surfaced as a reactive ref in Anime.vue |
| `translation-id` | `''` (empty string) | Each player component (`KodikPlayer`, `AnimeLibPlayer`, etc.) emits its currently-selected translation up to Anime.vue, which forwards into InviteButton |

The backend `CreateRoomBody` (`services/watch-together/internal/handler/rooms.go`) treats both fields as opaque strings with no validation beyond non-emptiness for `episode_id`, so the empty `translation_id` is a clean no-op at the API boundary. The room snapshot will simply carry `translation_id: ""` and Phase 4's switching logic will overwrite it on first user action.
