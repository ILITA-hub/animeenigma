# Phase 3 Plan 03.1: usePlayerSyncBridge composable Summary

The watch-together HTML5-video bridge: a single composable that wires
bidirectional playback sync between any `<video>` template ref and a
`WatchTogetherRoomHandle`. Locks the contract Waves 2-4 build on; zero
player files were touched in this plan.

## What shipped

`frontend/web/src/composables/usePlayerSyncBridge.ts` ŌĆö 356-line named-export
composable. Consumer wires sync with one line in setup:

```ts
if (props.room) usePlayerSyncBridge(videoRef, props.room)
```

After this call the player gains:

- **Outbound sync**: native `@play` / `@pause` / `@seeked` events on the video
  element call `room.emitPlay(t)` / `emitPause(t)` / `emitSeek(t)` respectively.
- **Inbound sync**: `room.onPlaybackEvent` handlers apply the remote frame to
  the video. `play` seeks if drift > 1s then calls `video.play()`; `pause`
  pauses + seeks; `seek` seeks. All three are wrapped in `withApplyGuard()`
  so the resulting native echo events do NOT re-emit.
- **Heartbeat**: a `requestAnimationFrame` loop, gated by a 1000 ms `Date.now()`
  delta, emits `room.emitTimeTick(currentTime)` exactly once per second while
  the video is `!paused && !ended && !seeking`. Stops on pause/ended/unmount.
- **Drift correction**: `room.onCorrection` handlers compute a network-latency-
  compensated target time. Drift < 1s ŌåÆ `playbackRate` nudges to 0.97 (ahead)
  or 1.03 (behind) for 5s, then restores. Drift Ōēź 1s ŌåÆ direct `currentTime`
  jump. No toasts, no UI ŌĆö corrections are invisible per WT-SYNC-06.
- **Cleanup**: `onBeforeUnmount` cancels the RAF loop, clears the apply-guard
  watchdog + playbackRate-restore timers, restores `playbackRate = 1.0`,
  detaches all four native listeners, and drains the two room
  unsubscribers.
