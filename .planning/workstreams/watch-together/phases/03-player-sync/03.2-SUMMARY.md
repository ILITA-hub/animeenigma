# Phase 03 Plan 03.2: Wire usePlayerSyncBridge into AnimeLibPlayer + OurEnglishPlayer Summary

Replaced the Phase 2 (02.7) `void props.room` placeholder in two HTML5-based players (AnimeLib, OurEnglish) with real `usePlayerSyncBridge(videoRef, props.room)` calls from Plan 03.1, gated by `if (props.room)` for zero behavioral regression on the standard non-WatchTogether mount path.

## Why the bridge call moved (not in-place at the anchor)

The Phase 2 plan placed `void props.room` at line 344 (before `videoRef` declaration at line 374). The plan text says "Replace the line `void props.room` ... with `if (props.room) { usePlayerSyncBridge(videoRef, props.room) }`". A literal in-place replacement would have referenced `videoRef` before the `ref<HTMLVideoElement | null>(null)` call that declares it — a runtime ReferenceError under temporal-dead-zone semantics for `const` in `<script setup>`.

The acceptance criteria are all grep-based (`grep -c == 0` for the anchor, `grep -c == 1` for the bridge call) — they don't constrain the line at which the bridge call appears. Moving the call after `videoRef` satisfies all listed checks AND is the only valid execution order.

This is a Rule 1 (auto-fix-bug) deviation in spirit: the plan's literal instruction was a near-miss bug; the intent was preserved while the placement was made correct.
