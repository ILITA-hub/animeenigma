## Architecture Notes

### Adapter shape (KodikPlayer.vue)

```
                  ┌────────────────────────────────────────────────────┐
                  │  Parent window (KodikPlayer.vue setup)             │
                  │                                                    │
   props.room    ─┤  onMounted(): kick off probe + subscribe room      │
                  │     │                                              │
                  │     ├──> postCommand('get_time')                  ─┼──> iframe
                  │     │                                              │
                  │     │   2s timeout: kodikSyncAvailable=false       │
                  │     │   → fallback banner renders                  │
                  │     │                                              │
                  │     ├──> room.onPlaybackEvent → postCommand(play/  │
                  │     │    pause/seek) gated on kodikSyncAvailable   │
                  │     │                                              │
                  │     └──> room.onCorrection → speed nudge or seek   │
                  │                                                    │
                  │  handleKodikMessage(event):                        │
                  │     ├── kodik_player_time → resolve probe          │
   room.emit*  ◄──┤     ├── kodik_player_play → room.emitPlay         │
                  │     ├── kodik_player_seek → room.emitSeek          │
                  │     ├── kodik_player_pause → room.emitPause +     │
                  │     │                        existing progress save│
                  │     └── kodik_player_time_update → room.emitTimeTick + existing progress save │
                  └────────────────────────────────────────────────────┘
```

### Re-entry guard (`applyingRemote`)

Set true before `postCommand`, released 300ms after via `setTimeout`. This window covers Kodik's echo event (the iframe fires `kodik_player_play` etc. after the RPC dispatcher applies the command), preventing the local handler from re-emitting back to the room and looping.

### Boot probe lifecycle

1. `onMounted` (room mode only): schedule `probeKickoffTimer = setTimeout(tryProbe, 500)` — gives iframe ~500ms to register `window.player.api`.
2. `tryProbe`: `postCommand('get_time')` + `bootProbeTimer = setTimeout(...timeout..., 2000)`.
3. Happy path: iframe replies `kodik_player_time` → `handleKodikMessage` clears `bootProbeTimer` and sets `kodikSyncAvailable=true`.
4. Sad path: 2s elapse without reply → timeout callback sets `kodikSyncAvailable=false`.
5. `onUnmounted`: clear both timers + unsubscribe playback/correction handlers.

### Pre-Phase-3 behavior preservation

Outside a room (`props.room === null` or `undefined`):
- `handleKodikMessage` still receives `kodik_player_time_update` and `kodik_player_pause` and routes them to the existing progress-save path unchanged.
- No probe runs; no banner renders (banner v-if requires `props.room && kodikSyncAvailable === false`).
- No subscriptions are attached; no extra event listeners or timers leak.

This is verified by inspection of the conditional branches at lines 381, 401, and the template v-if at line 38.

## Daily CI Hook

The canary spec carries the literal `[canary]` substring in its test name. CI can filter it via:

```bash
cd frontend/web && bunx playwright test --grep canary --reporter=list
```

This will pick up `e2e/kodik-rpc-probe.spec.ts` (the only `[canary]`-tagged spec at the time of this plan's completion). Phase 5 of the watch-together workstream is responsible for wiring the actual nightly cron + alerting (most likely a GitHub Actions schedule trigger + Telegram notification using the same `TELEGRAM_ADMIN_CHAT_ID` env var the player ReportButton already uses).

The canary's skip behavior is deliberate: stack-down, catalog-drift, and iframe-not-found return `test.skip(...)` with descriptive messages. Hard failure is reserved for `get_time` → no `kodik_player_time` reply within 5s, which is the one signal indicating Kodik shipped a bundle update that removed or renamed the undocumented dispatcher. This keeps canary noise low so CI alerting stays meaningful.

### Plan-shape adjustments documented for future readers

- The plan's draft `handleKodikMessage` extension placed the `kodik_player_time` reply handler AFTER the early-out guard. I moved it BEFORE the guard because the boot probe needs to resolve even when `kodikSyncAvailable.value === null` (the very purpose of the probe). The plan's wording was ambiguous — chose the implementation that actually works.
- The plan's draft used a `for...of` over `browseBody.items` from `/api/anime/browse?status=ongoing&limit=10`. That endpoint doesn't exist on AnimeEnigma — the canonical browse path is `/api/anime/ongoing?page_size=N`. Adjusted the canary spec's anime-discovery logic to use:
  1. Bot's seeded `/api/users/me/anime-list?status=watching` (primary, stable in CI), then
  2. `/api/anime/ongoing?page_size=10` (fallback for empty seed states), and
  3. Per-candidate probe of `/api/anime/{id}/kodik/translations` to confirm Kodik availability before navigating.
  This keeps the canary robust to seed-data churn.
- The plan suggested `iframe[src*="youtube"]` exclusion in the iframe selector. Replaced with the positive `iframe[src*="kodik"]` selector because (a) it's the right semantic anchor and (b) it survives AnimeEnigma's multi-player UI where the page may have other iframes from other players.

### Skip-path additions (canary spec)

The plan listed "graceful skip when stack is down" but did not enumerate skip paths. Added explicit `test.skip` with diagnostic messages for:
- Stack down (gateway health check).
- Bot seed empty (no `watching` rows in ui_audit_bot's list).
- No Kodik-bearing anime in the first 12 candidates (catalog drift, NOT a Kodik regression).
- Kodik iframe never becomes visible within 15s (player activation failed — orthogonal).

These keep the canary signal high — only the get_time → no-reply path is a hard failure.
