# Phase 03 Plan 03.5: Sender-attribution toasts + connection-status overlay — Summary

Ships the UX layer over Phase 3's synced playback: when a remote member presses play/pause/seek, a fadeable toast narrates "Alice paused" / "Bob seeked to 12:34" at the bottom of the player. When the WebSocket transitions to `reconnecting` or `closed`, a non-blocking "Reconnecting…" banner overlays the player while the composable's exponential backoff (`[1s, 2s, 4s, 8s, 16s, 30s]`) does its work. Two presentational SFCs + 4 i18n keys + WatchTogetherView wiring.

## What Shipped

### Task 1 — i18n keys (commit `baaec58`)

4 new keys in `watch_together` namespace (en + ru):

```json
"sync_toast_played": "{username} played" / "{username} включил(а) воспроизведение",
"sync_toast_paused": "{username} paused" / "{username} поставил(а) на паузу",
"sync_toast_seeked": "{username} seeked to {time}" / "{username} перемотал(а) на {time}",
"connection_status_closed": "Connection lost — retrying…" / "Соединение потеряно — повтор попытки…"
```

Parity test (`watch-together-keys.spec.ts`) gained 4 entries in `expectedKeys` plus 3 new interpolation-preservation tests (one per attribution toast) — a future translator can't accidentally drop `{username}` or `{time}` placeholders without the test catching it.

### Task 2 — SyncToastStack (commit `696c1e7`)

`frontend/web/src/components/watch-together/SyncToastStack.vue` (146 lines): subscribes to `room.onPlaybackEvent` on mount, unsubscribes on unmount, appends `{id, kind, username, time, createdAt}` records to a reactive `toasts` ref. Max 3 stacked (`.slice(-MAX_STACK)`), 2000ms auto-removal via `setTimeout`, fade-out via vue `transition-group` (500ms opacity + translateY).

Username resolved via `room.members.value.find()` lookup; falls back to verbatim `"someone"` when the user isn't in the roster (the rare member-left-before-event race). Time formatted client-side as zero-padded `mm:ss`.

Spec (229 lines, 11 tests) covers: empty initial state, subscribe/unsubscribe lifecycle, label dispatch for each kind, `mm:ss` time formatting (300s → `05:00`, 754s → `12:34`), `someone` fallback, 2000ms auto-removal via `vi.useFakeTimers()`, stack cap at 3 (4th event drops oldest), `pointer-events-none` wrapper, and `font-medium`-only UI-SPEC compliance.

### Task 3 — ConnectionStatusOverlay (commit `4520189`)

`frontend/web/src/components/watch-together/ConnectionStatusOverlay.vue` (77 lines): accepts `status: ConnectionStatus` and renders a translucent banner only when status is `'reconnecting'` or `'closed'`. Container is `pointer-events-none`; inner banner is `pointer-events-auto` so the aria-live announcement reaches assistive tech. CSS-only spinner via Tailwind `animate-spin`.

Explicitly silent for `'idle'`, `'connecting'`, `'open'`, and `'failed'` — the loading branch + happy path + terminal-error branches all own those UX flows in WatchTogetherView. Documented in component header.

Spec (101 lines, 9 tests) drives all 6 `ConnectionStatus` values, asserts the spinner class, the `pointer-events-none` ↔ `pointer-events-auto` composition, and font-weight compliance.

### Task 4 — Mount in WatchTogetherView (commit `7a0fbe1`)

Added 2 imports + 2 mount points inside the v-else live-room branch (lines 261–303). Source order: `ConnectionStatusOverlay` → 5 conditional player branches → `SyncToastStack` → `ReactionBurstOverlay`. Mounts inside the `<div class="relative flex-1 min-w-0 bg-black">` wrapper so absolute positioning anchors to the player.

WatchTogetherView.spec.ts FakeHandle gained an `onPlaybackEvent: vi.fn(() => () => {})` no-op stub. Documented as Plan 03.5 Task 4 requirement in the type interface — without it, every test that reaches the live-room render path crashed with `TypeError: props.room.onPlaybackEvent is not a function` because `SyncToastStack` subscribes on mount.

## Deviations from Plan

**1. [Rule 3 — Blocking] WatchTogetherView.spec.ts FakeHandle missed `onPlaybackEvent`**

- **Found during:** Task 4 verification (running `bunx vitest run src/views/WatchTogetherView.spec.ts` after wiring the overlays).
- **Issue:** The existing FakeHandle interface only carried the Phase-2 subscriber surface (`onError` + `onRoomClosed`). The new `SyncToastStack` subscribes to `room.onPlaybackEvent` on mount, which crashed every live-room test path with `TypeError: props.room.onPlaybackEvent is not a function`.
- **Fix:** Added `onPlaybackEvent: ReturnType<typeof vi.fn>` to the `FakeHandle` interface and a `vi.fn(() => () => {})` no-op factory entry. Comment documents the Plan 03.5 Task 4 dependency so future readers see the why.
- **Files modified:** `frontend/web/src/views/WatchTogetherView.spec.ts`
- **Commit:** `7a0fbe1` (folded into Task 4 commit)

**2. [Sweep — Concurrent agent] `frontend/web/e2e/watch-together-sync.spec.ts` arrived in my SyncToastStack commit**

- **Found during:** After Task 2 commit (concurrent agent 03.6's two-browser e2e spec was also being committed in parallel).
- **Issue:** My `git add` invocation specified only the two SyncToastStack files, but the commit also picked up `frontend/web/e2e/watch-together-sync.spec.ts`. This file belongs to sibling plan 03.6 (two-browser sync e2e), not 03.5. Most likely the file was already staged at agent-start (the orchestrator may have pre-staged 03.6's work, or `git add` matched it via index state I didn't inspect).
- **Decision:** Left as-is. Reverting would actively destroy sibling-plan 03.6's parallel work, and the file is in the correct repo location with the correct contents (it's 493 lines of legitimate test code). Sibling agent 03.6 published their plan completion (`docs(03-player-sync/03.6) complete two-browser sync e2e spec plan` at `9b8c49d`) immediately after, confirming the file is theirs.
- **Mitigation for future:** When parallel agents share a working tree (no worktree isolation), `git status --short` + `git add` of specific paths is still safe for THEIR files, but pre-staged work by the orchestrator can ride along. Future enhancement: agents should `git diff --cached --stat` before `git commit` to confirm only their own paths are in the index.

**3. [Discretion — Plan didn't specify z-index]**

- **Decision:** Added `z-10` to SyncToastStack's outer container to ensure the toast layer reliably sits above the player but below `ReactionBurstOverlay` (which has its own absolute positioning + no explicit z-index). Plan was silent on z-index; I chose `z-10` (matching ConnectionStatusOverlay's explicit `z-10`) for consistency.
- **Files affected:** `frontend/web/src/components/watch-together/SyncToastStack.vue`
