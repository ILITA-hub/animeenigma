# Plan 04.1: Catalog `/internal/anime/{id}/episodes/validate` Endpoint Summary

One-liner: catalog-side validation endpoint (200 with `{valid, reason}`) that the watch-together service will call before broadcasting `room:state_changed`, with strict validation for kodik/animelib (delegating to the notifications detector's lookup service) and permissive v1.0 validation for ourenglish/hanime/raw.

## Objective Recap

Phase 4 of the watch-together workstream needs a catalog-side back-channel so `services/watch-together/`'s `state:change_{episode,player,translation}` inbound handlers can validate a `(player, episode_id, translation_id)` tuple before broadcasting `room:state_changed`. The existing `/internal/anime/{id}/episodes` endpoint (notifications detector — NOTIF-DET-01) is too narrow: it only supports kodik|animelib and only returns a "latest episode" int. Plan 04.1 ships a sibling endpoint with episode-exists semantics across all 5 players.

```go
    ctx context.Context,

```

**Branching logic:**

| Player | Mode | Behavior |
|--------|------|----------|
| `kodik` / `animelib` | full (translation_id set) | Call `EpisodesLookupService.LatestAvailable`; parse `episode_id` as int; `EPISODE_UNAVAILABLE` if non-numeric/≤0/>latest |
| `kodik` / `animelib` | player-change (translation_id empty) | Call `animeRepo.GetByShikimoriID`; `PLAYER_UNAVAILABLE` if missing |
| `ourenglish` / `hanime` / `raw` | any | Permissive v1.0 — `EPISODE_UNAVAILABLE` only if `episode_id` is empty; `// TODO(v1.1): tighten validation` anchored at two call sites |

**Error contract:**
- `apperrors.InvalidInput` — caller bug (missing/unknown input); HTTP 400.
- `apperrors.Internal` — infrastructure (DB/parser/cache down); HTTP 500.
- `ValidateResult{Valid: false, Reason: ...}, nil` — soft negative; HTTP 200. The WT service turns this into a sender-only error frame.

**Test coverage (16 cases, all PASS under `-race`):**
- Happy: kodik, animelib, ourenglish, hanime, raw
- Episode > latest (kodik); non-numeric episode (kodik); zero/negative episode (kodik); empty episode in full mode
- Empty episode in permissive mode (raw)
- Translation NotFound mapped to `TRANSLATION_UNAVAILABLE` (animelib)
- Player-change: anime exists → Valid=true (kodik); anime missing → `PLAYER_UNAVAILABLE` (animelib); repo error → 500
- Unknown player → 400; empty shikimori_id → 400
- `IsValidPlayer` helper (positive + negative cases)

## Key Decisions

1. **D-04 "smallest change" honored** — kodik/animelib delegate to `EpisodesLookupService` instead of re-implementing parser dispatch. The 5-min Redis cache that NOTIF-DET-01 already pays for absorbs rapid-fire validations from `state:change_*` calls.

2. **Permissive ourenglish/hanime/raw with explicit TODO** — Synchronous round-trips to the scraper microservice + AllAnime + Hanime parsers on every state-change event would be too expensive for the WT use case (rooms can have N members hammering switchers). v1.0 trusts the user's selection; tightening lands in v1.1 once we have real-world signal on which combos fail and how often.

3. **Translation-omitted mode (translation_id == "")** — The original spec implied translation_id was always set, but `state:change_player` does NOT have a current translation (it's being reset). Added explicit mode so a single endpoint handles all three `state:change_*` calls instead of forcing the WT service to special-case its catalog calls.

4. **Sibling endpoint, not an extended `/episodes`** — Keeps the notifications detector's NOTIF-DET-01 contract frozen (it only supports kodik|animelib and only returns a latest-episode int). Cleaner separation, no risk of accidentally breaking the production detector.

5. **Soft-negative 200 contract** — `ValidateResult{Valid:false, Reason:...}` returns HTTP 200, NOT 4xx. The watch-together service will translate `Reason` into a sender-only error frame; bubbling up a 4xx would conflate "WT service made a bad call" (5xx-class infra problem) with "user picked an invalid episode" (UX-class soft fail).

## Known Stubs

None. The permissive validation for ourenglish/hanime/raw is documented as a deliberate v1.0 decision (not a stub) with explicit `TODO(v1.1)` markers anchored at the call sites. The plan's `must_haves` lock this behavior into the acceptance contract.

## Threat Flags

None. The endpoint is mounted on the same root-router/no-middleware/docker-network-only path as two existing internal endpoints (`/internal/cache/invalidate/raw/*` and `/internal/anime/{id}/episodes`). It does not introduce a new trust boundary — the upstream gateway already blocks `/internal/*` traffic from public clients.
