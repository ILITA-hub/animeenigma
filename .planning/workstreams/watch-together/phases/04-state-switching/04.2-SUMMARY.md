# Phase 4 Plan 2: CatalogClient + New Error Codes + CATALOG_URL Config Summary

WT-STATE-02 plumbing for watch-together's validated state-change handlers — typed catalog HTTP client with 3s timeout and 5s positive-result cache, plus 2 new error code constants (`PLAYER_UNAVAILABLE`, `TRANSLATION_UNAVAILABLE`) and a `CATALOG_URL` config knob.

## What shipped

### `services/watch-together/internal/service/catalog_client.go` (new, 252 lines)

The HTTP back-channel watch-together uses to ask the catalog "is this (anime, player, episode, translation, watch_type) combination playable?". Calls Plan 04.1's new endpoint `GET /internal/anime/{shikimoriID}/episodes/validate?player=…&episode_id=…&translation_id=…&watch_type=…`.

Contract:

```go
res, err := c.ValidateEpisode(ctx, shikimoriID, player, episodeID, translationID, watchType)

// (ValidateResult{Valid:true},               nil)  — combo resolves cleanly → broadcast state_changed
// (ValidateResult{Valid:false, Reason:"…"}, nil)  — catalog says no → send sender-only error
// (ValidateResult{},                         err)  — transport / protocol failure → log + sender-only error
```

The error channel is reserved for transport failures; `Valid=false` is a domain answer carried inside the result. This mirrors `services/notifications`' `EpisodeChecker` and means the router in 04.3 can branch on `Valid` without inspecting an error chain.

**Cache strategy:**
- Positive results (`Valid=true`) cached 5s, keyed on the full `(shikimoriID|player|episode_id|translation_id|watch_type)` tuple.
- Negative results NEVER cached — if the catalog state changes between switcher clicks, the next attempt sees it immediately.
- Mutex protects the cache map; HTTP call happens **outside** the lock so concurrent unrelated lookups aren't serialized by a slow catalog.

**Test surface (`catalog_client_test.go`, 8 cases under `-race`):**
1. Happy path `Valid=true` → no error, single HTTP hit.
2. Happy path `Valid=false + Reason="EPISODE_UNAVAILABLE"` → no error.
3. Catalog 500 → returns error.
4. Catalog 400 → error contains `"catalog rejected"`.
5. Cache hit → second call within TTL serves from cache (HTTP hit count stays at 1).
6. Cache TTL expiry → clock-fast-forward 6s → second call hits HTTP again (count == 2).
7. Negative result not cached → second call still hits HTTP (count == 2).
8. URL construction → asserts path = `/internal/anime/12345/episodes/validate` and all 4 query params present.
9. Context cancellation aborts well under the 3s timeout (proves `http.NewRequestWithContext` wires ctx properly).
