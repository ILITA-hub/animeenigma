# Phase 4 Plan 04.3: Validated state change handlers Summary

WT-STATE-02 lockdown — every state:change_* WS inbound now validates against the catalog (via Plan 04.2's CatalogClient) before mutating Redis or broadcasting room:state_changed. Invalid switches and transient catalog failures surface as sender-only error envelopes so other room members never observe an unvalidated state change.

## What shipped

### Task 1 — Validated handlers + interface + tests
**Commit:** `84db737` `feat(04-state-switching/04.3): validated state-change handlers (WT-STATE-02)`

- **`internal/service/inbound.go`** — Replaced the single pass-through `handleStateChange(ctx, conn, data, field)` with three typed handlers:
  - `handleChangeEpisode` — decodes `StateChangeEpisodeData`, reads current Room, calls `catalog.ValidateEpisode(animeID, room.Player, episodeID, room.TranslationID, "")`. On `Valid=true`: HSET `episode_id` + reset playback. On `Valid=false` or transport error: sender-only error envelope. On `repo.ErrNotFound` (room TTL'd out): silent drop.
  - `handleChangePlayer` — decodes `StateChangePlayerData`. Bogus player values (outside the 5-member set `kodik/animelib/ourenglish/hanime/raw`) short-circuit as BAD_PAYLOAD without round-tripping to catalog. Calls catalog in **player-change mode** (empty episode/translation). On `Valid=true`: HSET `player` + reset `translation_id=""` + `episode_id="1"` + playback. On rejection: sender-only PLAYER_UNAVAILABLE.
  - `handleChangeTranslation` — decodes `StateChangeTranslationData`, calls `catalog.ValidateEpisode(animeID, room.Player, room.EpisodeID, payload.TranslationID, "")`. On `Valid=true`: HSET `translation_id` + reset playback. On rejection: sender-only TRANSLATION_UNAVAILABLE.

## Decisions Made

### `episode_id` reset to literal `"1"` on player change
The plan's CONTEXT §Claude's Discretion explicitly authorized this. Player-specific episode IDs vary by parser (Kodik uses series_id/episode_id pairs; AnimeLib uses ALCID numbers; OurEnglish uses provider slugs). The frontend player's source-loading logic already resolves "what's the first episode for this anime on this player" on mount, so a sentinel "1" is good enough for the handoff. Documented inline at the handler call site referencing CONTEXT.md.

### Transport-error policy: NEVER mutate Redis
A 5xx or DNS failure from catalog means we cannot prove the new state is reachable. Refusing the change (with a sender-only error) is strictly better than silently broadcasting a state the catalog hasn't blessed — the user retries explicitly, the room stays self-consistent for the other members.

### Bogus player short-circuits before catalog
`payload.Player == "youtube"` and similar obviously-invalid values reject as BAD_PAYLOAD without hitting catalog. Saves a round-trip on inputs that can never be valid; the 5-element `validPlayers` set is the canonical truth (no risk of drift because there are exactly 5 players in the frontend code base).

### Exported `CatalogValidator` interface (vs. unexported)
Initially scoped as `catalogValidator` (lowercase) since only `inbound_test.go` needed to stub it. But `handler/websocket_test.go` in the sibling package also constructs an `InboundRouter` (for the WS upgrade tests), and a cross-package import of an unexported interface is impossible. Exported the interface; documented in the docblock that it's exported specifically to enable cross-package test stubs. No production code outside `service/` and the WS test stub touches it.

### Per-handler typed payload decode (no `map[string]interface{}`)
The Phase 1 pass-through used a flexible map-based decode so a single handler could service all 3 message types. With 3 distinct handlers now, typed decoding (`domain.StateChangeEpisodeData` etc.) is clearer, gives better error messages on malformed payloads, and matches the per-message-type design.
