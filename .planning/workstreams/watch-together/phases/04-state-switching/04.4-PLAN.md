<objective>
Wire the WatchTogetherView to drive state switching end-to-end on the frontend. The composable already (from Phase 2) updates `room.episode_id|player|translation_id` reactively on `room:state_changed` and exposes `emitChange*` methods. This plan:

1. Adds a 5-tab `PlayerTabBar` so users can request a player switch from inside the watch-together view (the Anime.vue tabs are not mounted in `/watch/room/:id`).
2. Drives a `:key`-based remount of the active `<*Player>` when `room.player` changes (Vue's reactivity alone is not enough — players cache local state in setup that wouldn't reset on prop change).
3. Wires error-code → toast UX: sender-only EPISODE_UNAVAILABLE / PLAYER_UNAVAILABLE / TRANSLATION_UNAVAILABLE errors surface as `useToast().push(message, 'error')` without changing local state.
4. Adds the 2 missing error code constants on the frontend (`ERR_PLAYER_UNAVAILABLE`, `ERR_TRANSLATION_UNAVAILABLE`) + 8 new i18n keys per locale (5 player_tab_* + 3 state_change_*_unavailable).

Output: PlayerTabBar component + tests, view wiring + tests, frontend type extensions, locale updates with parity test. Runs in parallel with 04.5 (no shared files).

Per CONTEXT §Claude's Discretion: the existing composable's auto-mutation of `room.value.player|episode_id|translation_id` makes the view's job small — it just needs to subscribe + render. Episode/translation prop changes flow through the existing `:initial-episode` binding without remount; player changes need a fresh mount.
</objective>
