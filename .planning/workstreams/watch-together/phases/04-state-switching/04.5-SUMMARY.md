# Phase 04 Plan 04.5: Per-player switcher routing through room.emitChange* Summary

One-liner: Inserted a 4-line `if (props.room) { emitChangeXxx; return }` guard at the top of `selectEpisode` (and `selectTranslation` where present) in all 5 video players, so user-driven switcher clicks flow through the Watch Together room backend instead of mutating local state directly when mounted inside a room.

## Implementation Details

### Guard placement

In each of the 5 players, the user-click selector (referenced from `@click=` in the template — confirmed via grep) received the guard as its FIRST executable line. The programmatic sibling (`_selectEpisode` / `_selectTranslation`) is intentionally NOT guarded — those are the legitimate local-state mutation point invoked from:
1. Initial auto-pick paths (on mount, via `:initial-episode` prop resolution)
2. Inbound `room:state_changed` broadcast handlers (Phase 04.4, via the existing prop-driven re-render path)

### Per-player notes

**AnimeLibPlayer.vue** — Two guards: `selectEpisode(ep)` emits `String(ep.id)`, `selectTranslation(tr)` emits `String(tr.id)` (translation ids are numeric per the parser type, so explicit String() coercion is required to satisfy the wire contract).

**OurEnglishPlayer.vue** — One guard: `selectEpisode(ep)` emits `String(ep.id)`. The server picker is `v-model="selectedServerId"` with a downstream watcher — not a click handler — so it falls outside the "user-click selector" pattern this plan targets. Server selection inside Watch Together remains a single-user concern (the inbound `room:state_changed` for translation_id would re-pick on a different axis: scraper provider, not server).

**KodikPlayer.vue** — Two guards: `selectEpisode(episode)` (numeric) and `selectTranslation(translationId)` (numeric). The Kodik iframe RPC inbound handlers (`handleKodikMessage`) are NOT touched — those are Phase 3's wired sync bridge for inbound playback events.

**HanimePlayer.vue** — One guard: `selectEpisode(ep, idx)` emits `String(ep.slug)`. `HanimeEpisode` has only `{ name, slug }` — no `id` or `number` field — so the slug is the canonical episode identifier. End-of-episode auto-advance (line ~385, `selectEpisode(episodes.value[nextIdx], nextIdx)`) flows through the same guard, which is the desired behavior: both room members advance together when one player's video ends.

**RawPlayer.vue** — One guard: `selectEpisode(ep)` emits `String(ep.id)` (RawEpisode has a stable string id per `frontend/web/src/types/raw.ts`). The Jimaku subtitle picker (`selectedSubKey`, `OtherSubsPanel`, etc.) is NOT routed through the room — subtitles are an explicit per-user UX choice (different members may prefer EN/RU/JA subs simultaneously), not shared room state. This is codified in the inline comment block + an acceptance grep that confirms `props.room.emitChangeTranslation` is absent from RawPlayer.

### Single-user mode (zero-regression guarantee)

When `props.room` is `null` or `undefined`, every selector executes its pre-Phase-4 code path byte-identically. The guard is a single conditional with an early `return` — no behavior change is possible when `props.room` is falsy.

## Deviations from Plan

None — plan executed exactly as written. Each of the 5 players received the guard at the prescribed position with the prescribed inline comment. ID coercion (`String(...)`) was applied at every emit call as specified.

The only judgment call: OurEnglish's server picker (v-model + watcher) was not converted into a guarded click handler, since the plan's `provides: "selectEpisode + (translation handler if applicable) guarded"` qualifies "translation" as "if applicable" — and the server-id axis is meaningfully different from a translation axis (it picks a scraper-provider server within the same upstream-resolved stream). The room-shared "translation" concept doesn't map cleanly onto OurEnglish's server picker. If a future plan wants to add it, it would need to convert the `<select>` to an explicit @change handler.
