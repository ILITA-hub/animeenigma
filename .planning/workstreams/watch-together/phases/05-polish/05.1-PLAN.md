<objective>
Implement a 5-minute reconnect grace period (WT-POLISH-02) so the room state survives a last-member-disconnect long enough for a returning member to rejoin with full snapshot intact. Also close the Phase 1 deviation #5 TODO — host DELETE /rooms/{id} now broadcasts room:closed before deleting.

Purpose: Without grace, a user whose WiFi blips for 20 seconds loses their entire room. With grace, transient disconnects are invisible to the social fabric of the room. WT-POLISH-02 success criterion: "Last member disconnects → room state remains queryable for 5 min; reconnecting within that window restores full state."

Output: A new `internal/service/grace.go` providing a thread-safe per-room timer registry, plus integration into the WS upgrade + close hooks. The grace timer NEVER refreshes TTL (so keys expire naturally if no one returns); it fires only to broadcast room:closed + DeleteRoom after the window elapses with no reconnect.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WS connection → Hub Register | Authenticated user (JWT validated upstream); the only attacker-controlled input is the roomID query param, already validated by the upgrade handler. |
| Grace timer goroutine → Redis | Internal, no external input. The timer fires with a roomID captured at Start time. |
| DELETE /rooms host check → hub.Broadcast | Authenticated host user; broadcast payload is empty JSON `{}`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
