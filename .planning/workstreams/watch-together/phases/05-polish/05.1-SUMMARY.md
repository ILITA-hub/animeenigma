# Phase 05-polish Plan 05.1: GraceManager Summary

JWT-aware 5-minute reconnect window for watch-together rooms — when the last
WS connection in a room drops, room state survives for cfg.GracePeriod
(default 5 min) so a transient disconnect (WiFi blip, mobile network swap,
laptop lid close) is invisible to the room's social fabric. WT-POLISH-02
satisfied; 01-04-SUMMARY.md deviation #5 (host DELETE silently deletes
without notifying members) closed in the same pass.

## What shipped

### Backend service — `services/watch-together/internal/service/grace.go` (new, 302 lines)

- **`GraceManager`** — per-room timer registry over `sync.Map`. Locking is
  intentionally per-key (no global mutex) so distinct rooms never contend.
- **`Start(roomID)`** — schedules a `time.AfterFunc(cfg.GracePeriod,
  g.fire)`. Double-Start atomically replaces the prior timer (clock resets
  on re-Start). No-op when the manager has been Closed (suppresses the
  SIGTERM-driven OnClose storm).
- **`Cancel(roomID) bool`** — `sync.Map.LoadAndDelete` is the atomic
  race-winner primitive. Returns true iff a pending timer was successfully
  stopped (the "user reconnected in time" signal). Bumps
  `wt_grace_recoveries_total` only when Stop() returns true (actual
  recovery, not a deletion race with a fired timer).
- **`Active(roomID) bool`** + **`Period() time.Duration`** — diagnostics +
  log-context getters.
- **`Close()`** — flips atomic closed flag (Start becomes a no-op), Stops
  every pending timer, clears the map. Idempotent.
- **`fire(roomID)`** — `LoadAndDelete`s its own entry (loser of the race
  with Cancel returns silently without touching Redis), builds a 5s
  background context, broadcasts `room:closed {}` via the HubFanout,
  then calls `repo.DeleteRoom`. Broadcast intentionally precedes
  DeleteRoom so any race-y in-flight `GetRoom` still finds a live room.
- **Prometheus**: `wt_grace_started_total` + `wt_grace_recoveries_total`
  registered via `sync.Once` so multiple managers in the same process
  (tests) don't panic. Consumed by Plan 05.2's Grafana panel.

### WS handler wiring — `services/watch-together/internal/handler/websocket.go`

  consumer-side interface `GraceCanceller{Cancel, Start, Period}` lives

## Deviations from Plan

### 1. [Rule 3 — blocking issue] DELETE handler authz preflight via svc.Get

**Found during:** Task 2B (rooms.go DELETE wiring).

**Issue:** Plan §B specified `roomSvc.Delete → broadcast → roomSvc.Delete`,
but svc.Delete is the only authz point (it returns ErrNotHost / ErrNotFound
internally). The plan implied broadcasting BEFORE deletion, which would
require knowing the request is authorized BEFORE we broadcast. Without a
preflight, we'd either broadcast on every DELETE attempt (including
non-host 403s — channel spam) or broadcast AFTER svc.Delete (contradicting
the plan's BEFORE intent).

**Fix:** Added an explicit preflight `svc.Get` + host check in the handler
BEFORE the broadcast. svc.Delete still re-validates host ownership (defense
in depth — the service is the single mutation surface, handler-side checks
could go stale under future refactors). Added 2 negative tests
(`TestDelete_Missing_NoBroadcast`, `TestDelete_NonHost_NoBroadcast`) to
lock in the "non-host / missing room → 0 broadcasts" contract.

**Files modified:** `services/watch-together/internal/handler/rooms.go`,
`services/watch-together/internal/handler/rooms_test.go`.

**Commit:** `ce39174 feat(05-polish/05.1): broadcast room:closed on host DELETE`.

### 2. [Rule 1 — bug] Data race in `probingHubImpl.seenLive` test field

**Found during:** Task 1 GREEN initial test run under `-race`.

**Issue:** The probing hub's `seenLive` field was written under a mutex
inside `Broadcast` but the assertion read it without locking.
`go test -race` caught it immediately.

**Fix:** Added a locked `observedLive()` getter; switched the assertion
to use it.

**Files modified:** `services/watch-together/internal/service/grace_test.go`.

**Commit:** Folded into `e68cbaf feat(05-polish/05.1): GREEN — GraceManager`.

### 3. [Plan extension] StartAfterClose test added to Task 1

**Reason:** Task 2D required a `TestGraceManager_StartAfterClose_NoOp`
test, but it tests a property OF the GraceManager itself (not the WS
wiring), so folding it into Task 1's suite kept the unit-vs-integration
split clean. Task 2's `closed` atomic was thus already tested before
Task 2 even started.

**Net effect:** Task 1 ships 10 unit tests instead of the 8 the plan
listed; Task 2 doesn't need a redundant copy.

## Threat Flags

None. The plan's `<threat_model>` covered all 4 STRIDE entries
(T-05.1-01..04) and the implementation mitigates each as specified:

- **T-05.1-01 (DoS via timer accumulation)**: per-room sync.Map, single
  entry per room (double-Start replaces), Close stops every timer.
- **T-05.1-02 (Tampering of room:closed broadcast)**: existing
  host-only authz path unchanged; preflight `svc.Get` + host check
  gates the broadcast.
- **T-05.1-03 (Information disclosure)**: grace fire deletes only the
  3 keys for the specific roomID captured at Start time; no
  enumeration risk.
- **T-05.1-04 (Race between fire and cancel)**: `sync.Map.LoadAndDelete`
  is the atomic primitive; whichever code path deletes first wins.
  Verified by `TestGraceManager_Cancel_AfterFire_ReturnsFalse` +
  `TestGraceManager_Fire_BroadcastsRoomClosedBeforeDelete`.
