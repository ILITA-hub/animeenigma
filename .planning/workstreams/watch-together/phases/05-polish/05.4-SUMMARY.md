### RoomSidebar.vue dual-mode template (Task 2)

The component now renders TWO `<aside>` siblings in its template:

```vue
<!-- Desktop right-rail (>= lg) — Phase 2 layout, gated CSS-only -->
<aside class="hidden lg:flex flex-col h-full w-full lg:w-96 …">
  …
</aside>

<!-- Mobile bottom-sheet (< lg) — Plan 05.4 WT-POLISH-03 -->
<aside
  class="lg:hidden fixed inset-x-0 bottom-0 z-30 flex flex-col …"
  :style="{ height: sheetHeight }"
  :aria-expanded="sheetExpanded"
  @touchstart.passive="onTouchStart"
  @touchmove.passive="onTouchMove"
  @touchend.passive="onTouchEnd"
>
  …
</aside>
```

Tailwind's `lg:hidden` / `hidden lg:flex` modifiers gate visibility on viewport — no JS-side matchMedia guard. Both branches render in the DOM at all times, which makes jsdom unit tests trivial: assert class presence rather than simulating viewport.

#### State machine

```typescript
type SheetTab = 'chat' | 'reactions'
const activeTab = ref<SheetTab>('chat')         // chat is active by default
const sheetExpanded = ref<boolean>(false)        // collapsed at first render
const sheetHeight = computed(() => sheetExpanded.value ? '60vh' : '80px')

function onTabClick(key: SheetTab): void {
  if (key === activeTab.value && sheetExpanded.value) {
    // Re-tap the active tab → collapse (toggle)
    sheetExpanded.value = false
    return
  }
  activeTab.value = key
  sheetExpanded.value = true                     // tap any tab → expand
}
```

#### Drag-gesture handlers

- `touchstart` captures `clientY` to a module-scoped `touchStartY`
- `touchmove` accumulates `lastDeltaY = clientY - touchStartY`
- `touchend` decides:
  - `lastDeltaY < -50` (dragged UP at least 50px) → `sheetExpanded = true`
  - `lastDeltaY > 50` (dragged DOWN at least 50px) → `sheetExpanded = false`
  - otherwise → no-op (jitter filter)

All three handlers use `.passive` so the browser keeps native scroll responsiveness. The 200ms CSS transition (`transition-[height] duration-200 ease-out`) provides a graceful fallback when touch isn't available (desktop devtools rotation, accessibility devices).

#### Member count strip

Instead of adding a third "Members" tab (which would violate the WT-POLISH-03 2-tab constraint) or a compact avatar strip (which would require adding a `variant` prop to MemberList.vue), we show a simple `{{ members.length }}/{{ MAX_MEMBERS }}` count on the right side of the tab bar. `MAX_MEMBERS = 10` is hardcoded — Phase 1's rooms service enforces this cap.

## Approach decisions

### Approach B for member display (numeric count, not avatar strip)

The plan offered two approaches for showing the member roster on mobile:
- **Approach A**: Add a `variant?: 'full' | 'compact'` prop to MemberList.vue, render a horizontal avatar row in compact mode
- **Approach B**: Skip the embedded roster and just show `{{ members.length }}/10` in the tab bar

Picked **Approach B**:
1. Satisfies WT-POLISH-03's hard 2-tab constraint without adding a third tab
2. Avoids touching MemberList.vue entirely (zero blast radius — Phase 2 tests for that component stay green by definition)
3. Member count is the high-signal information for the "is the room busy?" glance; full roster is one tap away if the user opens the Chat tab and remembers that pre-Phase-5 the roster lived above chat
4. Approach A is still available as a future polish pass if user feedback says "I miss seeing who's in the room at a glance on mobile"

### Drag-gesture testability

The 13 spec tests include drag-up + drag-down + jitter-no-op assertions using vue-test-utils' `trigger('touchstart', { touches: [...] })`. jsdom + vue-test-utils 4.x handles TouchEvent simulation cleanly via the type-cast pattern shown in the spec — no `defineExpose` test-mode escape hatch needed. The implementation captures the gesture on the `<aside>` itself (not a sub-handle element) so there's no z-index / event-bubbling fragility.
