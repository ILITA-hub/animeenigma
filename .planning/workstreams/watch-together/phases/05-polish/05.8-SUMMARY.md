# Phase 05 Plan 05.8: Daily Kodik RPC canary GitHub Actions workflow — Summary

**One-liner:** Wires the daily 7:13 UTC GitHub Actions cron that runs the Kodik
`postMessage` RPC canary spec against `https://animeenigma.ru` and Telegram-alerts
admin on failure (silent on success).

## What shipped

A single new workflow file: `.github/workflows/watch-together-kodik-canary.yml`
(99 lines). The file mirrors `player-health.yml` almost verbatim with these
adaptations:

1. **Schedule** — `cron: '13 7 * * *'` (daily 7:13 UTC). Offset from
   `player-health.yml`'s 6:07 UTC to avoid fleet contention on shared
   GitHub Actions runners. Rationale documented in the file's header
   comment so future operators don't accidentally collide the times.

2. **Browser scope** — Chromium only. The matrix `[chrome, firefox, webkit]`
   pattern from `player-health.yml` is intentionally NOT replicated: the
   canary's signal is "the undocumented `kodik_player_api` dispatcher exists
   and replies". If Kodik removes the dispatcher, it's gone in every browser
   simultaneously. Matrix-ing would triple CI minutes with no proportional
   signal gain.

3. **Test invocation** — `bunx playwright test e2e/kodik-rpc-probe.spec.ts
   --project=chromium` with `BASE_URL=https://animeenigma.ru`. The spec
   itself (Plan 03.4) carries the `[canary]` tag in its test name, but
   since the canary lives in its own dedicated file we target the file
   directly rather than via `--grep`.

4. **Artifacts** — Two upload steps, both with `if: always()` and
   `retention-days: 7`:
   - `kodik-canary-playwright-report` → `frontend/web/playwright-report/`
     (the HTML report — primary postmortem signal)
   - `kodik-canary-test-results` → `frontend/web/test-results/`
     (raw artifacts: screenshots, videos, console captures)

5. **Notify job** — Single curl POST to the Telegram bot API with a
   ready-to-act debug recipe. Uses `secrets.TELEGRAM_BOT_TOKEN` and
   `secrets.TELEGRAM_ADMIN_CHAT_ID` (same secrets `player-health.yml`
   relies on — confirmed present by reading lines 65-66 of the existing
   workflow).

6. **Manual trigger** — `workflow_dispatch:` is wired so operators can
   poke the canary on-demand (e.g. immediately after a suspected Kodik
   bundle update) without waiting for the cron.

## Secrets availability

The workflow uses the same two secrets `player-health.yml` already relies
on for daily alerting:

- `TELEGRAM_BOT_TOKEN` — bot API token
- `TELEGRAM_ADMIN_CHAT_ID` — admin's Telegram user ID

These secrets are assumed present in the GitHub repository's Actions
secrets store because the existing `player-health.yml` workflow has been
successfully alerting on the same channel since at least March 2026.
No new secret provisioning required.

`gh secret list` would confirm definitively post-merge; not invoked from
the agent shell because the canary workflow only references these
secrets at runtime — wrong-secret errors would surface as a 401 from
the Telegram API in the notify job's logs, which is exactly the
artifact-retention story this plan covers.

## Post-merge validation recipe (deferred — requires PR merge first)

After the workflow file lands on `main`:

```bash
# 1. Confirm the workflow registered
gh workflow view watch-together-kodik-canary.yml

# 2. Fire a manual run to validate end-to-end wiring
gh workflow run watch-together-kodik-canary.yml

# 3. Watch the latest run
gh run watch

# Success criteria (manual run):
#   - bun install succeeds
#   - bunx playwright install --with-deps chromium succeeds
#   - the canary executes against https://animeenigma.ru
#   - test PASSES → canary job green, notify job skipped (silent success)
#   - test FAILS  → canary job red, notify job runs, admin Telegram message lands
```

If the manual run trips on a real Kodik regression the artifact upload
provides the playwright-report HTML which surfaces the exact poll-timeout
message ("kodik_player_api get_time RPC did not reply ... within 5s").
