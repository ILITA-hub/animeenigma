       - Set up: create a room (POST /rooms), connect WS via `wscat` (or a small node helper), capture the WS handshake to confirm room:snapshot arrives.

       - Wait 3 seconds (well under the grace period — production is 5min; for the smoke, override WATCH_TOGETHER_GRACE_PERIOD=5s via env if the test container supports it, OR just verify the metric bumped without waiting for the timer fire).
       - Reconnect a new WS to the same room.
       - Verify: the WS receives a fresh room:snapshot; the room HASH still exists; `curl /metrics` shows `wt_grace_recoveries_total >= 1`.

    5. Room-closed broadcast on host DELETE (Plan 05.1 closing 01.4 TODO)
       - Create a room as host A.
       - Connect WS as host A.
       - Send DELETE /rooms/{id} as host A.
       - Verify: the WS received a `room:closed` envelope BEFORE the connection closed.
       - Verify: subsequent `GET /rooms/{id}` returns 410.

    6. Capacity gate (WT-POLISH-04 backend verification)
       - Create a room.
       - In a loop, open 10 WS connections from 10 different users (or the same user with multi-tab — pick the simpler path: same user 10 tabs since the capacity is on raw connections per Phase 1).
       - The 11th connection attempt: assert HTTP response (the upgrade attempt returns 400 with body `{"error":"CAPACITY_FULL"}` per Phase 1 spec OR a close-frame envelope — verify by reading Phase 1 SUMMARY which gates this).
       - Hint: Phase 1's smoke already covers "second WS connects and gets member:joined"; extend to the 11th-connection FAIL case.

       - This is gated by D-04-01 (catalog redeploy blocker). If `curl http://localhost:8081/internal/anime/{any}/episodes/validate?player=kodik&episode_id=1` returns 404, mark this step as SKIP with the explanatory note.     8. Cleanup (EXIT trap)
       - DELETE every room created during the smoke.
       - Disconnect any lingering WS.
