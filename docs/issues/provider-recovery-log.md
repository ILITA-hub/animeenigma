## 2026-07-16 — gogoanime

**State before:** `policy=auto, health=recovering, status=degraded` — reason: the 07-15 "4th confirm" text (health_since 2026-07-15T06:00:26Z, last_probed_at 2026-07-16T00:00:37Z, that latest tick `pass:true`). Picked it anyway — not to re-ship the same escalated fix blind a 5th time, but because the automated tick's `pass:true` (a first, after 4 straight `pass:false` confirms) was itself unexplained and worth understanding before assuming either "fixed" or "flaky."

**Root cause — reconfirmed the exact same code-level bypass as 07-11/07-14 (unchanged), but discovered WHY the probe flipped to `pass:true` and that this is a real analytics-probe / real-user divergence, not a recovery, previously undocumented:**

Walked `episodes→servers→stream` end-to-end with `prefer=gogoanime` on two anime via the real public gateway path (`/api/streaming/m/<token>/...`), confirming codecs with `ffprobe` on actual bytes, not HTTP status:

1. **"Кот и дракон" ep1 — one server (HD-1), real, unchanged.** `cdn.mewstream.buzz` → `ffprobe`: `h264 1920x1080 / aac`, genuine MPEG-TS. Consistent with all four prior findings.
2. **Frieren ep1 (the automated probe's own anchor) — now has TWO servers, not one:**
   - **HD-1** (`newplayer.php?...type=hd-1...`): resolves through the sidecar wrapper (`stealth-scraper:3000/hls?...&url=...9hjkrt.nekostream.site...`) to the **exact same byte-identical fake** as 07-06/07-09/07-11/07-14 — `ffprobe`: `codec_name=png, width=1, height=1`, **809,780 bytes**, identical size to every prior finding. 5th consecutive confirmation, byte-for-byte.
   - **HD-2** (`newplayer.php?...type=hd-2...`), never checked by any prior entry: resolves directly (no sidecar wrapper at all — `fxpy7.watching.onl/anime/.../master.m3u8`) to **genuinely real content**: `ffprobe` on the first segment confirms `h264 1920x1080`, real MPEG-TS.

**This fully explains today's `pass:true` anchor tick, and it is NOT a false positive:** the analytics probe's `HTTPResolver.Resolve` (`services/analytics/internal/probe/resolver.go:104-132`) calls `/servers` then iterates **every** returned server, resolving+validating each independently. `Engine.probeProvider` (`services/analytics/internal/probe/engine.go:163-202`) marks a ref `played` if **any** of its per-server verdicts is `ReasonPlayable` (line 193-199) — so Frieren's ref counts as played because HD-2 passed, even though HD-1 (byte-identical fake) failed. This is correct probe behavior, doing exactly what it's designed to do.

**But real users never see HD-2.** The user-facing cold path (`GetStreamWithGate`, `client.go:1086-1103`) takes a completely different route for `engine=browser` providers: `browserEnabled()` (line 934, 1093) skips the streamprobe/poison gate entirely, and `pickBrowserEmbed` (`client.go:1044-1057`) returns the **first** server whose ID matches a known wrapper-host substring — for Frieren both HD-1 and HD-2 match (`gogoanime.me.uk`), so it deterministically returns HD-1, always, on every real cold-path request. There is no fallback to HD-2 and no knowledge that HD-2 even exists as an alternative — the gate that would normally try multiple servers and cache the winner (`coldPathGated`, used by non-browser providers) is entirely bypassed. Confirmed the poison-cache mechanism that WOULD catch this (`poisonCached`/`cachePoisonVerdict`, `client.go:1209-1235`, the current byte-sniff + 24h-Redis-poison design that replaced the retired static blocklist on 07-14) is only wired into that same bypassed non-browser gated loop — never called from `streamViaBrowser` or `pickBrowserEmbed`.

**Net effect: the health dashboard is now trending toward "up" for gogoanime (accurately reflecting that a real stream exists somewhere in the server list), while real users hitting Frieren's first play via gogoanime keep getting the identical fake placeholder that's been reported 5 times.** This is the opposite-direction sibling of the 07-12/07-13 analytics bugs (which under-reported healthy providers as down) — here the probe is *correctly* thorough, but that same thoroughness makes the health signal diverge from the narrower, blind first-match the real cold path actually uses.

**Action taken:**
1. Did **not** re-attempt the previously-escalated broad fix (wiring full `streamprobe.Probe` into every browser-engine cold-path request) — still correctly out of scope for a blind during-rotation patch: it adds a synchronous second sidecar/Camoufox round-trip to the hot path for every request that need it, shared across gogoanime/animepahe/miruro/nineanime, and the latency/Camoufox-capacity tradeoff is exactly the un-made owner decision from 07-11/07-14.
2. Investigated a narrower alternative made possible by today's finding: reusing the **existing** 24h host-poison cache (`poisonCached`/`cachePoisonVerdict`) by (a) writing a poison verdict from `streamViaBrowser`'s own resolved host post-resolution — cheap, no new network call, the URL is already in hand — and (b) teaching `pickBrowserEmbed`/the browser branch of `GetStreamWithGate` to skip a server whose *last-known* resolved host is poisoned, falling through to the next matching server. This is smaller than the full-gate proposal (no new synchronous sidecar call on the common case) but still needs a new serverID→host memory (poison is currently keyed by host, not by embed/wrapper URL, since the wrapper URL alone doesn't reveal which CDN it'll resolve to until the browser actually runs it) — genuine new state and test surface, not a same-day patch. Recording the shape here so it's not re-derived from scratch.
3. Did not submit a `probe-result` override — the automated tick's `pass:true` is genuinely correct given what it tested; overriding it would be wrong in the other direction now.
4. No source changes shipped. Journal-only entry, worktree → pushed straight to `main` per the golden rule.

**Outcome:** ⚠️ Partial — same unfixed root cause as 07-11/07-14 (5th confirmation, HD-1/nekostream byte-identical), but with a materially new, previously-undocumented finding: the automated health signal and real-user experience are now diverging in a new direction for this provider (probe accurately thorough → trending "up"; real cold-path users still gated to a single deterministic, sometimes-fake, server choice). Not a functional recovery for real users.

**Next step:** Owner decision still needed (unchanged ask from 07-11/07-14: wire a playability gate into the browser cold path, or formally accept it as unguarded), but now with a cheaper third option to weigh: extend the existing poison-cache design (host-keyed, 24h, byte-sniff-derived) to also (a) get populated from the browser path's own resolved host with zero added round-trips, and (b) be consulted by `pickBrowserEmbed` via a new serverID→host memory, so a server proven fake once is skipped for the rest of its 24h window without a synchronous double-resolve on every request. This is meaningfully cheaper than full `streamprobe.Probe` gating but still real new plumbing — a plan-and-implement task, not a same-rotation patch. Also worth flagging separately: since the dashboard may soon show `gogoanime` trending `up`/`auto` on the strength of HD-2-type alternates the real path never uses, whoever reviews the health dashboard next should not read "up" as "real users are unaffected" for this specific provider until the gate gap closes.

This is a real structural gap but a **large/risky/uncertain** cross-cutting change, matching the runbook's "do NOT ship blind" guardrail — same posture as the 07-07 AA_CRYPTO_MISSING and 07-11/07-14 gogoanime entries.
2. Did **not** submit a `probe-result pass:true`. Unlike the nineanime/miruro precedents, no fix was shipped today to justify overriding the automated verdict — and the automated verdict about the *anchor specifically* is true (Frieren genuinely has no working ok.ru source right now). 4. This entry (journal-only, no source changes) written in a worktree per the golden rule, pushed straight to `main`.

**Outcome:** ⚠️ Partial — provider reconfirmed genuinely healthy for the general case (real 1.9MB MPEG-TS segment verified end-to-end through the public masked-proxy path), but still correctly `down`/`manual` on the automated anchor-gated probe, now for a precisely identified and different reason than any prior entry captured. **Next step:** Human decision needed on how to close the reroll-eligibility gap for stream-stage (not search-stage) failures, in one of two shapes: (a) extend `engine.go`'s reroll path to also fire when a resolver returns a definitively-non-infra content-unavailable signal at the stream stage — requires first exposing that distinction through the HTTP layer, since `resolver.go` currently only sees a bare non-200 status and can't tell `ErrExtractFailed` (content/parse gap, safe to reroll) apart from `ErrProviderDown` (real transport failure, must NOT reroll) — both map to a 502 today; or (b) add a per-provider anchor-override so `allanime-okru`'s slot in the shared spotlight `AnimeSet` skips titles it's known to be permanently blocked on (Frieren), rather than changing shared reroll semantics for every provider.
