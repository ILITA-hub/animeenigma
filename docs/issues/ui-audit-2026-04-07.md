##### [UA-001] Dead component: `ContinueCard.vue` is exported but never imported anywhere — Severity 1 (minor)

**View:** Home (also affects all views via tree-shaking inefficiency)
**Heuristic:** Consistency and standards / dead code
**Evidence:**
- `frontend/web/src/components/anime/index.ts` line 4: `export { default as ContinueCard } from './ContinueCard.vue'`
- `Grep "ContinueCard" frontend/web/` returns only that single export line — no imports anywhere in the codebase
- `Grep "Продолжить" frontend/web/src/views/Home.vue` returns 0 matches
- `document.body.innerText.includes('Продолжить')` returned `false` on the live Home view
- Watch history table has 3 seeded rows for `ui_audit_bot`, confirmed via DB count

**Why it matters:**
1. The user has seeded `watch_history` data but **no "Continue Watching" section appears anywhere on Home**. This is one of the most common patterns on streaming sites — its absence is a UX gap, not just dead code.
2. The component itself exists (`frontend/web/src/components/anime/ContinueCard.vue`) and is barrel-exported, suggesting an unfinished feature.
3. Tree-shaking should eliminate it from production bundles, but exporting unused symbols hides intent and confuses future maintainers.

**Citations:**
- `frontend/web/src/components/anime/index.ts — found via grep "ContinueCard"`
- `frontend/web/src/components/anime/ContinueCard.vue — found via grep "ContinueCard"` (file exists)

**Proposed fix:** Either (a) wire `ContinueCard` into `Home.vue` above the 3-column grid, gated on `authStore.isAuthenticated && watchHistory.length > 0`, OR (b) delete `ContinueCard.vue` and remove the export. Option (a) is the higher-impact fix because it adds a missing feature for logged-in users.

##### [UA-002] Auth-aware navbar uses 2-letter username initials with no fallback for short usernames — Severity 0 (cosmetic)

**View:** Home (and global — applies to Navbar.vue everywhere)
**Heuristic:** Aesthetic and minimalist design / consistency
**Evidence:**
- After login as `ui_audit_bot`, the desktop navbar shows `RU | UI` at the right edge
- `UI` appears to be the first 2 chars of the username, uppercased
- Confirmed via `document.querySelector('nav, header').innerText` returning "Anime\nEnigma\nГлавная\nКаталог\nOP/ED\nКомнаты\nRU\nUI"
- The avatar field for `ui_audit_bot` is empty string in the DB (`avatar: ''`), so the chip falls back to initials

**Why it matters:**
- A 2-letter all-caps abbreviation is correct as a fallback, but the navbar **does not show the full username on hover** or visually distinguish the account chip from a generic UI badge — at first glance, "UI" reads as a UI label (context: this is a frontend), not as an account.

**Citations:**
- `frontend/web/src/components/layout/Navbar.vue — found via grep "username"` (file pending Read for line anchor)

**Proposed fix:** Add `:title="user.username"` to the avatar chip so hover reveals the full handle. Optionally use the full username on screens >1600px wide, or add a visible chevron/dropdown indicator that this is a clickable user menu.

##### [UA-003] Home page is 2439px tall but most content lives in 600px-tall internally-scrolled columns; users may not realize there's content below — Severity 2 (major)

**View:** Home
**Heuristic:** Visibility of system status / Recognition rather than recall
**Evidence:**
- Page `scrollHeight` measured at **2439px**, viewport 787px
- Three columns with class `space-y-3 max-h-[600px] overflow-y-auto custom-scrollbar` (one per `Онгоинги`, `Топ аниме`, third pending capture)
- Column scrollHeights captured: 2216px, 1624px, 1636px — meaning each column has 3-4× its visible content scrollable internally
- The outer page has additional content below the 3-column grid (totaling 2439px) that requires scrolling the **window**, while the columns require scrolling **inside** them. This creates **dual-axis scroll confusion**.

**Why it matters:**
- Standard streaming-site pattern is to scroll vertically through stacked horizontal carousels. AnimeEnigma's pattern of "3 columns with internal vertical scroll + outer page also vertical scroll" is non-standard and creates two interaction modes for the same gesture.
- Users who hover over a column and scroll get column-internal scrolling. Users who hover between columns get window scroll. Users on touch devices may not be able to reliably scroll the window once their finger lands inside a column.
- The `custom-scrollbar` styling makes the inner scroll less visually obvious than a default browser scrollbar.

**Citations:**
- `frontend/web/src/views/Home.vue — found via grep "max-h-\[600px\] overflow-y-auto"`

**Proposed fix:** Pick one scroll model. Either (a) drop `max-h-[600px]` from the columns and let them grow naturally with the page, OR (b) show only the top N items per column with a clear "See all" CTA at the bottom (the current "Все" link in the header is too small and easy to miss). Option (a) is closer to what most streaming sites do.

##### [UA-004] Search box has no submit indicator besides a magnifier icon, and the icon button is not labeled for screen readers — Severity 1 (minor) → likely 2 (major) once axe is run

**View:** Home (and the search itself is global, also in Navbar.vue)
**Heuristic:** Help and documentation / accessibility
**Evidence:**
- `Home.vue` lines 14-21 (verified via Read): the magnifier `<button>` wraps an `<svg>` with no `aria-label`, no visible text, no tooltip
- The `<input>` uses `:placeholder="$t('search.placeholder')"` ("Поиск аниме...") with no `aria-label` either — placeholder is not a substitute for a label per WCAG 2.4.6
- Submit happens on `@keyup.enter` OR the icon button click

**Why it matters:**
- Screen reader users hit an unlabeled button (`button`) with no announcement
- Keyboard-only users who don't know "Enter to search" must Tab to the icon, which has no focus-visible label

**Citations:**
- `frontend/web/src/views/Home.vue — found via grep "search.placeholder"`

**Proposed fix:** Add `aria-label="Search"` (or `$t('search.submit')`) to the magnifier `<button>` and `aria-label="Search anime"` (or matching) to the `<input>`. Bonus: add a visible `aria-hidden="true"` label below the input for "Press Enter to search" or a kbd hint chip.

##### [UA-005] `<html lang="en">` despite Russian-locale UI — Severity 2 (major) — accessibility / SEO

**View:** Home (and global — applies to every view because it's set on `<html>`)
**Heuristic:** Match between system and the real world / accessibility / SEO
**Evidence:**
- `document.documentElement.lang` returned `"en"` on the live page
- Page text is entirely in Russian: "Главная", "Каталог", "Поиск аниме...", "Онгоинги", "Топ аниме", etc.
- Only the cosmetic `RU` switcher in the navbar reflects the user's locale; the `<html lang>` does not
- This is a stable bug — almost certainly true on every view, not just Home

**Why it matters:**
1. **Screen readers** pick the wrong pronunciation engine — Russian text read with English phonemes is unintelligible
2. **Browser translation prompts** are wrong — Chrome/Edge offer to translate to English when the user is already on the Russian site
3. **Search engines** mis-classify the page language and may penalize Russian-search rankings
4. WCAG 2.1 SC 3.1.1 (Level A) requires the language of the page to be programmatically determinable and matching the content

**Citations:**
- `frontend/web/index.html — found via grep "lang="` (root html template — verify the static value)
- `frontend/web/src/i18n/* — found via grep "locale.set\|setLocale\|html.lang"` (find where lang switching SHOULD update `document.documentElement.lang`)

**Proposed fix:** In the i18n initialization or locale-change watcher, sync `document.documentElement.lang = newLocale` (e.g. `'ru'` for Russian, `'en'` for English, `'ja'` for Japanese). This is a one-line fix once you find the right hook in `i18n.ts` or wherever locale changes are handled.

## Findings by severity (Addendum)

### Catastrophic

##### [UA-024] `/themes` crashes on every load — Severity 3 (catastrophic) — data handling

**View:** `/themes` (desktop + mobile; bug is in JS not CSS)
**Heuristic:** #9 Help users diagnose/recover, #5 Error prevention
**Evidence:**
- Navigating to `https://animeenigma.ru/themes` as `ui_audit_bot` renders the App.vue error fallback:
  ```
  Something went wrong
  Cannot read properties of null (reading 'id')
  [Reload]
  ```
- Console error: `[App Error] TypeError: Cannot read properties of null (reading 'id') at Themes-Dmdq1Rt4.js:1:11974`
- API probe confirms the backend returns `{data: null, success: true}` for the current spring-2026 season:
  ```
  fetch('/api/themes?year=2026&season=spring&sort=rating')
  → { data: null, success: true }
  ```
- Frontend code at `frontend/web/src/views/Themes.vue` does `themes.value = resp.data?.data || resp.data || []`. When `resp.data.data === null`, the `||` chain falls through to `resp.data` (the envelope object), so `themes.value` becomes `{data: null, success: true}` — an object, not an array.
- Then `<ThemeCard v-for="theme in themes">` iterates the object's properties (`{data, success}`), the first value is `null`, and `theme.id` → `null.id` → crash.
- The OP/ED link is in the primary Navbar — one of only 4 top-level nav items — so it is hit constantly.

**Why it matters:**
1. A primary Navbar link is broken for 100% of users. Anyone clicking "OP/ED" sees a crash.
2. The reload button does nothing — it just re-triggers the same crash.
3. The error message is in English on a Russian-locale site (compounds with UA-030).
4. The `/status` page reports Themes as healthy (UA-029), so there is no observability that would surface this.

**Citations:**
- `frontend/web/src/views/Themes.vue — found via grep "resp.data?.data || resp.data"`
- `frontend/web/src/views/Themes.vue — found via grep "ThemeCard"` (the v-for that crashes)
- `services/themes/` (backend returning `null` instead of `[]` for empty result sets — needs its own investigation to find the exact handler)

**Proposed fix (choose one or both):**

Backend (`services/themes/internal/handler/themes.go` or similar) — return an empty array, not `null`, for empty result sets. This is the *right* fix because it prevents a whole class of frontend bugs.

```go
// instead of: return nil
themes := []*domain.Theme{}
// ... populate themes ...
return themes
```

Frontend (`frontend/web/src/views/Themes.vue`) — validate the response shape defensively:

```ts
const raw = resp.data?.data
themes.value = Array.isArray(raw) ? raw : []
```

Both fixes should land; the backend fix for correctness, the frontend fix for defense in depth.

##### [UA-027] `/auth` has no redirect-if-authed guard — Severity 2 (major) — user control

**View:** `/auth` (desktop)
**Heuristic:** #3 User control and freedom
**Evidence:**
- Logged in as `ui_audit_bot`, navigated directly to `https://animeenigma.ru/auth`.
- The Navbar correctly displays the avatar ("UI" initials) indicating the session is active.
- The page still renders a fresh QR code with "Войти через Telegram" and a 298-second countdown.
- No check on `authStore.isAuthenticated` in `Auth.vue`'s `onMounted`, nor a `beforeEnter` guard in the router.

**Why it matters:**
1. Clicking "Login" out of habit when already authed is confusing — user can't tell if they were logged out.
2. Starting a new deeplink poll while another session is active is wasteful (re-mints tokens, burns Telegram bot API quota).
3. A second device logging in via the same QR could be interpreted as account hijacking.

**Citations:**
- `frontend/web/src/views/Auth.vue — found via grep "onMounted"` (no auth check at mount)
- `frontend/web/src/router/index.ts — found via grep "'/auth'"` (no `beforeEnter` guard)

**Proposed fix:** In `Auth.vue` `onMounted`, early-return with `router.replace('/')` (or `sessionStorage.returnUrl || '/'`) if `authStore.isAuthenticated` is true. Alternative: add a `beforeEnter` to the `/auth` route record.

##### [UA-029] `/status` reports `/themes` as healthy while it is catastrophically broken — Severity 2 (major) — observability

**View:** `/status`
**Heuristic:** #1 Visibility of system status (ironically)
**Evidence:**
- `/status` shows "Опенинги и эндинги" (Themes service) with a green dot and 1ms response time.
- Simultaneously, `/themes` crashes on every load with UA-024.
- The backend returns HTTP 200 for `/api/themes?...` — the data is just `null`, which the frontend can't handle. A service-liveness healthcheck that only looks at HTTP status will not notice this.

**Why it matters:**
1. The page meant to tell users "is something broken?" gets the most visible broken thing wrong.
2. Any operator relying on `/status` to page an oncall would miss the Themes outage entirely.
3. Undermines user trust — if a green status page shows red-down pages, users stop trusting it at all.

**Citations:**
- `frontend/web/src/views/StatusPage.vue — found via grep "getStatus"`
- `services/status/` or gateway healthcheck handler (needs investigation — the status endpoint's implementation wasn't read in this pass)

**Proposed fix:** The healthcheck for Themes should call `/api/themes` with the current year/season and verify the response is `Array.isArray(data) && data !== null`. More generally, every service's healthcheck should make an end-to-end sanity call — not just a liveness ping — so that broken-but-reachable services are flagged.

### Minor

##### [UA-026] Schedule and Status have only one inbound edge each — Severity 1 (minor) — recognition

**Views:** `/schedule`, `/status`
**Heuristic:** #6 Recognition rather than recall
**Evidence:**
- `/schedule` is reachable only from the Home page "Расписание" button. It is **not** in the Navbar. Users arriving via direct link to an anime detail (e.g. from a shared link) have no way to discover the schedule.

**Why it matters:**
1. Schedule is a core feature for ongoing-series viewers — hiding it behind a single button on Home means direct-link visitors miss it completely.
2. Status is useful precisely when other things are broken; if users can't find it during an outage, it loses its purpose.

**Citations:**

- `frontend/web/src/App.vue — found via grep "/status"` (the only footer link)

**Proposed fix:** Add `/schedule` to the Navbar's `navLinks` array. For `/status`, add a small link to the user dropdown menu or a footer expansion, or add a keyboard shortcut (e.g. Shift+?).

##### [UA-028] `/game/:roomId` silently falls back to the lobby for invalid room IDs — Severity 1 (minor) — error visibility

**View:** `/game/:roomId`
**Heuristic:** #9 Help users recognize, diagnose, and recover from errors
**Evidence:**
- Navigated to `/game/non-existent-room-id` as `ui_audit_bot`.
- Page renders the same "Game Rooms" lobby as `/game` with the empty state "Нет активных комнат".
- URL stays at `/game/non-existent-room-id` — no correction, no "room not found" flash.
- `Game.vue` handles both routes with the same component and internal state; there's no "room fetch failed" branch.

**Why it matters:**
1. Stale invite links (posted in chats, shared screenshots, bookmarks) fail silently. User expects to join a room, lands in an empty lobby, thinks the system is broken or that someone cancelled the room.
2. No way to distinguish "room never existed" from "room was just closed" from "socket connection failed".

**Citations:**
- `frontend/web/src/views/Game.vue — found via grep "roomId"` (no error state for invalid room)

**Proposed fix:** Detect the invalid-room case (either 404 from `gameApi.getRoom(roomId)` or socket join failure) and render a "Эта комната не существует или была закрыта" state with a "Вернуться к списку комнат" button.

##### [UA-030] App-level error fallback is hardcoded English on a Russian-locale site — Severity 1 (minor) — i18n

**Heuristic:** #2 Match between system and real world

**Citations:**
- `frontend/web/src/App.vue — found via grep "Something went wrong"`

##### [UA-032] Schedule and Game views have no error states — Severity 1 (minor) — error visibility

**Views:** `/schedule`, `/game`
**Heuristic:** #9 Help users diagnose errors
**Evidence:**
- `Schedule.vue` template has `v-if="loading"`, `v-else-if="schedule.length === 0"`, and `v-else` (the loaded state). **No `v-else-if="error"` branch.**
- `Game.vue` has no visible error branch for socket disconnects — if the WebSocket fails to connect, the user sees the lobby spinning forever.

**Why it matters:**
1. Backend failures become silent — user sees "no data" or spinner, can't tell if the system is broken or just has no content.
2. No retry affordance.

**Citations:**
- `frontend/web/src/views/Schedule.vue — found via grep "v-else-if"`
- `frontend/web/src/views/Game.vue — found via grep "v-if="` (no error-state v-if)

**Proposed fix:** Add an error-state branch to each view's template with a retry button. Both views already have loading and empty states so adding error is a 5-line change each.
