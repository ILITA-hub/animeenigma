## NEW findings

### [UA-034] `/themes` — 82 color-contrast violations on OP/ED cards — Severity 2 (major) — accessibility

**View:** `/themes` (desktop)
**Heuristic:** WCAG 2.1 SC 1.4.3 (contrast minimum)
**Evidence:**
- axe rule `color-contrast`, impact serious, **82 nodes** on the Themes view
- Failing class patterns: `text-white/30` (artist subtitle, vote count subtitle) and `text-white/70` (card title over `bg-white/5`)
- Sample: `<span class="text-white/30"> - Keisuke Kuwata</span>` (artist after `-`)
- Sample: `<h3 class="text-white/70 font-medium text-sm leading-tight truncate">` (card title)
- No instances of this violation at ~82 nodes on any other view → ThemeCard component is the culprit

**Why it matters:** axe-core's single biggest contrast hit of the whole re-audit. OP/ED cards are the entire content of the Themes view. Users with low contrast sensitivity or in bright-ambient conditions cannot read card titles or artist names.

**Citations:**
- `frontend/web/src/components/themes/ThemeCard.vue — found via grep "text-white/30\|text-white/70"` (likely component — verify)

**Proposed fix:** Promote `text-white/30` → `text-white/60` and `text-white/70` → `text-white/85` across ThemeCard. Measure; if any row drops below AA, bump to `/90`. Single-component change; fixes 82 nodes at once.

### [UA-035] `/themes` — 2 unlabeled `<select>` elements (critical select-name) — Severity 2 (major) — accessibility

**View:** `/themes`
**Heuristic:** WCAG 4.1.2 (Name, Role, Value)
**Evidence:**
- axe rule `select-name`, impact **critical**, 2 nodes
- Failing sample: `<select class="bg-white/5 border border-white/10 rounded-lg px-3 py-2 ...">` (no label, no aria-label)
- Visible labels in the view text: "All / Openings / Endings" and "По рейтингу / По названию / Новые" — these are the two `<select>` controls (type + sort)
- Distinct from Browse, which has no `<select>` at all anymore (filter UI rewritten to custom dropdowns)

**Proposed fix:** Either add a visible `<label>` above each select, or `aria-label="Тип"` / `aria-label="Сортировка"` (or i18n keys).

**Citations:**
- `frontend/web/src/views/Themes.vue — found via grep "<select"`

### [UA-036] Anime detail — 15 nodes failing color-contrast on accent buttons — Severity 2 (major) — accessibility

**View:** `/anime/:id` (Frieren)
**Heuristic:** WCAG 2.1 SC 1.4.3
**Evidence:**
- axe rule `color-contrast`, impact serious, **15 nodes**
- Failing sample: `<button class="... accent-bg-muted accent-text border accent...">`
- Class pattern `accent-bg-muted accent-text` indicates the theme-color-adaptive button styling used for things like watch-preference toggles, translation picker chips, player switcher
- Matches: a cluster of 15 small controls using the accent CSS variable as both bg and fg with insufficient contrast

**Proposed fix:** Inspect `frontend/web/src/assets/accent.css` (or wherever `accent-bg-muted` / `accent-text` are defined). Either darken the accent-muted bg or lighten the accent-text fg until ≥4.5:1 (or ≥3:1 if visually-large). This is one CSS-variable pair change; fixes 15 nodes at once.

**Citations:**
- `frontend/web/src/views/Anime.vue — found via grep "accent-bg-muted"` (consumer sample)
- `frontend/web/src/**.css — found via grep "accent-bg-muted"` (definition)

### [UA-040] NotFound — `<title>` tag is English on Russian locale — Severity 0 (cosmetic) — i18n

**View:** `/404` (and any route matching the wildcard)
**Heuristic:** Consistency
**Evidence:**
- `document.title === "Not Found - AnimeEnigma"` while body copy is Russian
- Other views also leak some English in titles (Home: "Home - AnimeEnigma", Themes: "Openings & Endings - AnimeEnigma")
- Pattern: the `<title>` tag is not bound to i18n

**Proposed fix:** Wire `router.afterEach` or a `useHead`/`useTitle` composable to set `document.title = t(route.meta.titleKey)` when locale changes. Or at minimum, replace literal English titles with i18n interpolation in each view's `onMounted`.

### [UA-041] `/themes` + Profile — `heading-order` axe violation (h3 without h2) — Severity 1 (minor) — accessibility

**Views:** `/themes`, `/user/:publicId`
**Heuristic:** WCAG 1.3.1
**Evidence:**
- Both views return axe `heading-order` (moderate impact) because they jump from h1 → h3
- On /themes: h3 card titles directly under the h1 "Опенинги и Эндинги"
- On Profile: h3 card titles directly under h1 username

**Proposed fix:** Either promote section headings to h2, or demote h3 → h2 where appropriate. Usually a small number of strings.
