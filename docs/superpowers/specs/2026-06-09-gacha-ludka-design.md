# Лудка — гача-коллекция аниме-персонажей

**Дата:** 2026-06-09
**Статус:** Design (на ревью)
**Тип:** Новая фича + новый микросервис

---

## 1. Что это и зачем

«Лудка» — гача-система коллекционирования аниме-персонажей за внутреннюю валюту
сайта. Пользователь зарабатывает монеты за обычную активность (в первую очередь —
просмотр аниме), тратит их на крутки на **баннерах** (тематических подборках
карточек) и собирает выпавших персонажей в **коллекцию** в профиле.

Цель — геймификация и ретеншн: дать повод заходить ежедневно, досматривать тайтлы
и возвращаться. На сайте сейчас **нет никакой валюты/экономики/геймификации** —
эта фича вводит её с нуля.

### Метрики (по `.planning/CONVENTIONS.md`)

- **UXΔ = +3 (Better)** — новый залипательный контур вовлечения, не ломает
  существующие потоки (отдельная страница + вкладка профиля).
- **CDI = 0.06 * 89** — новый сервис + новая навигация + хук начисления в `player`.
  Spread средний (новый сервис изолирован, но трогает gateway, навбар, профиль,
  player-хук), Shift низкий (ничего существующего не переписываем). Effort_Fib = 89.
- **MVQ = Kraken 80%/85%** — большой самодостаточный домен со своими щупальцами
  (кошелёк, карты, баннеры, крутки, коллекция, pity), высокая slop-resistance за
  счёт чётких границ сервиса.

## 2. Решения, зафиксированные в брейншторме

Сжатый журнал принятых решений (чтобы не переоткрывать):

| # | Решение | Выбор |
|---|---------|-------|
| 1 | Тип награды | Гача-карточки = аниме-персонажи (коллекция) |
| 2 | Источник карточек | **Ручная курация админом** (не авто-редкость по Shikimori) |
| 3 | Картинка карты | Вариант C — URL **или** загрузка файла, обе кладутся в MinIO |
| 4 | Хранилище картинок | Отдельный бакет MinIO **`gacha-cards`** |
| 5 | Карта vs баннер | Карта — самостоятельная сущность; баннер — просто подборка карт |
| 6 | Баннеры с нуля | Да (вариант B), не один общий пул |
| 7 | Дубликаты | **Вариант D** — просто счётчик «×N», без осколков/возврата (осколки — задел на v2) |
| 8 | Режимы крутки | x1 + **x10** (x10 со скидкой 10%) |
| 9 | Шансы редкости | **N 69% / R 22% / SR 8% / SSR 1%** |
| 10 | Гарант x10 | Каждая пачка ×10 содержит ≥1 карту **SR или выше** |
| 11 | Hard pity | Гарант SSR на **90-й** крутке без SSR; **счётчик у каждого баннера свой** |
| 12 | Источники валюты | Просмотр эпизода (основа) + ежедневный стрик + стартовый бонус + мелочи |
| 13 | Player-UI лудки | Просто **список включённых баннеров** → экран крутки |
| 14 | Коллекция | Сетка по редкости; невыбитые — чёрный силуэт `filter: brightness(0)` |
| 15 | Бэкенд | **Отдельный сервис `services/gacha/`** (порт 8093) |
| 16 | Группировка карт | **Группы** (админ сам создаёт именованные группы, M:N с картами) — для поиска в библиотеке и пачкового добавления в баннер. НЕ игровой пул (это баннер). |

## 3. Архитектура

### 3.1. Новый сервис `services/gacha/` (порт 8093)

Самодостаточный микросервис, владеющий всем гача-доменом. Стандартная раскладка
Go-сервиса проекта (`cmd/`, `internal/{config,domain,handler,service,repo,transport}`).
Зависимости: `libs/database` (GORM/Postgres), `libs/cache` (Redis), `libs/logger`,
`libs/errors`, `libs/httputil`, `libs/metrics`, и MinIO-аплоадер
(переиспользуем паттерн `services/library/internal/minio/writer.go`).

> **Чеклист подключения нового сервиса** (по памяти проекта про новые модули, но
> здесь это **сервис**, не `libs/`): добавить в `docker/docker-compose.yml`
> (контейнер, env, healthcheck, порт `127.0.0.1:8093:8093`), gateway-роут,
> `make`-таргеты (`redeploy-gacha`, `logs-gacha`), Prometheus scrape. Сервис
> имеет свой `go.mod` и **не** входит в `go.work`-граф проверки, если не тянет
> локальные `libs/` через replace — а он тянет, значит как у остальных сервисов:
> require+replace на используемые `libs/`, и его собственный Dockerfile копирует
> нужные `libs/*/go.mod`. **Новые `libs/` мы не создаём**, так что трогать 13
> чужих Dockerfile не нужно.

### 3.2. Роутинг через gateway

- `/api/gacha/*` → `gacha:8093` (JWT required для пользовательских операций).
- `/api/gacha/admin/*` → `gacha:8093`, но под `AdminRoleMiddleware` (как
  существующие `/api/admin/*`).
- `/admin/gacha` (и `/admin/gacha/*`) — fall-through в web-SPA (как уже сделано
  для `/admin/recs`, `/admin/collections`, `/admin/feedback`), чтобы рендерились
  Vue-страницы админки карт/баннеров.
- `POST /internal/gacha/credit` — **только Docker-сеть**, не проксируется
  gateway (как `/internal/notifications`, `/internal/effects`). Через него
  сервисы-источники начисляют валюту.

### 3.3. Начисление валюты (internal-credit паттерн)

Главный источник — просмотр эпизода. `player` уже отмечает эпизод досмотренным
дискретным событием: `ListService.MarkEpisodeWatched` → `ProgressRepository.MarkCompleted`.
В этой точке, **только при первом** переходе эпизода в completed, `player`
шлёт `POST http://gacha:8093/internal/gacha/credit`:

```json
{ "user_id": "...", "reason": "episode_watched",
  "ref": "<anime_id>:<episode>", "amount": 22 }
```

- Продюсер **неблокирующий + drop-on-full** (как egress-recorder в catalog):
  падение gacha никогда не ломает отметку «просмотрено».
- **Идемпотентность на стороне gacha**: уникальный ключ леджера `(user_id, reason, ref)`
  — повторный credit за тот же эпизод игнорируется (INSERT … ON CONFLICT DO NOTHING).
  Это и есть анти-абуз «не платим за один эпизод дважды».

Остальные источники (см. §5) добавляются такими же internal-credit вызовами из
своих сервисов или собственными эндпоинтами gacha (daily-claim, starter).

## 4. Доменная модель (таблицы Postgres)

Все ID — UUID `gen_random_uuid()`. Все денежные операции идут **через леджер** —
баланс это производная (или денормализованное поле кошелька, обновляемое в одной
транзакции с записью леджера).

## 5. Экономика (стартовые числа, тюнятся через config)

**Валюта называется «Энигмы»** (в тему AnimeEnigma). Дальше по тексту «монеты» =
«Энигмы». Нужна иконка валюты (стилизованная под бренд).

> Числовые параметры (стоимости, шансы, суммы начислений) — **env/config**
> сервиса gacha, не хардкод. Отдельной админ-UI для тюнинга чисел в v1 не делаем
> (YAGNI) — меняются через конфиг. Админ-UI — только карты и баннеры.

### 5.1. Стоимость круток
- **x1 = 100 монет**
- **x10 = 900 монет** (скидка 10%)

### 5.2. Начисления
| Событие | Сумма | Источник |
|---------|-------|----------|
| Просмотр эпизода (впервые) | **22** | `player` → internal-credit, дедуп по `anime:ep` |
| Ежедневный заход (claim) | **50** база | gacha daily-claim эндпоинт |
| Стрик-бонус | +10 за каждый день подряд, кап +100 (7 дней) | в daily-claim |
| Досмотрел тайтл | **80** | `player`, дедуп по `anime` |
| Стартовый бонус новичку | **300** (≈3 пробные крутки) | gacha, при первом доступе к кошельку |
| Оценил OP/ED | **5** | `themes` → internal-credit (мелочь, опционально в v1) |
| Реакция/отзыв | **5** | опционально в v1 |
| Watch-together сессия ≥X мин | **30** | опционально в v1 |

**Объём v1 по источникам:** обязательны — просмотр эпизода, ежедневный стрик,
стартовый бонус, досмотр тайтла. Остальные (`themes`, реакции, WT) помечены
опциональными — добавляются тем же паттерном, можно во вторую очередь.

### 5.3. Шансы и гаранты
- Базовые шансы по тиру: **N 69% / R 22% / SR 8% / SSR 1%** (применяются в пределах
  карт баннера; если в баннере нет карт какого-то тира — его вес перераспределяется
  на ближайший доступный тир вниз).
- **Гарант x10:** среди 10 карт гарантированно ≥1 SR-или-выше. Если все 10
  нароллились N/R — последняя апается до случайной SR-карты баннера.
- **Hard pity SSR:** `pulls_since_ssr` инкрементится каждой круткой по баннеру;
  на **90-й** крутке без SSR следующая крутка принудительно SSR. Любой SSR (везучий
  или форсированный) сбрасывает счётчик в 0. Счётчик **на каждый баннер свой**.

### 5.4. Алгоритм одной крутки (атомарно, в транзакции)
```
1. Проверить баланс ≥ cost; иначе 402/ошибка.
2. Списать cost (леджер + wallet.balance) с reason=pull_x1/pull_x10.
3. Для каждой крутки в пачке:
   a. pulls_since_ssr++ ; если == pity_threshold → форс SSR.
   b. иначе выбрать тир по весам (с учётом наличия карт тиров в баннере).
   c. если тир == SSR → pulls_since_ssr = 0.
   d. выбрать случайную карту этого тира из карт баннера.
4. (x10) если ни одной SR+ — апнуть последнюю до случайной SR.
5. upsert gacha_collection (count++ для дублей, отметить NEW для новых).
6. Сохранить pity. Вернуть список выпавших карт + флаги NEW/dup-count.
```
Слабые случайные источники не используем — `crypto/rand`/`math/rand` сервера;
розыгрыш **только на бэке**, фронт ничего не решает.

## 6. Пользовательский интерфейс

### 6.1. Навбар
- Новый пункт **«Лудка»** → `/gacha`.
- **Баланс «Энигм»** (иконка + число) виден с любой страницы; кликабелен → ведёт в
  лудку. Обновляется после крутки/начисления.

### 6.2. Страница `/gacha` — список баннеров
- Просто **список включённых (активных сейчас) баннеров**: шапка-арт + название +
  краткое описание. Стандартный баннер всегда присутствует.
- Пусто (кроме стандартного) если админ ничего не включил.
- Клик по баннеру → экран крутки.

### 6.3. Экран крутки баннера
- Арт баннера, твой баланс.
- **Счётчик pity этого баннера**: «До гаранта SSR: 73 / 90».
- Кнопки **Крутить ×1 (100)** и **Крутить ×10 (900)**; дизейбл если не хватает монет.
- Список карт баннера (за кем охота) — с пометкой, что уже выбито.
- Результат крутки: показ выпавшей карты (картинка + имя + подсветка редкости);
  для ×10 — все 10 разом. Новая карта помечена **NEW**, дубль — **×N**. Без
  сложных анимаций — чёткий показ «что выпало».

### 6.4. Коллекция (вкладка в профиле)
- **Альбом всех включённых карт библиотеки** (не только текущего баннера) —
  полная коллекция, сетка **сгруппирована по редкости** (SSR сверху → N снизу).
- Невыбитые карты — тот же `<img>`, но с `filter: brightness(0)` (чёрный силуэт),
  имя скрыто под «???», редкость видна (карты и так сгруппированы по тирам).
- Выбитые — нормальный показ + бейдж «×N» для дублей.
- Счётчик прогресса по тиру («SSR: 3 / 12»).
- API коллекции отдаёт по каждой карте `image_url` + `owned: bool` + `count`;
  силуэт — чисто фронтовый CSS, отдельные «силуэт-картинки» не генерим.

### 6.5. Соответствие дизайн-системе
Всё на токенах Neon-Tokyo и примитивах `@/components/ui` (Button/Card/Badge/Dialog/
Tabs). Редкости — через семантические цвета/бренд-акценты (не off-palette классы,
DS-lint это завалит). `font-medium`/`font-semibold` только. Любые визуальные
изменения смоук-тестятся в реальном браузере на desktop+mobile (DS-NF-06).

## 7. Админка

Существующая admin-SPA (`frontend/web/src/views/admin/`), новые страницы под
`/admin/gacha`:

### 7.1. Библиотека карт (`/admin/gacha/cards`)
CRUD: имя, аниме-источник, редкость (селект N/R/SR/SSR), картинка (вставить URL
**или** загрузить файл — оба пути ведут в MinIO `gacha-cards`), вкл/выкл,
**группы** (мультиселект из существующих групп). Список с фильтром по
редкости/вкл/**группе**. Backend: `POST/PATCH/DELETE/GET /api/gacha/admin/cards`.

### 7.1a. Группы карт (`/admin/gacha/groups`)
CRUD групп: создать/переименовать/удалить именованную группу. Назначение карт в
группу — либо с карточки (§7.1), либо из экрана группы (мультиселект карт).
Backend: `POST/PATCH/DELETE/GET /api/gacha/admin/groups`. В редакторе баннера
(§7.2) — кнопка **«добавить всю группу X»**: подтягивает все карты группы в состав
баннера одним действием.

### 7.2. Баннеры (`/admin/gacha/banners`)
CRUD: название, описание, арт, расписание (active_from/to), вкл/выкл, sort_order,
и **набор карт** (мультиселект из библиотеки карт). Backend:
`POST/PATCH/DELETE/GET /api/gacha/admin/banners`, управление составом через
`gacha_banner_cards`.

### 7.3. Загрузка картинок
`POST /api/gacha/admin/upload` (admin-only): принимает либо файл (multipart), либо
`{image_url}` → бэкенд скачивает по URL → кладёт в MinIO `gacha-cards` → возвращает
наш `image_url`. Бакет `gacha-cards` создаётся на старте сервиса (EnsureBucket,
как в library-writer).

## 8. Публичный API (эскиз)

Пользовательские (JWT):
- `GET /api/gacha/wallet` — баланс, стрик, выдать стартовый бонус при первом доступе.
- `POST /api/gacha/daily` — заклеймить ежедневный бонус (+стрик-логика).
- `GET /api/gacha/banners` — активные сейчас баннеры + (для выбранного) карты + мой pity.
- `POST /api/gacha/banners/{id}/pull` — `{ mode: "x1" | "x10" }` → списать, разыграть, вернуть выпавшее.
- `GET /api/gacha/collection` — мои карты (+ невыбитые силуэты по активному пулу/всем картам).

Внутренний (Docker-сеть):
- `POST /internal/gacha/credit` — начисление от сервисов-источников.

Админские (admin role):
- `…/admin/cards`, `…/admin/groups`, `…/admin/banners`, `…/admin/upload` (см. §7).
